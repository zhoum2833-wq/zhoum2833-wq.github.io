# 调试三件套

调试是什么？简单说，就是让你的程序在出错时，你能够**看见**它到底哪里出了问题。嵌入式系统的调试比普通软件困难——没有显示器，没有鼠标，只有一个黑乎乎的单片机芯片。本章介绍三种从简单到高级的调试方法，每一种都有各自的适用场景。

## 方法一：printf 大法

这是历史最悠久、使用最广泛的调试手段。就像在黑屋子里打开手电筒——你在代码的关键位置插入 `printf()` 语句，让单片机把内部变量值通过串口发出来，你在电脑上用串口助手就能看到。

```c
#include <stdio.h>

int main(void) {
    int adc_val = 0;

    printf("程序启动\r\n");

    while (1) {
        adc_val = ADC_Read(0);
        printf("ADC值: %d\r\n", adc_val);
    }
}
```

`\r\n` 不能省略——串口终端的换行需要回车+换行两个字符，少一个就会发生"字符从不对齐"的诡异现象。

使用 printf 的前提是你的 UART（通用异步收发器，也就是串口）已经初始化好了。在我们的项目中，`bsp_uart.c` 已经帮你做了这件事。

::: tip 实战技巧
在怀疑有 bug 的函数入口和出口各放一个 printf。如果只看到"进入"没看到"退出"，说明函数中途卡死或崩溃了。这比盯着代码猜一个小时有效得多。
:::

### 优缺点

| 优点 | 缺点 |
|------|------|
| 不需要任何专用硬件 | 会让程序变慢一点点 |
| 几乎所有平台都支持 | 变量太多时输出杂乱 |
| 调试信息可以保存为文本文件 | 不能暂停程序查看完整内存状态 |

## 方法二：断点调试

断点调试就像**给程序按暂停键**。你在 VS Code 里点一下行号左边，那里就多了一个红点（断点）。程序运行到这一行时自动暂停，然后你可以：

- 把鼠标悬停在变量上，看它当前的值
- 按 F10 单步执行（一行一行地走）
- 按 F11 进入函数内部查看

需要的工具链：VS Code + Cortex-Debug 插件 + JLink 调试器。

::: warning 注意事项
断点暂停了 CPU，但**现实世界不会暂停**。电机还在转，传感器的电压还在变化，PWM 信号也停了。所以断点调试不适合调试需要实时运行的代码（比如电机控制、陀螺仪数据读取）。
:::

## 方法三：Vofa+ 遥测

Vofa+ 是一个国产的上位机软件，可以把串口数据画成**实时曲线图**。这对于调 PID 参数来说简直是神器——你不需要盯着不断滚动的数字，而是直接看曲线的形态。

你的 2WD_Car 项目已经通过 `bsp_vofa.c` 实现了 13 通道 50Hz 的数据流。它同时发送 13 个变量的值（如左右轮速度、期望速度、陀螺仪角度、PID 输出等），全部画成曲线，就像医院的监护仪一样。

```c
// bsp_vofa.c 的数据发送格式
// 13 个浮点数用逗号分隔，每秒发送 50 次
Vofa_SendFrame(left_speed, right_speed, target_speed,
               gyro_z, error, p_out, d_out);
```

### 三种方法的选用指南

| 场景 | 推荐方法 |
|------|----------|
| 想知道程序有没有跑进某个 if 分支 | printf |
| 想知道某个变量的值对不对 | printf 或 断点 |
| 数组越界、野指针等内存问题 | 断点（看内存窗口） |
| PID 参数整定 | Vofa+ |
| 电机启动瞬间的电流波形 | Vofa+ |
| 复杂逻辑的逐步验证 | 断点（单步执行） |

::: tip 经验之谈
老手通常三种混用。在关键分支放 printf 保证程序"活着"，出问题时打断点深入查看，调控制算法时用 Vofa+ 看曲线。不要迷信任何一种——组合使用才是王道。
:::

## 小结

调试的核心心法：先确认硬件（设备管理器认到调试器了吗？）→ 再确认软件链（GDB Server + arm-none-eabi-gdb 都在吗？）→ 最后才看代码。90% 的调试问题不是代码 bug，而是调试环境没配通。记住最有用的三个调试手段：串口打印（最快）、断点 + 变量查看（最直观）、逻辑分析仪（调协议必备）。
