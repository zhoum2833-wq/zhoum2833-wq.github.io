# UART vs I2C vs SPI —— 三种协议对比

## 一句话概括

- **UART**：最简单的点对点通信，两根线交叉连接，适合调试和连接单一外设（GPS、蓝牙）。
- **I2C**：两根线挂一堆传感器，通过地址区分设备，适合低速多设备场景。
- **SPI**：四根线高速全双工，通过片选引脚区分设备，适合需要速度的场景（屏幕、SD 卡）。

## 核心参数速查表

| 对比项 | UART | I2C | SPI |
|--------|------|-----|-----|
| 最少引脚数 | 2（TX, RX） | 2（SCL, SDA） | 4（SCK, MOSI, MISO, CS） |
| 典型速度 | 最高几 Mbps | 100k / 400k / 1M bps | 数十 MHz |
| 最大距离 | 数米 | 同一块 PCB 或几十厘米 | 同一块 PCB 或几十厘米 |
| 通信方式 | 异步（无时钟线） | 同步（有时钟线） | 同步（有时钟线） |
| 双工方式 | 全双工 | 半双工（一根数据线） | 全双工 |
| 多设备支持 | 一对一（点对点） | 一对多（地址机制，最多 127 从机） | 一对多（CS 引脚机制） |
| 设备区分方式 | 无需区分（只有对方） | 7 位 / 10 位设备地址 | 每个从机独占一根 CS 引脚 |
| 硬件复杂度 | 最简单 | 中等（需要上拉电阻） | 较复杂（多根线） |
| 错误检测 | 可选校验位（奇偶校验） | 每个字节后有 ACK/NAK | 无内建机制 |
| 主从关系 | 对等（双方都能主动发） | 严格主从（只有主机能发起） | 严格主从（只有主机能发起） |

## 速度对比

速度的直观比较（同等条件下，传输 1KB 数据的耗时估算）：

| 协议 | 常用速度 | 1KB 耗时（约） |
|------|---------|---------------|
| UART | 115200 bps | ~90 ms |
| I2C（标准） | 100 kbps | ~100 ms |
| I2C（快速） | 400 kbps | ~25 ms |
| SPI | 10 MHz | ~1 ms |

::: tip UART 的实际速度不止 115200
UART 理论上可以达到几 Mbps（比如很多 GPS 模块默认 9600，但 STM32 的 UART 可以跑到 4.5 Mbps）。但 115200 是最常用的开发速度，不需要更高的场合不必折腾。
:::

## 多设备连接方式对比

### UART —— 只能一对一

```
MCU ──────────── 设备A
```

一个 UART 外设只能连接一个设备。如果需要多个 UART 设备，就要用 MCU 的多个 UART 外设（USART1、USART2、USART3……）。每个 UART 外设占用两个引脚。

### I2C —— 一根总线挂所有

```
           SDA ───┬─────┬─────┬─────
           SCL ───┬─────┬─────┬─────
                  │     │     │
                设备A  设备B  设备C
              (0x3C) (0x44) (0x68)
```

一条 I2C 总线（两根线 + 两个上拉电阻）可以挂 100+ 个设备。每个设备通过唯一的地址区分。

### SPI —— 共用数据线，独立 CS

```
           SCK ──┬──────┬──────
          MOSI ──┬──────┬──────
          MISO ──┬──────┬──────
                 │      │
           CS1 ──┘      │
           CS2 ─────────┘
                 │      │
               设备A   设备B
```

所有从机共用 SCK、MOSI、MISO，但每个从机独占一根 CS 线。主机通过拉低不同设备的 CS 来逐个"点名"。

## 如何选择：快速决策

```
你要做什么？
│
├─ 串口调试输出（printf）              → UART
├─ 连接蓝牙/GPS/WiFi 模块              → UART（这些模块本身就是 UART 接口）
├─ 连接多个传感器，引脚不够用           → I2C
├─ 连接 OLED 屏（引脚紧张时）           → I2C
├─ 连接 OLED/LCD 屏（追求刷新速度）     → SPI
├─ 读写 SD 卡                          → SPI
├─ 连接高速 ADC（采集波形）             → SPI
└─ 不确定                               → 先用 UART 调试，原型跑通再换
```

更具体的常见场景对照表：

| 具体应用场景 | 选哪个协议 | 理由 |
|-------------|-----------|------|
| printf() 调试 | UART | 最简单，电脑串口助手直接看 |
| HC-05 蓝牙模块 | UART | 蓝牙模块本身是 UART 接口 |
| JY61P 陀螺仪 | UART | 这款传感器输出 UART 数据 |
| GPS 模块 | UART | 99% 的 GPS 模块用 UART |
| OLED 显示屏（SSD1306） | I2C 或 SPI | 取决于你买的模块是哪种版本 |
| 温湿度传感器 SHT30 | I2C | 传感器是 I2C 接口 |
| MPU6050 陀螺仪 | I2C 或 SPI | I2C 更常见，引脚更少 |
| EEPROM / AT24C02 | I2C | 存储器是 I2C 接口 |
| TFT LCD 彩色屏 | SPI | 刷新速度快，画面流畅 |
| SD 卡模块 | SPI | 速度要求，标准接口 |
| LED 点阵屏（MAX7219） | SPI | 高速刷新 |
| 两个 MCU 之间通信 | UART | 简单可靠，双方对等 |

## 实际项目中三种协议同时使用

在一个典型的比赛小车项目中，你很可能同时使用三种协议：

```
MCU (STM32F103C8T6)
│
├── USART1 → CH340 → USB → 电脑（调试输出 printf）
├── USART2 → HC-05 蓝牙模块（遥控指令接收）
├── USART3 → JY61P 陀螺仪（姿态数据）
├── I2C1   → OLED 显示屏（显示状态）+ SHT30（温湿度）
├── SPI1   → SD 卡模块（数据记录）
├── GPIO   → 电机驱动（TB6612）
└── GPIO   → 编码器（电机转速测量）
```

**规划引脚的思路**：
1. 先列出所有需要的外设。
2. 根据每个外设的接口类型分到对应的通信外设上。
3. 用 CubeMX 检查引脚是否冲突（比如 PA9 被 USART1_TX 占了就不能同时用做 I2C）。
4. I2C 和 SPI 可以挂多个设备在同一条总线上，节省引脚。UART 不行。

## 容易搞混的地方

| 容易混淆的点 | 正确答案 |
|-------------|---------|
| I2C 的两根线和 UART 的两根线 | 完全不同！UART 是 TX/RX（两个方向的数据线），I2C 是 SCL/SDA（时钟+数据）。不能互换。 |
| SPI 也有 TX/RX？ | SPI 叫 MOSI/MISO，虽然功能类似 UART 的 TX/RX，但 SPI 多了一根时钟线（SCK），需要主机提供时钟。 |
| I2C 的地址和 SPI 的 CS | I2C 用软件地址区分从机，SPI 用物理的 CS 引脚区分从机。I2C 省引脚但不能太快，SPI 快但每个从机多吃一个引脚。 |
| 波特率 vs 时钟频率 | UART 说"波特率"，I2C 和 SPI 说"时钟频率"。本质上都是每秒传输多少 bit，但因为 UART 没有时钟线，叫法不同。 |

## 小结

没有"最好"的协议，只有"最合适"的协议。

- 追求**简单**：选 UART。两根线，调试神器。
- 追求**省引脚+连多个传感器**：选 I2C。两根线配地址，挂一堆设备。
- 追求**速度**：选 SPI。全双工，几十 MHz，屏幕和存储的首选。

UART、I2C、SPI 三种协议覆盖了嵌入式 90% 以上的设备通信场景。其余的（CAN、RS-485、USB、Ethernet）都是在这三个基础之上的特定场景扩展，用到再学。
