# CAN 总线

## 为什么需要 CAN

UART、I2C、SPI 是"点对点"或"一主多从"的短距离通信，设备之间聊聊天没问题。但如果你在造一辆真正的汽车——发动机、刹车、方向盘、车灯、空调、仪表盘……几十个模块都要互相通信，用 UART 或 I2C 会是什么后果？

- 几十对 TX/RX 线，线束比手臂还粗
- 发动机的电磁干扰让 UART 天天误码
- 刹车信号和空调数据抢总线——谁先发？谁重要？

**CAN（Controller Area Network，控制器局域网）就是为解决这些问题而生的。** 它用两根线连接所有设备，抗干扰、仲裁优先级、自带错误检测。

## CAN 的物理层

```
        CAN 总线（两根线：CAN_H 和 CAN_L，双绞在一起）
  ════════════════════════════════════════════
    │          │          │          │
 ┌──┴──┐   ┌──┴──┐   ┌──┴──┐   ┌──┴──┐
 │节点1 │   │节点2 │   │节点3 │   │节点4 │
 │MCU  │   │传感器│   │电机  │   │仪表  │
 └─────┘   └─────┘   └─────┘   └─────┘
    ↑
总线两端各一个 120Ω 终端电阻（消除信号反射）
```

**差分信号**：CAN_H 和 CAN_L 互为镜像——CAN_H 拉高时 CAN_L 拉低，接收端只看两者之差。和 USB 的差分信号原理一样（详见第九篇 USB 章节），抗干扰能力极强。

| 状态 | CAN_H | CAN_L | 差值 |
|------|-------|-------|------|
| 显性（逻辑 0） | 3.5V | 1.5V | 2V |
| 隐性（逻辑 1） | 2.5V | 2.5V | 0V |

::: tip 显性优先
CAN 总线的关键设计——**显性位（0）优先于隐性位（1）**。多个节点同时发数据时，谁先发 1 谁就退出发送，让发 0 的继续——这就是 CAN 的**无损仲裁**。刹车信号（高优先级 ID = 更早出现 0）永远赢过空调数据。
:::

## STM32 的 CAN 外设

STM32F103 及以上基本都带 **bxCAN**（基本扩展 CAN）控制器，支持 CAN 2.0A（标准帧 11 位 ID）和 CAN 2.0B（扩展帧 29 位 ID）。

**你需要外接一个 CAN 收发器**（MCU 的 CAN_TX/RX 是 3.3V 逻辑电平，不能直接驱动 CAN 总线）：

```
STM32                   CAN 收发器（TJA1050 / SN65HVD230）
────────                ──────────────
CAN_TX (PB9)  ────────→ TXD
CAN_RX (PB8)  ────────→ RXD
                        CANH ── CAN 总线高
                        CANL ── CAN 总线低
                        3.3V / 5V
                        GND
```

## CAN 帧格式（够用版）

一个标准 CAN 数据帧长这样：

```
  | SOF | ID(11位) | RTR | DLC(4位) | DATA(0~8字节) | CRC | ACK | EOF |
    1      11        1       4           0~64         15    2     7    (位数)
```

不用背诵——你只需要知道：
- **ID**：不是地址！是**优先级**。ID 越小优先级越高。
- **DLC**：数据长度（0~8 字节）。一帧最多 8 字节。
- **DATA**：有效载荷。8 字节看似少，但对于控制指令足够了。

```c
// STM32 HAL 发送一帧 CAN 数据
CAN_TxHeaderTypeDef tx_header;
tx_header.StdId = 0x123;        // 标准 ID
tx_header.IDE = CAN_ID_STD;     // 标准帧
tx_header.RTR = CAN_RTR_DATA;   // 数据帧
tx_header.DLC = 4;              // 4 字节数据

uint8_t data[] = {0x55, 0xAA, 0x01, 0xFF};
uint32_t mailbox;

HAL_CAN_AddTxMessage(&hcan, &tx_header, data, &mailbox);
```

## 电赛里的 CAN

| 场景 | 需要 CAN 吗？ |
|------|-------------|
| 巡线小车（一块 MCU） | 不需要。所有传感器和电机都接同一块板子 |
| 多 MCU 分布式系统（主控+从机 各管一部分） | 可以考虑 |
| 四轴飞行器 | 通常不用 CAN（用 PWM/DSHOT/串口） |
| RoboMaster / 智能车竞赛 | **推荐**——底盘、云台、视觉三块板子用 CAN 通信 |

::: tip 实话实说
电赛入门题 95% 不需要 CAN。一块 STM32 管完所有传感器和电机就够了。但如果你在做多 MCU 分布式的系统——比如一块板跑巡线 PID，另一块板跑视觉识别——CAN 比 UART 可靠得多。这篇知道 CAN 是干什么的、什么时候该上，就够了。
:::

## 小结

CAN 总线是汽车和工业控制领域的"标准通信语言"——两线差分、优先级仲裁、抗干扰强。电赛入门阶段大概率用不上，但理解它的存在和适用场景，是你从"单片机入门"到"嵌入式系统设计"的重要一步。
