# 怎么和 AI 正确聊天

很多人用 AI 的效果不好，不是因为 AI 不够聪明，而是因为**提问方式不对**。AI 就像一个非常勤奋但缺乏背景信息的实习生——你给的上下文越清晰，它的产出质量越高。

## 好提示的四个要素

一个完整的提示（Prompt）应该包含：

| 要素 | 说明 | 示例 |
|------|------|------|
| **背景** | 你在做什么项目、用了什么平台 | "我正在为 MSPM0G3507 写嵌入式代码" |
| **目标** | 你要实现什么功能 | "想通过 I2C 驱动 SSD1306 OLED 显示 'Hello'" |
| **约束** | 有什么限制和特殊要求 | "使用 TI DriverLib，不能用 HAL 库" |
| **期望输出** | 你希望 AI 给你什么 | "请给我 bsp_oled.c 和 bsp_oled.h 两个文件" |

## 好提示 vs 坏提示

**坏提示（太模糊）：**
> "帮我写个 OLED 驱动。"

AI 不知道你的芯片型号、不知道你用哪种库、不知道你的 I2C 引脚分配、不知道你想要什么接口。它只能猜，猜错的部分你再反复修改——浪费时间和 token。

**好提示（要素齐全）：**
> "我正在为 MSPM0G3507 写嵌入式代码（背景），想要通过 I2C 驱动 SSD1306 OLED 显示屏，显示 'Hello World' 字符串（目标）。使用 TI DriverLib 的 I2C API，I2C 地址是 0x3C，引脚是 SCL=P1.2, SDA=P1.3（约束）。请给我 bsp_oled.c 和 bsp_oled.h 两个文件，包含初始化、清屏、显示字符串、显示数字四个函数（期望输出）。"

## 迭代式对话

AI 的第一版答案通常不是完美的。这很正常——把 AI 当成协作伙伴，而不是神谕：

```
第 1 轮：你发出需求
第 2 轮：AI 给出初步方案
第 3 轮：你指出问题："显示函数没有处理中文字符，请修改"
第 4 轮：AI 修改后的问题："I2C 初始化里忘了配置上拉电阻，请补充"
第 5 轮：最终确认
```

::: tip 不要接受第一版答案
即使 AI 的回答看起来没问题，也要逐行审查。AI 可能遗漏了边界条件（如通信失败的处理），或者用了一个对新手来说过于复杂的实现方式。多问一句"有没有更简单的写法？"往往能获得更好理解的代码。
:::

## 常见提问技巧

| 技巧 | 示例 |
|------|------|
| 要求简单实现 | "请用最基础的方式写，不要用高级 C 特性" |
| 要求加注释 | "每三行代码加一行中文注释" |
| 要求解释 | "请解释 TIM4 的 Encoder Mode 是如何工作的" |
| 要求对比 | "LDO 和 DC-DC 在什么场景下分别选用？用表格比较" |
| 要求查错 | "这段代码编译报错 'undefined reference to Timer_Init'，帮我分析原因" |
| 要求简化 | "这个 P 控制器能不能写得更简单？我不需要积分项" |

## 代码审查提示

当你拿到 AI 写的代码，用这些问题来审查：

1. 变量名有意义吗？（`x`, `tmp`, `val` 通常不是好名字）
2. 函数长度合理吗？（超过 50 行的函数通常需要拆分）
3. 有错误处理吗？（I2C 通信失败会怎样？）
4. 有注释说明"为什么这样做"而不是"做了什么"吗？
5. 我能对着代码向别人解释每一行吗？（如果不行，让 AI 加注释）

::: warning 对 AI 代码的态度
把 AI 生成的代码当作**初稿**而不是**终稿**。你是代码的最终责任人——如果代码烧进单片机后出了事故，比赛裁判不会接受"是 AI 写的"作为理由。
:::
