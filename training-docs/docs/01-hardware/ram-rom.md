# RAM 与 ROM（Flash）

你写了一个 `int x = 5;`——这个 `x` 存在哪里？你编译下载的程序本身又存在哪里？为什么断电后重新上电程序还在，但变量值全丢了？答案在单片机的两种存储器里：RAM 和 ROM（Flash）。搞懂它俩的分工，单片机的"记忆"逻辑就没秘密了。

| | RAM | Flash |
|------|-----|-------|
| 速度 | 极快 | 较慢 |
| 断电 | 数据丢失 | 保留 |
| 容量 | KB 级 | KB~MB 级 |
| 存放 | 变量、栈、堆 | 程序代码、常量 |

Flash 存程序，RAM 存数据。Flash 大但慢，因为程序代码体积大但不常变。RAM 小但快，因为变量频繁读写。

## 编译后的内存布局

C 代码编译后分成几个段：

| 段 | 位置 | 内容 |
|----|------|------|
| `.text` | Flash | 代码指令 |
| `.rodata` | Flash | 常量、字符串 |
| `.data` | RAM | 已初始化全局变量（初值从 Flash 拷过来） |
| `.bss` | RAM | 未初始化全局变量（上电清零） |
| stack/heap | RAM | 局部变量、函数调用、动态分配 |

```c
const int table[] = {1,2,3};  // .rodata → Flash
int count = 0;                 // .data → RAM（初值 0 在 Flash）
int buffer;                    // .bss → RAM（自动清零）

void f(void) {
    int x = 10;                // stack → RAM（函数返回即释放）
}
```

用 `arm-none-eabi-size` 可以查看各段大小。Flash 溢出编译就报错，RAM 溢出只会在运行时崩溃——更难排查。


## 小结

把单片机想象成一个小房间：Flash 是书架，存着你的程序这本书——容量大，但翻书慢，不过断电了书还在。RAM 是桌子上的一小片工作区，容量小但伸手就能拿到东西——可惜一关灯（断电），桌上的东西就全没了。理解 `.data` 和 `.bss` 的区别很关键：有初值的全局变量其实在 Flash 里悄悄存了一份"底稿"，上电时才抄到 RAM 里；没初值的直接清零就好。作为新手，你的两条黄金法则就是：程序写不下时先怀疑 Flash 满了，程序跑着跑着莫名其妙死机时先怀疑 RAM 溢出了。
