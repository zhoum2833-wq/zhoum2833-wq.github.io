# 为什么需要操作系统

你已经学会写裸机程序了——一个 `while(1)` 处理所有事情。那为什么还需要操作系统？这一章回答这个问题。

## 裸机够用的时候

如果你的程序逻辑很简单——比如"检测按键 → 点亮 LED"或者"读红外传感器 → 控制舵机转固定角度"——裸机完全够用。一个 `while(1)` 循环，按顺序执行，清晰直观。

```c
// 裸机：简单直接
while (1) {
    read_sensor();      // 读传感器
    compute_pid();      // 计算控制量
    set_motor();        // 输出到电机
}
```

这种模式的特点是：**你完全掌控执行顺序，每一行代码什么时候运行你知道得一清二楚。**

## 裸机不够用的时候

但当你加入更多功能时，事情开始失控：

```
任务清单：
1. 电机控制 —— 必须每 1ms 执行一次（精确！）
2. 屏幕刷新 —— 每 30ms 刷新一次
3. WiFi 通信 —— 数据包随时可能到达
4. 按键检测 —— 任何时候都可能被按下
5. 编码器计数 —— 不能丢一个脉冲
6. 串口调试输出 —— 不影响主要功能
```

如果把这些全塞进一个 `while(1)` 里：

```c
while (1) {
    motor_control();     // 这个需要精确 1ms...
    screen_update();     // 这个要跑很久...
    wifi_check();        // 这个不知道什么时候来数据...
    button_check();      // 这个错过了就感觉延迟...
    encoder_read();      // 这个不能漏...
    debug_print();       // 这个慢一点没关系...
}
```

问题立刻暴露：

- `screen_update()` 执行需要 15ms，期间 `motor_control()` 得不到执行，电机控制周期就乱了。
- `wifi_check()` 可能在等数据包的超时中阻塞 100ms，其他所有任务全部停顿。
- 你想给不同的任务不同的执行频率，但在一个 `while(1)` 里，只能用 `if (count % 30 == 0)` 这种丑陋的方式。

## 操作系统如何解决

操作系统（OS）的核心能力是**任务调度**：在多个任务之间快速切换，让每个任务看起来都在"同时运行"。

```
时间轴：

裸机（串行执行）：
|--任务A--|--任务B--|--任务C--|--任务A--|--任务B--|--任务C--|--...
每个任务必须等前一个完成

RTOS（时间片轮转）：
|A|B|C|A|B|C|A|B|C|A|B|C|...
在任务之间快速切换，切换速度极快（微秒级）
人眼看起来就是"同时运行"
```

就像饭店里只有一个厨师（CPU），但多个炉灶同时炖着菜——厨师在灶台间快速移动，每个菜都在"同时"烹饪，而不是做完一道才做下一道。

## 操作系统帮你管理什么

| 管理内容 | 裸机怎么做 | OS 怎么做 |
|----------|-----------|----------|
| 任务切换 | 你自己安排执行顺序 | 调度器自动切换 |
| 定时 | `HAL_Delay()` 死等 | 任务休眠，CPU 去跑别的任务 |
| 通信 | 全局变量（容易出错） | 队列（安全传递） |
| 同步 | 标志位（flag） | 信号量（Semaphore） |
| 资源共享 | 关中断保护 | 互斥锁（Mutex） |

## 三种层次的对比

```
         ┌──────────────────────────────┐
         │       Linux / 大型 OS         │
         │  · 文件系统、网络协议栈        │
         │  · 图形界面、多用户           │
         │  · 非实时（响应时间不确定）    │
         │  · 需要 MMU（内存管理单元）    │
         ├──────────────────────────────┤
         │       RTOS（FreeRTOS 等）     │
         │  · 任务调度、队列、信号量      │
         │  · 实时性（硬实时，响应确定）  │
         │  · 占用资源极少（几 KB ROM）   │
         │  · 无需 MMU，单片机直接跑      │
         ├──────────────────────────────┤
         │       裸机（Bare-Metal）      │
         │  · 一个 while(1) 统治一切      │
         │  · 完全掌控、零开销            │
         │  · 适合简单任务               │
         └──────────────────────────────┘
```

## 什么时候该上 RTOS？

::: tip
**经验法则**：当你的裸机 `while(1)` 里出现超过 3 个"必须在不同频率执行"的任务，而且你开始写大量 `if (tick % N == 0)` 这样的代码时，就该上 RTOS 了。
:::

对于竞赛小车来说，典型的判断标准：

**裸机够用的场景**：
- 只有巡线（单任务）
- 巡线 + 简单遥控

**应该上 RTOS 的场景**：
- 巡线 + 实时遥测 + 紧急停止 + 屏幕显示
- 多个传感器需要不同频率采样
- 需要同时处理多个通信协议（UART + I2C + CAN）

## 小结

裸机和操作系统不是"谁对谁错"的问题——是"哪个更适合当前项目"的问题。裸机简单可控，适合简单任务。RTOS 让多任务编程变得优雅，适合复杂项目。Linux 功能丰富但不实时，竞赛场合用不到。下一章我们来看 FreeRTOS——竞赛小车最常用的 RTOS。
