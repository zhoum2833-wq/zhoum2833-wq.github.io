# Linux — 当单片机不够用了

前面我们学了裸机和 RTOS。现在来了解一个更高层次的话题：Linux。这一章不是教你用 Linux，而是让你**知道它是什么、什么时候用、以及为什么竞赛里用不到它**。

## STM32 和 MSPM0 能跑 Linux 吗？

**不能。** 这是一个必须首先明确的事实。

原因很简单：Linux 需要一个叫 **MMU（Memory Management Unit，内存管理单元）** 的硬件组件。MMU 负责把程序的"虚拟地址"翻译成"物理地址"，这是 Linux 多进程隔离的基础。

STM32 的 Cortex-M 系列和 TI 的 MSPM0 属于**微控制器（MCU）**，它们没有 MMU。能跑 Linux 的芯片是**应用处理器（Application Processor）**，比如：

| 能跑 Linux 的芯片 | 搭载的开发板 |
|------------------|------------|
| BCM2711 (Cortex-A72) | 树莓派 4/5 |
| 全志 H3/H5/H616 (Cortex-A53) | 香橙派 |
| NXP i.MX 6/8 系列 | 各种工业板 |
| TI AM335x (Cortex-A8) | BeagleBone |

这些芯片和 STM32 的区别不只是"快一点"——它们的架构（Cortex-A vs Cortex-M）完全不同，就像摩托车和汽车的区别：都是车，但结构和工作方式天差地别。

## 微控制器 vs 应用处理器

```
应用处理器（Application Processor）
┌────────────────────────────────────┐
│  · 有 MMU，能跑完整的 Linux/Android │
│  · 主频高（1~3 GHz）                │
│  · 功耗高（几瓦）                   │
│  · 外接 DDR 内存（几 GB）           │
│  · 典型代表：树莓派、手机处理器     │
│  · 适合：图形界面、AI、网络服务器   │
├────────────────────────────────────┤
│  微控制器（Microcontroller / MCU）  │
│  · 无 MMU，跑裸机或 RTOS           │
│  · 主频低（几十 ~ 几百 MHz）       │
│  · 功耗极低（毫瓦级）               │
│  · 内置 RAM 和 Flash（KB~MB 级）   │
│  · 典型代表：STM32、MSPM0、ESP32   │
│  · 适合：电机控制、传感器采集       │
└────────────────────────────────────┘
```

::: tip
**一个容易混淆的点**：ESP32 能跑什么？ESP32 是 MCU（没有 MMU），不能跑 Linux。但它通常运行 FreeRTOS（ESP-IDF 框架内置）。网上说的"ESP32 Linux"要么是错的，要么指的是别的含义。
:::

## GNU/Linux 是什么

很多初学者好奇"Linux"到底指什么。严格来说，我们日常说的"Linux 系统"其实是 **GNU/Linux**：

- **Linux**：只是一个**内核**（Kernel）。内核管理硬件（CPU、内存、硬盘、网卡），提供最基本的系统服务。
- **GNU**：一套**自由软件工具集**，包括 GCC（编译器）、Bash（命令行）、Coreutils（基本命令如 `ls`、`cp`）。这些工具是操作系统"用户能用"的部分。

内核 + 工具集 = 一个完整的操作系统。就像汽车引擎（Linux）+ 方向盘、座椅、仪表盘（GNU）= 一辆能开的车。

## Linux 发行版是什么

GNU/Linux 只是"原材料"。不同的人或组织把这些原材料打包成**发行版（Distribution）**，预装不同的软件、使用不同的包管理器（安装/卸载软件的工具）。

常见的发行版：

| 发行版 | 特点 | 适合谁 |
|--------|------|--------|
| Ubuntu | 用户友好、资料最多 | 新手 |
| Arch Linux | 滚动更新、高度自定义、KISS 原则 | 喜欢折腾的用户 |
| Debian | 极其稳定、保守 | 服务器 |
| Fedora | 新技术先行 | 开发者 |

::: tip
**Arch Linux 的 KISS 原则**：Keep It Simple, Stupid（保持简单，傻瓜）。这里的"简单"不是对用户简单，而是对系统设计者简单——系统不替你做决定，由你自己掌控一切。Arch 的用户手册（Arch Wiki）被公认为最好的 Linux 文档之一。
:::

## 为什么竞赛控制不用 Linux

```
        实时性                       功能丰富度
        高 ▲                          高 ▲
          │                             │
    裸机   │    实时领域                 │   Linux
    RTOS   │   （竞赛在这里）             │   服务器/桌面
          │                             │
          │            Linux             │         裸机/RTOS
          │          （不适合）           │        （简单够用）
          │                             │
        低 ▼                          低 ▼
        确定性响应                      功能复杂度
```

Linux 不适合竞赛小车的原因很具体：

| 问题 | 说明 |
|------|------|
| 响应时间不确定 | Linux 不是实时操作系统。当内核正在做别的事（写硬盘、网络中断），你的控制任务可能被延迟几十甚至几百毫秒——小车已经撞墙了。 |
| 启动太慢 | Linux 从加电到能跑你的程序需要几秒到十几秒。而 STM32 裸机/RTOS 上电几十毫秒就进 main() 了。竞赛裁判可不会等你开机。 |
| 功耗太高 | 应用处理器动辄几瓦，而 STM32 在毫瓦级别。小车用电池供电，功耗直接决定能跑多久。 |
| 不必要的复杂 | 小车控制只需要：读传感器 → 算 PID → 控制电机。不需要文件系统、虚拟内存、多用户权限管理这些 Linux 提供的东西。 |
| 成本高 | 一个能跑 Linux 的核心板至少几十元，而 STM32F103C8T6 只要几块钱。 |

## 你该关注什么

```


裸机 ──────────▶ FreeRTOS ──────────▶ Linux
（基础，必会）    （进阶，强烈推荐）    （了解即可）

竞赛主战场 ──────▶
```

::: danger
**不要被"Linux 更高级"的说法迷惑。** 技术的选择永远是"合适的才是最好的"。对于竞赛小车控制这个具体场景，裸机和 FreeRTOS 才是正确的答案。把这个关系搞清楚，比会装十个 Linux 发行版都有意义。
:::

## 小结

Linux 是一个功能强大的通用操作系统，运行在带 MMU 的应用处理器上。STM32 和 MSPM0 是微控制器，没有 MMU，不能跑 Linux。竞赛小车是实时控制系统，需要确定性响应和快速启动——这两点 Linux 都做不到。你的学习路径应该是：先吃透裸机，再掌握 FreeRTOS，知道 Linux 的存在和它的定位就够了。
