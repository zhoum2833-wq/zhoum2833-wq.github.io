# 单片机是如何运作的

当你按下电源开关，单片机的内部到底发生了什么？这一章我们一步一步来看。

## 上电三步骤

把单片机想象成一个刚睡醒的人：先伸懒腰（初始化身体状态），再穿衣服（准备好工作环境），然后开始干活（执行你的程序）。

```
电源接通
    │
    ▼
┌─────────────────────────────┐
│  1. 硬件复位                │
│  CPU 从 Flash 地址          │
│  0x08000000 读取第一条指令  │
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  2. 启动文件执行            │
│  · 设置栈指针 (SP)          │
│  · 复制 .data 段到 RAM      │
│  · 清零 .bss 段             │
│  · 调用 main()              │
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  3. main() 函数运行         │
│  你的程序开始执行            │
│  通常是一个 while(1) 死循环  │
└─────────────────────────────┘
```

### 第一步：硬件读取第一条指令

单片机内部有一块 Flash 存储器（就像 U 盘，断电后内容不丢失）。你的程序就烧录在里面。CPU 知道程序一定从地址 `0x08000000` 开始——这是厂商设计芯片时就定好的规矩，就像一本书的正文总是从第 1 页开始。

### 第二步：启动文件做了什么

启动文件（startup file）是用汇编语言写的一段"开场白"。它的任务很简单：

1. **设置栈指针（SP, Stack Pointer）**：栈是程序运行时用来存放临时数据的区域。栈指针告诉 CPU："临时数据放这里"。想象栈就是一叠盘子，最后放上去的最先拿下来。

2. **复制 .data 段**：你的代码里如果有 `int speed = 100;` 这样的全局变量（有初始值），它的初始值 `100` 存储在 Flash 里。启动文件把它复制到 RAM 中，因为程序运行时变量必须在 RAM 里才能被修改。Flash 只读，RAM 可读可写——就像笔记本（RAM）可以反复擦写，而印刷书（Flash）只能读。

3. **清零 .bss 段**：如果你的全局变量没有初始值，比如 `int count;`，启动文件会把它所在的那片 RAM 全部写成 0。这确保变量从一个确定的状态开始，不会读到上次断电残留的随机值。

4. **调用 main()**：准备工作就绪，跳转到你的 `main()` 函数，你的程序正式开始。

::: tip
**记住这个顺序**：Flash（程序存储）→ 启动文件（布置环境）→ main()（你的代码）。每次上电都是这样。
:::

## 什么是程序计数器（PC）

PC（Program Counter）是 CPU 内部一个特殊的**寄存器**。它永远保存着**下一条要执行的指令的地址**。

把 PC 想象成你读书时的食指：你的食指指到哪个字，你的眼睛就读哪个字。CPU 的 PC 指到哪个地址，CPU 就执行哪个地址里的指令。执行完一条，PC 自动加一，指向下一条——如此循环往复。

## 为什么是 while(1) 死循环

你写的 main 函数通常长这样：

```c
int main(void) {
    // 初始化代码（只执行一次）
    SystemClock_Config();
    GPIO_Init();

    // 主循环（反复执行，永不退出）
    while (1) {
        // 读取传感器
        // 处理数据
        // 控制电机
    }
}
```

为什么必须是一个死循环？因为在桌面电脑上，程序运行完可以 `return` 给操作系统。但裸机（bare-metal）没有操作系统，如果你的 `main()` 函数执行完了，CPU 会跑到哪里去？它会读到一个未知地址里的随机数据，把它当指令执行——结果就是程序跑飞，单片机"死机"。

::: danger
**裸机程序中 main() 绝不能 return！** 一旦退出，CPU 行为不可预测。`while(1)` 不是在浪费 CPU，而是在保护程序的运行。
:::

## 一个完整的例子

```c
#include "stm32f1xx_hal.h"

int speed = 50;        // .data 段 —— 初始值从 Flash 复制到 RAM
int counter;           // .bss 段 —— 启动时清零

int main(void) {
    HAL_Init();                    // 初始化 HAL 库
    SystemClock_Config();          // 配置时钟（比如 72MHz）

    // --- 以上初始化只执行一次 ---

    while (1) {                    // 从这里开始反复执行
        counter++;                 // counter 每轮加 1
        if (counter > 1000) {
            HAL_GPIO_TogglePin(GPIOC, GPIO_PIN_13);  // 翻转 LED
            counter = 0;
        }
    }
}
```

上电后，这个程序的执行路径是：

1. CPU 从 `0x08000000` 取指令
2. 启动文件：设置栈 → 把 `speed = 50` 写入 RAM → 把 `counter` 清零
3. 跳转到 `main()`
4. `HAL_Init()` 和 `SystemClock_Config()` 各执行一次
5. 进入 `while(1)`，`counter` 不断加一，每 1001 次翻转一次 LED

## 寄存器是什么

寄存器是 CPU 内部**最快**的存储单元。它们不是 RAM，也不是 Flash——它们就在 CPU 内部，访问速度是纳秒级别。

| 存储类型 | 速度 | 容量 | 用途 |
|---------|------|------|------|
| 寄存器 | 最快（1 个时钟周期） | 极少（几十个） | CPU 正在操作的数据 |
| RAM | 快 | 中等（几 KB~几 MB） | 运行时的变量 |
| Flash | 慢（相对） | 大（几十 KB~几 MB） | 存储程序和常量 |

**外设寄存器**则不同——它们不在 CPU 内部，而在外设模块里。你往这些寄存器写数据，就是在给硬件下命令。比如往某个寄存器写 `1`，对应的 LED 就亮了。

::: tip
**一句话总结**：CPU 从 Flash 取指令，在 RAM 中操作数据，通过寄存器控制硬件。上电 → 启动文件 → main() → while(1) 永远跑下去。
:::

## 小结

单片机从按下电源到执行你的代码，经历了上电复位→读向量表→初始化→进 main() 的完整流程。理解这个启动过程，你就知道为什么代码烧进去能跑、为什么 BOOT0 跳线能切换下载模式。
