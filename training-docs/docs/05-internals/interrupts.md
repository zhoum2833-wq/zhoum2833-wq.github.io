# 中断 — 别让 CPU 傻等

## 生活类比：做作业与门铃

你正在房间写作业（执行主循环）。突然门铃响了（硬件事件发生）。你放下笔（保存现场），起身走到门口开门（执行中断服务程序 ISR），处理完访客的事情（处理完毕），回到桌前拿起笔继续写作业（恢复现场）。

这个过程中，**你不需要每隔一秒钟抬头看门口有没有人**。门铃会在需要的时候通知你。这就是中断的核心思想——让硬件在有事的时候主动通知 CPU，而不是让 CPU 反复去问"好了没？好了没？"。

## 轮询的悲剧

没有中断时，CPU 只能用**轮询（Polling）**方式检测事件：

```c
// 轮询方式：CPU 反复检查按键
while (1) {
    if (HAL_GPIO_ReadPin(GPIOB, GPIO_PIN_0) == GPIO_PIN_RESET) {
        // 按键按下了！
        HAL_GPIO_TogglePin(GPIOC, GPIO_PIN_13);
    }
    // 做其他事情...
    // 但如果其他事情耗时太长，按键响应就会延迟！
}
```

轮询的问题很明显：

| 轮询 | 中断 |
|------|------|
| CPU 反复主动检查 | 硬件主动通知 CPU |
| 99% 的检查是白费力气 | CPU 只在需要时响应 |
| 响应延迟不确定（取决于循环速度） | 响应在微秒级 |
| 简单、容易理解 | 稍复杂，但极其重要 |
| 适合不重要、不紧急的事件 | 适合需要立即响应的事件 |

想象你在等一个重要的快递。轮询就是每隔一分钟开门看看快递到了没有——一天开门 1440 次，其中 1439 次白看。中断就是快递员按门铃——你只在需要时起身。

## 中断向量表

中断向量表是一张**函数地址表**，存储在 Flash 的固定位置。当某个中断发生时，硬件自动查这张表，找到对应的处理函数地址，然后跳转过去。

```
中断向量表（概念示意）
┌──────────────────┬────────────────────────┐
│  中断源           │  处理函数地址           │
├──────────────────┼────────────────────────┤
│  复位             │  0x08000000 (启动文件) │
│  NMI（不可屏蔽）   │  NMI_Handler 地址      │
│  HardFault       │  HardFault_Handler 地址 │
│  SysTick         │  SysTick_Handler 地址   │
│  EXTI0 (按键中断) │  EXTI0_IRQHandler 地址  │
│  TIM2 (定时器中断)│  TIM2_IRQHandler 地址   │
│  USART1 (串口中断)│  USART1_IRQHandler 地址 │
│  ...             │  ...                   │
└──────────────────┴────────────────────────┘
```

::: tip
**向量**这个词听起来很数学，在这里它就是"指向目标的方向"——每个表项存着一个函数的地址，告诉 CPU 发生这种中断时该往哪里跳。
:::

## 中断优先级

不是所有中断都平等。紧急的事情可以**打断**不太紧急的事情。

比如：你正在处理串口数据（USART 中断），突然电机过流保护触发（更高优先级中断）。CPU 会暂停串口处理，先去处理电机保护，处理完再回来继续串口处理。这叫**中断嵌套**。

STM32 的中断优先级用 0~15 表示（数字越小优先级越高）：

```c
// 设置 EXTI 中断优先级
HAL_NVIC_SetPriority(EXTI0_IRQn, 2, 0);   // 优先级 2
// 设置定时器中断优先级（比按键更紧急）
HAL_NVIC_SetPriority(TIM2_IRQn, 1, 0);    // 优先级 1（更高！）
```

## 完整的执行流程

```
主循环运行中...
    │
    ▼
[硬件事件发生]（比如按键按下、定时器到时间）
    │
    ▼
[CPU 自动保存现场]（把当前寄存器的值压入栈）
    │
    ▼
[查中断向量表，跳转到 ISR]
    │
    ▼
[执行 ISR 代码]
    │
    ▼
[CPU 自动恢复现场]（从栈里弹出之前保存的寄存器值）
    │
    ▼
[回到主循环刚才被打断的位置，继续执行]
```

::: danger
**ISR 里不能干重活！** 中断服务函数必须尽可能短小精悍。如果你在 ISR 里做大量计算或者调用 `HAL_Delay()`，CPU 长时间陷在 ISR 里出不来，其他中断全被挡住，系统就卡死了。ISR 的原则：进来 → 快速处理 → 立即退出。重活留给主循环。
:::

## 实战代码：外部中断让按键即时响应

```c
// 按键中断初始化（STM32 HAL）
void Button_EXTI_Init(void) {
    GPIO_InitTypeDef GPIO_InitStruct = {0};

    // 使能时钟
    __HAL_RCC_GPIOB_CLK_ENABLE();

    // 配置 PB0 为输入，上升沿/下降沿都触发中断
    GPIO_InitStruct.Pin = GPIO_PIN_0;
    GPIO_InitStruct.Mode = GPIO_MODE_IT_RISING_FALLING;
    GPIO_InitStruct.Pull = GPIO_PULLUP;
    HAL_GPIO_Init(GPIOB, &GPIO_InitStruct);

    // 设置中断优先级并使能
    HAL_NVIC_SetPriority(EXTI0_IRQn, 2, 0);
    HAL_NVIC_EnableIRQ(EXTI0_IRQn);
}

// 中断服务函数 —— 放在 stm32f1xx_it.c 里
void EXTI0_IRQHandler(void) {
    // 检查是否是 PB0 触发的中断
    if (__HAL_GPIO_EXTI_GET_IT(GPIO_PIN_0) != RESET) {
        __HAL_GPIO_EXTI_CLEAR_IT(GPIO_PIN_0);  // 清除中断标志（必须！）
        HAL_GPIO_TogglePin(GPIOC, GPIO_PIN_13); // 翻转 LED
    }
}
```

这个代码做的事情：按键接在 PB0 引脚上。按下或松开按键时，引脚电平变化触发 EXTI0 中断。CPU 立刻暂停当前工作，执行 `EXTI0_IRQHandler`。函数里翻转 PC13 上的 LED，然后清除中断标志位（必须清除，否则会反复触发），最后返回。整个过程在微秒内完成，你感觉不到任何延迟。

::: warning
**别忘了清除中断标志！** 中断标志是硬件记录"我触发过中断"的一个小标记。不清除的话，退出 ISR 后硬件以为中断还没处理完，会立刻再触发一次——导致 ISR 被反复调用，程序卡死。
:::

## 三种常见中断应用场景

| 中断类型 | 触发条件 | 用途 |
|----------|----------|------|
| 外部中断（EXTI） | 引脚电平变化 | 按键响应、编码器计数 |
| 定时器中断（TIM） | 定时器到设定值 | 精确延时、LED 闪烁、PID 计算周期 |
| 串口中断（UART） | 收到数据 | 接收上位机命令、传感器数据 |

## 小结

中断让单片机从"傻等"变成"即时响应"。轮询是 CPU 反复主动去问，中断是硬件主动通知。ISR 要短小精悍，优先级高的中断可以打断优先级低的。理解中断，你就理解了单片机对"实时性"的保障。
