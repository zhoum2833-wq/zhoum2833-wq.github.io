# 团队分工

## 没有纯粹的"硬件人"和"软件人"

很多新人组队时的第一句话是："我搞硬件，你搞软件。"这听起来很合理——分工明确。但在嵌入式竞赛中，**这种分法行不通**。

原因很简单：嵌入式系统的硬件和软件是深度耦合的。比如：

- 写 PWM 控制电机的人需要知道电机的电气特性（电压、电流、频率）
- 调传感器的人需要知道芯片的时序要求（I2C 的 SCL 频率、UART 的波特率）
- 测编码器的人需要知道定时器的"编码器模式"是怎么工作的

**每个人都得碰硬件，每个人都得写代码。**

## 推荐的分工方式：按子系统分

不要把"硬件"和"软件"分开——把"功能"分开。

### 三人团队推荐分工

```
队员 A：感知与控制
├── 灰度传感器（接线 + 读取代码）
├── 陀螺仪 JY61P（接线 + 数据解析）
├── 编码器（接线 + 速度/里程计算）
├── 其他传感器（超声波、红外等）
└── PID 控制器（核心控制算法）

队员 B：执行与通信
├── 电机驱动 TB6612（接线 + PWM 控制）
├── UART / I2C / SPI（通信协议代码）
├── 无线串口（蓝牙调试）
├── OLED 显示（调试信息输出）
└── 电池电压检测

队员 C：队长 / 集成与策略
├── 系统架构设计（整体方案选型）
├── 状态机（任务调度逻辑）
├── 接口协调（A 和 B 之间的数据格式）
├── 竞赛报告（技术文档撰写）
└── 最终集成 + 全系统调试
```

### 学习阶段的分工

在备赛学习阶段，可以按"偏向"来分：

| 角色 | 偏向硬件 | 偏向软件 |
|------|---------|---------|
| 重点学习 | GPIO 配置、电源电路、PCB 设计、焊接 | 通信协议、中断、DMA、PID 算法 |
| 但也得懂 | 能读懂简单的控制代码 | 能用万用表量电压、看原理图 |
| 共同需要 | **都会** GPIO 配置、定时器使用、串口调试 | |

::: tip 队长的角色
队长不一定是技术最强的——但一定是最懂"全局"的。队长需要：
- 理解每个人的工作在干什么
- 能判断"这里有问题，是谁负责的子系统"
- 做技术决策（选 A 方案还是 B 方案）
- 确保两块代码能拼到一起不冲突（接口协商）
:::

## 分工中的常见问题

### 问题 1：接口不匹配

```
队员 A 说："我发的速度值是 0-1000"
队员 B 说："我接收的速度值是 0-100"
→ 代码拼起来发现不对，凌晨三点排查
```

**解决方法：** 在开始写代码之前，**先约定接口文档**：

```c
// 接口约定文档（代码注释形式）
// ==================== 电机控制接口 ====================
// 函数：motor_set_speed(int16_t left, int16_t right)
// 参数范围：left, right ∈ [-1000, 1000]
//   正数 = 前进，负数 = 后退
//   1000 = 100% 占空比
// 调用者：队员 A（控制模块）
// 实现者：队员 B（电机驱动模块）
// ==================== 接口约定结束 ====================
```

### 问题 2：一个人卡住全队等

队员 A 的陀螺仪调了三天还没通 → 队员 B 在边上干等。

**解决方法：** 
- 先做可以独立测试的部分。比如电机驱动可以先用"固定速度前进"测试，不需要等 PID 调完。
- 用**模拟数据**先跑通逻辑。A 还没调通陀螺仪？先用 `angle = 90` 这个常量写 B 的转向代码。

### 问题 3：重复造轮子

A 写了一个 `delay_ms()`，B 也写了一个 `delay()`——功能一样，名字不同，占两份空间。

**解决方法：** 建立一个共享的 `common.h` 和 `common.c`，存放公共函数。每个人新增的工具函数第一时间放进去。

```c
// common.h — 公共工具函数
#ifndef COMMON_H
#define COMMON_H

void delay_ms(uint32_t ms);
void delay_us(uint32_t us);

#endif
```

## 日常协作模式

| 阶段 | 做什么 | 谁做 |
|------|--------|------|
| 方案设计 | 讨论、画框图、定接口 | 全员参与，队长拍板 |
| 各自开发 | 每人独立调试自己的子系统 | A + B 并行，C 写框架 |
| 接口联调 | 分步集成：先传感器→再电机→再闭环 | A + B 结对接，C 旁观检查 |
| 整机调试 | 在赛道上跑，发现问题 | 全员 |
| 写报告 | 整理文档 | C 主笔，A + B 补充 |

## 两人团队怎么办

如果只有两个人：

```
队员 A：控制 + 电机（从上层的控制算法到底层的电机驱动）
队员 B：传感器 + 通信 + 报告（所有感知 + 调试工具）
```

或者更简单的分法：

```
队员 A：让车跑起来（电机 + 传感器 + 控制）
队员 B：让车聪明起来（状态机 + 策略 + 调试 + 报告）
```

---

> **团队分工的本质不是"划清界限"，而是"减少协调成本"。** 好的分工让每个人大部分时间都在自己的子系统里工作，只需要在接口处沟通。
