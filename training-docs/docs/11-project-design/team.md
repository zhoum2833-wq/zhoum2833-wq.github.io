# 团队分工

## 核心策略

**不要试图教会每个人所有知识。** 给队友"配方"而不是"原理"——告诉他怎么做能跑通，而不是为什么。

## 队长负责

队长写核心代码、搭系统框架、做最终集成。具体来说：

- 系统架构：选什么 MCU、用什么通信协议、IO 怎么分配
- 核心控制：PID 控制器、状态机、赛道策略
- 工程框架：搭好初始工程模板，配好 CubeMX、时钟、外设初始化
- 接口定义：把每个模块的 `.h` 文件写好（函数声明、参数、返回值）
- 最终集成：调试、写报告

队长的代码量大概占 60-70%。

## 队员 A：传感器 + 显示

负责传感器接线和读数、OLED 显示调试信息。

入门顺序：先学会接灰度传感器、OLED，能跑通"读传感器值 → 打印到 OLED"。然后学编码器、陀螺仪。

## 队员 B：电机 + 通信

负责电机驱动接线和 PWM 控制、无线串口调试。

入门顺序：先学会让电机转起来（PWM 调速），再学无线串口收发。不需要理解 UART 协议原理——照着模板接好线、改个波特率就行。

## 队员入门路径（前两周）

| 周 | 队员任务 |
|----|---------|
| 第 1 周 | 在队长搭好的工程模板上，改代码、烧录、看结果。不要求理解 |
| 第 2 周 | 能独立接线、独立烧录、独立改参数。遇到问题先问 AI |

前两周不要求队员懂"为什么"——先让 LED 闪起来、电机转起来，建立信心。

## 关键原则

**队长写给队员的，不是教程，是可执行的步骤。** 比如：

```
✅ 好的交付：
1. 打开 2WD_Car/MDK-ARM/Project.uvprojx
2. 找到 bsp_led.c 的第 15 行
3. 把 GPIO_PIN_13 改成 GPIO_PIN_0
4. 按 F7 编译，按 F8 下载
5. 看 PC0 上的 LED 有没有闪

❌ 差的交付：
"你配一下 GPIO，把 LED 引脚改了就行"
```

**每个队员只碰自己负责的模块文件，不碰别人的代码。** 接口通过 `.h` 文件约定好，实现塞在对应的 `.c` 里。

## 日常协作

| 时间段 | 做的事 |
|--------|--------|
| 开始工作前 | 队长更新公共框架，队员 pull 最新代码 |
| 开发中 | 各改各的模块，互不干扰 |
| 遇到问题 | 先贴代码和错误信息问 AI，解决不了再找队长 |
| 功能完成 | 队长 review 代码、合并、统一测试 |

## 两人团队

```
队长：控制 + 电机 + 集成（核心路径）
队员：传感器 + 通信 + 报告（辅助路径）
```

队员的目标是"能独立采集传感器数据、发调试信息"，不要求会写控制算法。

---

> **团队分工的本质不是"划清界限"，而是"减少协调成本"。** 好的分工让每个人大部分时间都在自己的子系统里工作，只需要在接口处沟通。
