# AI 辅助嵌入式开发

## 嵌入式开发里 AI 能干什么？

你可能觉得 AI 编程就是"让 AI 写一个网页"、"让 AI 写一个 Python 脚本"。但嵌入式开发——写 C 语言、操作寄存器、看芯片手册——AI 能帮上忙吗？

答案是：**能。而且越复杂的嵌入式项目，AI 的帮助越大。**

4WD_Car 项目就是全程用 Claude Code（一个运行在终端/VS Code 里的 AI 编程助手）辅助开发的。这一节分享实际经验。

## AI 在 4WD_Car 项目中的实际应用

### 1. 写 BSP 驱动

以前写一个新外设的 BSP 驱动，你需要：翻芯片手册查引脚复用功能、查 RCC 时钟映射、查 GPIO 配置寄存器、查中断向量表。这一套流程每个外设都要走一遍，耗时且容易配错。

有了 AI：
> "参考 bsp_uart.c 的风格，帮我写一个 bsp_imu.c，JY61P 姿态传感器接在 USART2（PD5/PD6），波特率 115200，每帧 11 字节，需要解析欧拉角。"

AI 会生成完整的 `bsp_imu.c` 和 `bsp_imu.h`，包括引脚宏定义、RCC 时钟使能、GPIO 复用配置、USART 初始化、NVIC 中断配置、数据解析函数。你只需要检查引脚是否正确、微调参数。

### 2. 写 PID 控制算法

PID 控制的数学原理很简单，但从数学公式到 C 代码有很多细节：增量式还是位置式？积分限幅怎么做？输出饱和怎么处理？

> "帮我写一个增量式 PID 四路独立速度环，输入是编码器计数/周期，输出是 PWM 占空比 -99~+99，包含积分分离和输出限幅。"

AI 不仅生成代码，还会解释为什么增量式比位置式更适合电机控制（无积分饱和问题）、为什么需要积分分离（防止长时间误差累积）。

### 3. Debug 和查 Bug

这是 AI 最能帮上忙的地方。遇到诡异的 bug——

编码器 E2 读数总是跳变？把代码贴给 AI：
> "E2 编码器用软件 EXTI 计数，但高速时脉冲丢失严重，每次读的值会跳变几百。可能是什么原因？"

AI 分析后指出：软件 EXTI 在高速脉冲下响应不及时，建议改用硬件编码器模式。你一拍脑袋——TIM5 还有空通道！

### 4. 理解陌生代码

面对 ST 官方的 HAL 库源码（几百行一个函数），与其硬啃，不如让 AI 帮你解读：
> "解释 STM32F4 HAL 的 UART 中断接收流程，起点是 USART1_IRQHandler。"

AI 会梳理出完整的调用链：`USART1_IRQHandler → HAL_UART_IRQHandler → UART_Receive_IT → UART_RxISR_8BIT → HAL_UART_RxCpltCallback`，并说明每一环做了什么。

### 5. 写辅助 Python 脚本

比赛时你可能需要：采集 500 张数字图片、训练一个 CNN、导出 ONNX 模型。这些任务用 Python 比 C 高效得多，AI 写 Python 更是"秒出"。

4WD_Car 的 `tools/` 目录下大部分 Python 脚本都是 AI 辅助完成的——数据采集、模型训练、ONNX 导出、数据标注工具。

## 怎么用好 AI？

### 1. 写清楚 CLAUDE.md

CLAUDE.md 是给 AI 看的"项目说明书"。4WD_Car 的 CLAUDE.md 包含：
- 项目架构（三层结构、调用规则）
- 引脚分配表
- 编码规范（命名风格、文件头格式）
- 初始化顺序
- 注意事项（已踩过的坑）

有了这个文件，AI 就像一个已经接手项目两周的队友，你不需要每次都从头解释项目背景。

### 2. 给 AI 看参考代码

直接说"帮我写一个电机驱动"太模糊了。给它一个已有的 BSP 模块做参考：
> "参考 bsp_uart.c 的风格和模板，帮我写 bsp_xxx.c"

AI 会模仿你的编码风格（缩进、命名、注释格式），不会造出风格迥异的代码。

### 3. 分步骤、别一次要太多

```
❌ "帮我写整个小车的全部代码"
✅ "先写 bsp_motor.h，定义好引脚宏和 API"
✅ "好了，现在写 bsp_motor.c 的 Init 函数"
✅ "现在写 SetSpeed 函数"
```

嵌入式开发的上下文非常密集（寄存器、引脚、时序……），一次给太多会让 AI 出现"幻觉"——编造不存在的引脚、配置不兼容的参数。分步骤来，每步验证再继续。

### 4. 让 AI 帮你做代码审查

写完一段功能后：
> "检查这段代码有没有潜在的 bug，特别是中断安全和边界条件。"

AI 会发现你可能遗漏的问题：ISR 里用了非原子操作、数组越界、除零风险、定时器重装载值溢出等。

## 工作流

```
1. 你跟 AI 描述需求，AI 生成初始代码
2. 你把代码放进 VS Code/Keil5，编译运行
3. 遇到问题，把错误信息或异常行为描述给 AI
4. AI 分析原因，建议修改
5. 重复 2-4 直到功能正常
6. 让 AI 帮你写这段代码的注释和文档
```

::: tip 关键心态
AI 不是"全自动代码生成器"，它是你的**结对编程搭档**。你是主驾驶，负责做决策、验证结果；AI 是副驾驶，负责写初稿、查资料、提建议。最终代码质量由你把关。
:::

## 小结

AI 工具在嵌入式开发中真正改变的是**效率**——新写一个 BSP 驱动从 2 小时变成 15 分钟，查一个寄存器配置 bug 从"翻一天手册"变成"贴代码问 AI"。但前提是你自己**懂**单片机的原理——AI 可能犯错（编造寄存器地址、用不存在的 HAL 函数），你需要能判断它说的对不对。
