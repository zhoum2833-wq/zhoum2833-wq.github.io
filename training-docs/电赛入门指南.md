# 什么是单片机

**单片机**顾名思义，**单片微型计算机**，它把所有功能压缩到唯一一块芯片里，这里的 “单” 就是单一块、单个的意思，**而不是单个CPU**的意思

从硬件结构来看，单片机整体由四大核心部分构成：CPU内核、片内存储器、片上外设以及时钟、复位、电源等基础电路。

单纯将各个模块分开讲太枯燥了，我就直接讲单片机是怎么运行的吧。

# 单片机是怎么运行的

简而言之，单片机上电复位、启动时钟后，CPU 从 Flash 起始地址循环执行取指、译码、执行操作；

平时按主循环顺序运行，遇到中断会暂停当前程序优先处理紧急任务；

运行时代码存放于 Flash，临时数据存放于 RAM，CPU 通过寄存器控制各类外设完成控制、通信、采集功能。

## 一、上电启动阶段

通电后电源模块给芯片所有模块供电，复位电路产生复位信号，CPU 内核、存储器、外设全部恢复初始状态，寄存器清零。
时钟电路开始工作，生成稳定时钟脉冲，整个芯片所有硬件都依靠这个节拍同步工作。
CPU 自动跳转到Flash 程序存储器的起始地址，准备读取程序。

## 二、取指 — 译码 — 执行（CPU 内核循环工作核心）

单片机程序本质是不断重复三步流水线操作：

### 取指令

程序计数器 PC 保存当前指令地址，CPU 根据地址从 Flash 读出一条指令，存入指令寄存器。

### 译码

控制器 CU 解析这条指令，判断是加减运算、读写 IO、跳转、中断等哪一类操作。

### 执行

根据译码结果驱动对应硬件完成动作：

运算指令：交给 ALU 做加减乘除、逻辑运算；

读写指令：访问 RAM 变量或 GPIO、外设寄存器；

跳转指令：修改 PC 的值，实现 if、for 循环、函数调用；

执行完成后 PC 自动自增，读取下一条指令，循环往复。

## 三、两种运行模式：裸机顺序执行 + 中断实时响应

### 主循环（裸机常态）

没有操作系统时，程序写完初始化代码后，会进入一个无限 while (1) 主循环，内部代码从上到下依次顺序运行。

### 中断机制（打断顺序流程，实现实时控制）

当外部触发信号到来（按键按下、定时器计时到、串口收到数据），硬件自动触发中断：

CPU 立刻暂停当前正在执行的代码，保存现场（寄存器、PC 值压入堆栈）；

跳转到对应的中断服务函数执行处理任务；

中断处理完毕，恢复之前保存的现场，回到被打断的地方继续运行原来的程序。

**中断机制存在多处容易出错的关键点：**

栈溢出、未清除中断标志、共享全局变量缺少 volatile、中断服务函数执行耗时过长等。

#### 堆栈空间有限，中断嵌套 / 递归容易栈溢出

进入中断会把一大堆寄存器压栈，多层中断嵌套、中断里再调用大数组、递归，栈会快速占满，直接死机、程序跑飞。
单片机 RAM 栈很小，不像电脑内存充足，这是**最常见 bug**。

#### 中断服务函数不能做耗时操作

中断优先级高，如果在中断里写长延时、while 等待串口收发、循环查询，主程序会卡死，实时性完全失效。
原则：中断只做标记、采集、简单赋值(**设置标志位**)，复杂逻辑丢到 while (1) 主循环处理。**printf 底层依赖串口阻塞发送，放在中断内会大幅拉长中断执行时间，丢失其他中断事件。**

#### 忘记清除中断标志位，造成重复进中断

定时器溢出、外部按键、串口接收中断触发后，硬件会置起中断标志；
如果代码不手动清零标志，退出中断后会立刻再次触发中断，程序卡在中断里出不来。

#### 全局变量不加 volatile 修饰（C语言）

主循环和中断同时读写同一个全局变量时，编译器会优化缓存到寄存器，主循环永远读取不到中断更新后的值。
解决方案：共享变量必须加 volatile。


## 四、数据存储配合运行

程序代码、固定常量存放在 Flash，上电只读，不会丢失；
程序运行产生的变量、中间计算数据、堆栈临时数据存放在 RAM，读写速度快，断电清空；
程序运算、外设交互的数据都需要在 RAM 中中转。

## 五、外设协同工作

CPU 通过读写外设寄存器控制片上外设：

控制 GPIO 输出高低电平，读取按键输入；

配置定时器生成延时、PWM 波形；
通过 UART/SPI/I2C 和外部模块收发数据；

高速数据搬运可开启 DMA，不用占用 CPU，自动完成内存数据传输。

## 六、异常与复位（保障稳定运行）

程序跑飞、运算溢出、时钟故障时会触发硬件异常；若程序长时间卡死，看门狗定时器溢出会自动触发复位，芯片重新从头启动程序。

# RAM 与 ROM（Flash）

**ROM** （只读存储器，单片机里主要是 Flash），用于存放提前烧录好的程序代码、固定常量、查表数据；断电后数据不会丢失。正常运行过程中仅支持读取，修改需要专用下载器或特殊擦写指令。

**RAM**（随机存取存储器）相当于是内存（功能等同于电脑内存），程序运行时临时存放全局变量、局部变量、运算中间数据，同时划分出一段区域作为堆栈（栈 Stack）。
函数调用、中断响应、递归运算都会向栈中压入数据；如果递归深度过大、局部数组定义过大，会超出栈分配空间，引发栈溢出，造成程序死机、跑飞，是嵌入式开发中十分常见的内存故障。

# 定时器与 PWM

## 定时器是什么

定时器（Timer）本质就是**一个自动数数的计数器**。每个时钟周期(CPU 最小工作节拍)，计数器加 1。加到某个值——触发事件（中断、或改变引脚电平）。

```
  时钟脉冲: 方波 

  计数器:   0→1→2→3→...→(ARR/自动重装载值)→0→1→...

  预分频器（PSC）:设CPU主频是72MHz，如果设置PSC为71，那么CPU的主频经过预分频器后的频率为72MHz÷(71+1)=1MHz

  自动重载值（ARR）: 计数器从 0 数到 ARR，然后归零
```

## 定时器功能

### PWM：定时器最常用的功能

PWM（Pulse Width Modulation，脉冲宽度调制）用定时器的比较输出功能产生方波。
占空比 = 高电平占脉冲周期的百分比

PWM常用于电机调速，开关电源、DC-DC 稳压，舵机角度控制

### 输入捕获：定时器也能"听"

除了输出 PWM，定时器还能**测量**外部信号的频率和脉宽。比如测编码器脉冲的频率算速度，或者测超声波回波的时间算距离。

原理：捕获引脚上电平变化时的计数值。两次上升沿之间的计数值差 = 信号周期。

# GPIO 与片内外设

单片机的引脚可以配置为输入或输出，这是它和外部世界交互的基础。这种可编程配置的引脚就叫 GPIO（通用输入输出）。

## 输出

输出模式有两种：

**推挽输出**是最常用的模式。它能主动输出高电平或低电平，驱动能力强。点灯、发信号都用它。

**开漏输出**只能主动拉低，不能主动拉高。要输出高电平，必须外接上拉电阻。I2C 总线和多设备共享信号线时用它。

## 输入

输入模式用来读取外部信号——按键有没有按下、传感器输出是高是低。如果引脚悬空（什么都没接），电压不定，读到的是什么完全看运气。

解决方法：启用内部上拉或下拉电阻(上拉时引脚默认高电平，下拉时默认低电平)，让引脚在悬空时有一个确定的默认电平。

## 片内外设

同一根引脚可以复用为外设的专用接口。以下是常用外设与常用用途：

| 外设 | 作用 |
|------|------|
| UART | 两个芯片之间串行通信 |
| I2C | 两根线挂多个传感器 |
| SPI | 四线高速通信 |
| ADC | 把电压转成数字量 |
| PWM | 通过占空比控制电机转速|
| Timer | 精确计时|

# ADC 与 DAC

单片机内部全是 0 和 1（数字）。但真实世界不是——温度是连续变化的，声音是连续的波形，光强是连续变化的亮度。要让单片机和真实世界对话，就需要两座桥梁：

- **ADC**（模数转换器）：模拟 → 数字（把电压变成数字）
- **DAC**（数模转换器）：数字 → 模拟（把数字变成电压）

## ADC 的核心参数

| 参数 | 含义 | STM32F103 典型值 | 怎么理解 |
|------|------|-----------------|---------|
| **分辨率** | 能把电压分成多少份 | 12 位 | 2^12 = 4096 份，3.3V / 4096 ≈ 0.8mV 每份 |
| **参考电压** | ADC 量程的上限 | 3.3V | 输入超过 3.3V → 读出来是 4095（饱和了） |
| **采样率** | 一秒钟能测多少次 | 最大 1MHz | 够测音频（44.1kHz），电赛里的传感器最多几十 Hz |
| **通道数** | 能同时/轮询测几路信号 | 最多 16 路 | 一般的 MCU 有 8~16 个 ADC 输入引脚 |

## ADC

**轮询模式**，CPU 等着 ADC 采完。如果你每毫秒采一次还 OK，但连续采几千个点就会拖慢主循环。
**DMA 模式**，让 ADC 自己往内存里写数据，CPU 该干吗干吗。

**有效位数损失**，理想 ADC 标称位数（如 12 位）是理论最大值；噪声、量化误差、温漂、基准波动、非线性会让实际可用精度下降，

## DAC

DAC 是 ADC 的逆过程——给一个数字值，输出对应的模拟电压。

# 引脚电气数据

## 工作电压

单片机工作电压是 3.3V。引脚的高电平是 3.3V，低电平是 0V。

| | 低电平 (LOW) | 高电平 (HIGH) |
|------|-------------|--------------|
| 输出 | ~0V | ~3.3V |
| 输入 | < 0.8V | > 2.0V |

输入电压在 0.8V~2.0V 之间时，芯片可能读成 0 也可能读成 1，结果不可靠。悬空引脚容易落在这个区间。

## 电流限制

GPIO 引脚用来传信号，不是供动力。单引脚最大输出电流约 20mA。

## 引脚电平

有些传感器模块的引脚是**5V**的——它们的信号引脚输出 5V 高电平。但 STM32 的 GPIO 工作在 3.3V。

**把 5V 信号直接接到 STM32 的普通引脚上，会烧掉那个引脚——甚至烧掉整个芯片。**

STM32的部分引脚是5V容忍的，这些引脚可以承受 5V 输入。不同系列5V容忍的引脚比例是不同的（如 STM32F4（F407 这类）大部分引脚是 5V 容忍 ， STM32F1（F103C8T6）只有少部分容忍5V ）

# 最小系统

单片机芯片不能仅靠供电就跑起来。它需要一些最基本的外围电路才能复位、启振并工作。这些加起来就是单片机最小系统板。

## 四个必要条件

| 组件 | 作用 |
|------|------|
| 电源 | 提供稳定的 3.3V |
| 晶振 | 提供时钟信号 |
| 复位电路 | 上电时让 CPU 从第一条指令开始执行 |
| 下载接口 | 把程序烧进去 |

# USB 转串口芯片(有线)

我们知道**电脑使用的USB口（TypeC,TypeA）**使用USB 协议。
而**单片机的 UART口**使用UART 协议。两个设备之间正常来说是不能通信的，然而当有了USB 转串口芯片时，器会在两者中间做双向翻译，使两者间能通信

## USB转串口驱动

USB 转串口芯片把单片机与电脑连接在一起后，电脑还需要一种驱动，识别对应的USB转串口芯片，将这个的USB设备虚拟化为串口，从而可以在上位机中打开串口，接收单片机通过串口发送的调试信息，辅助调试单片机

# 中小型工程架构

1. 裸机无操作系统（三层）
调用流向：APP 业务层 → BSP 板级封装层 → HAL 硬件抽象层

    1.APP 业务层：业务逻辑、控制算法、状态机、业务流程

    2.BSP 板级封装层：板载外设统一封装（LED、按键、电机、ADC、传感器）

    3.HAL 硬件抽象层：芯片原厂底层外设驱动，屏蔽寄存器操作

2. 搭载 RTOS 实时操作系统（四层扩展架构）
调用流向：APP 应用层 → OSAL & 中间件层 → BSP 板级支持层 → HAL 硬件抽象层

    1.APP 应用层：各业务任务、控制算法、业务状态机、上层业务逻辑

    2.OSAL & 通用中间件层

    （1）OSAL 操作系统抽象层：封装 FreeRTOS 原生 API（任务、队列、信号量、互斥锁、阻塞延时），统一系统调用接口，方便后期更换 RTOS；

    （2）通用中间件组件：日志打印、LVGL 图形界面、Modbus 通信、WiFi 驱动、文件存储、OTA 升级等通用模块；

    3.BSP 板级支持层
    绑定硬件 PCB，封装板上所有外设操作，隔离业务与底层硬件；

    4.HAL 硬件抽象层：ST 官方底层驱动，直接操作片内外设 GPIO/TIM/ADC/USART/DMA 等。

# UART —— 最基础的串口通信

## 什么是 UART？

**UART**（通用异步收发器）是嵌入式里最简单，最常用的通信方式。两个设备之间只需要**两根线**（外加一根公共地线）就能互相发送数据。

- **TX**（Transmit，发送）：数据从本设备送出。
- **RX**（Receive，接收）：数据从本设备接收。
- **GND**（Ground，地线）：提供一个共同的电压参考点。

连接规则非常简单：**A 的 TX 接 B 的 RX，A 的 RX 接 B 的 TX。交叉连接。**

```
设备 A                    设备 B
  TX ──────────────────→ RX
  RX ←────────────────── TX
 GND ─────────────────── GND
```

## "异步"是什么意思？

通信分为两大类：**同步**和**异步**。

- **同步通信**：除了数据线，还有一根**时钟线**。发送方在时钟线上一拍一拍地打节拍，接收方跟着节拍读取数据。就像跳舞时有人喊"一、二、三、四"。
- **异步通信**：**没有时钟线**。双方提前约定好一个共同的"节奏"（波特率），然后各自用自己的时钟来发送和接收数据。

UART 是异步的——没有时钟线，双方靠**事先约定好的波特率**来同步。这意味着双方必须严格用相同的波特率，否则数据就会变成乱码。

## 一个字节是怎么被发送出去的？

假设你要发送一个字节的数据 `0x41`（二进制是 `01000001`，也就是 ASCII 字符 `'A'`），配上最常见的配置 **8-N-1**（8 个数据位、无校验位、1 个停止位），TXD 线上的电平变化如下：

```
空闲状态（高电平）
  ↓
  |‾‾‾‾‾‾‾|__|‾‾|__|__|__|__|__|‾‾|__|‾‾‾‾‾‾‾‾‾|
  |  Start | D0 D1 D2 D3 D4 D5 D6 D7 | Stop  | 空闲
  |   0    | 1  0  0  0  0  0  1  0  |   1   |
  |        |        0x41 (LSB first)  |       |
```

逐段解释：

1. **空闲状态**：线路上保持高电平（逻辑 1）。双方知道此时没有数据在传输。
2. **起始位（Start Bit）**：发送方把线路拉低一个比特周期。接收方检测到这个下降沿，"哦，数据来了！"，开始准备接收。
3. **数据位（Data Bits）**：紧接着起始位之后是 5 到 9 个数据位（最常见的是 8 位）。**最低位先发送**（这叫 LSB First，Least Significant Bit first）。上图中 `D0` 是最低位。
4. **校验位（Parity Bit，可选）**：如果开启了校验，数据位之后会多一个校验位，用来检测传输中是否出现了错误。8-N-1 的 "N" 就是 No parity，不校验。
5. **停止位（Stop Bit(s)）**：数据发送完毕，发送方把线路拉回高电平并保持 1 到 2 个比特周期。这是一个字节传输结束的标志。

## 常见配置：8-N-1

99% 的场景下，使用的都是 **8-N-1** 配置：

| 参数 | 值 | 含义 |
|------|-----|------|
| 数据位 | 8 | 一个字节就是 8 位，刚好对应一个 `char` |
| 校验位 | None (N) | 不做奇偶校验 |
| 停止位 | 1 | 1 个停止位 |

**UART 是嵌入式开发中最常用的通信方式**

## 接收数据的两种方式

UART 接收数据有三种常见方式：

1. **阻塞接收（Blocking）**：调用 `HAL_UART_Receive()`，CPU 一直等着，直到收到指定数量的数据才继续往下执行。适合简单应用，但等数据期间 CPU 什么别的也干不了。
2. **中断接收（Interrupt）**：调用 `HAL_UART_Receive_IT()`，开启中断。数据到达时硬件触发中断，CPU 来处理。**最常用的方式。**
3. **DMA 接收（Direct Memory Access）**：数据自动存到内存，完全不占用 CPU。适合高速、大批量的数据接收。

# I2C —— 两线制主从通信

## 什么是 I2C？

**I2C**（集成电路间总线）是一种只用**两根线**就能连接**很多个设备**的通信协议。

两根线分别是：
- **SCL**（Serial Clock，串行时钟线）：由主机（MCU）产生，为通信提供节拍。
- **SDA**（Serial Data，串行数据线）：承载实际的数据，双向传输。

主机先喊"地址"，只有对应地址的设备才回应。

## 硬件连接

I2C 引脚内部是开漏输出，无法主动输出高电平，必须在 SDA、SCL 分别外接上拉电阻到 VCC（3.3V）
常用阻值是4.7kΩ

## 主从架构和设备地址

I2C 是**主从（Master-Slave）架构**：
- **主机（Master）** 只有一个，通常是MCU。主机发起所有通信，产生时钟信号。
- **从机（Slave）** 可以有很多个，每个都有一个唯一的 **7 位地址**（或 10 位地址，但 7 位最常见）。

每个 I2C 从机（传感器、存储器、显示屏等）在出厂时就被分配了一个或多个地址。数据手册上会写清楚。

## 通信时序：一次完整的 I2C 对话

逐阶段解释：

1. **起始条件（Start Condition）**：SCL 为高电平时，SDA 从高变低。主机宣告"通信开始，大家注意！"
2. **地址 + 读写位**：主机发送 7 位从机地址，再跟 1 位读写方向（0 = 主机写数据给从机，1 = 主机从从机读数据）。
3. **应答位（ACK）**：第 9 个时钟脉冲时，主机释放 SDA，如果从机认出了自己的地址，就把 SDA 拉低表示"我在！"（ACK）。如果没有任何从机响应，SDA 保持高电平（NACK）。
4. **数据传输**：接下来是实际的数据，每个字节（8 位）后面都跟着一个应答位。
5. **停止条件（Stop Condition）**：SCL 为高电平时，SDA 从低变高。主机宣告"通话结束"。

## 为什么两根线能连很多设备？

硬件层面：总线并联拓扑

I2C 所有设备的 SDA 全部并联在一起，所有 SCL 也全部并联。
- SCL 时钟线：只有主机输出时钟，所有从机被动同步接收时钟；
- SDA 数据线：全部开漏输出，同一时刻只有一台设备拉低总线，其余全部释放，不会短路冲突。

开漏结构决定了多设备可以共用两根线，推挽总线做不到这一点。所以 I2C 用两根线连接 100+ 个设备从物理上是可行的（实际上受限于总线电容和地址空间）。

I2C 标准规定总线总电容≤400pF；每个 I2C 芯片引脚自带 5~15pF 寄生电容，再加 PCB 走线、排针电容。

8 个传感器就会占用 80~120pF，长线轻松突破限值；电容越大，开漏拉高波形上升沿越慢，时序不满足就通信失败。
上拉电阻驱动能力有限

默认 4.7kΩ 上拉，电容负载大时拉高速度不足；换更小 2.2kΩ 会增大总线灌电流，功耗上升。

# SPI —— 高速全双工通信

## 什么是 SPI？

**SPI**（串行外设接口）是 Motorola 公司发明的一种高速串行通信协议。和 I2C 不同，SPI 的核心理念是一个字——**快**。

SPI 用**四根线**（至少）实现通信：

| 信号线 | 全称 | 方向 | 作用 |
|--------|------|------|------|
| **SCK** | Serial Clock | 主机 → 从机 | 时钟线，主机提供节拍 |
| **MOSI** | Master Out Slave In | 主机 → 从机 | 主机发送数据给从机 |
| **MISO** | Master In Slave Out | 从机 → 主机 | 从机发送数据给主机 |
| **NSS / CS** | Slave Select / Chip Select | 主机 → 从机 | 片选信号|

## 全双工

SPI 和 I2C、UART 最大的区别是**全双工**（Full Duplex）：

- **UART**：全双工（两根独立的 TX 和 RX 线，可以同时收发）。  
- **I2C**：半双工（只有一根数据线 SDA，发送和接收只能轮流来）。  
- **SPI**：全双工（两根独立的数据线 MOSI 和 MISO，可以同时收发）。

## 片选信号

I2C 用地址来区分不同的从机。SPI 不用地址，而是用**片选信号**：每个从机独占一根 CS 线。

通信过程：当主机要和从机 1 通信时，把 CS1 拉低（从高电平变成低电平），然后开始在 SCK/MOSI/MISO 上传输数据。其他从机看到自己的 CS 线保持高电平，就完全忽略总线上的数据。

**CS 信号是低电平有效**（Active Low）——空闲时是高电平，选中设备时拉低。

## SPI 的四种模式

SPI 的**时钟极性和相位**能组合出 4 种工作模式。

- **CPOL**（Clock Polarity，时钟极性）：空闲时 SCK 是什么电平？0 = 低电平，1 = 高电平。
- **CPHA**（Clock Phase，时钟相位）：在 SCK 的第几个边沿采样数据？0 = 第一个边沿，1 = 第二个边沿。

四种模式：

| 模式 | CPOL | CPHA | 空闲时 SCK | 采样边沿 |
|------|------|------|-----------|---------|
| Mode 0 | 0 | 0 | 低 | 上升沿（第一个边沿） |
| Mode 1 | 0 | 1 | 低 | 下降沿（第二个边沿） |
| Mode 2 | 1 | 0 | 高 | 下降沿（第一个边沿） |
| Mode 3 | 1 | 1 | 高 | 上升沿（第二个边沿） |

**不同外设出厂规定了自己读取 MOSI/MISO 的时钟时机。而SPI 必须主从 Mode 完全一致，通信才能正常。** 绝大多数外设默认 Mode 0。次常用 Mode 3

## SPI 的数据传输时序（Mode 0）

下面是一次完整的 SPI 字节传输（Mode 0）的时序图：

空闲状态
CS 高电平，SCK 低，MOSI/MISO 高阻 / 闲置，无通信。

拉低片选 CS，选中从机
CS = 0，通信开始，SCK 维持低。

循环 8 次，逐位发送 / 接收（高位先行 MSB-first，绝大多数器件）
每一位时序：
- 主机先把当前要发送的 bit 放到 MOSI 线上；
- SCK 从低→高（上升沿）：从机立刻读取 MOSI 上这一位；同时主机读取 MISO 上从机输出的 bit；
- SCK 从高→低（下降沿）：主机移出下一位到 MOSI，从机移出下一位到 MISO；

重复 8 轮，收发完整 1 字节。

8 位全部传输完成
- SCK 停在低电平（回到空闲极性）。

拉高 CS，结束本次通信
- CS=1，释放从设备。

## SPI 的实际应用

SPI 因为速度快，常用于需要大量数据传输的场景：

- **SPI OLED/LCD 显示屏**：用 u8g2 等图形库驱动。SPI 的速度让屏幕刷新不卡顿。
- **SD 卡**：读写大文件需要 SPI 的高速能力。
- **高速 ADC**：采集高速模拟信号（比如音频、振动）。
- **SPI Flash 存储器**：存储大量数据（字体、图片、日志）。
- **SPI LED 驱动芯片**（如 WS2812 的 SPI 模拟方式）。

## SPI 的优缺点总结

| 优点 | 缺点 |
|------|------|
| 速度快（可达几十 MHz） | 引脚多（至少 4 根 + 每个从机多一根 CS） |
| 全双工，同时收发 | 没有标准协议层（不像 I2C 有地址/ACK 机制） |
| 硬件实现简单 | 没有应答确认（发送方不知道收没收到） |
| 没有地址冲突问题 | 不适合太多从机（CS 引脚不够用） |

# UART vs I2C vs SPI —— 三种协议对比

## 一句话概括

- **UART**：最简单的点对点通信，两根线交叉连接，适合调试和连接单一外设（GPS、蓝牙）。
- **I2C**：两根线挂一堆传感器，通过地址区分设备，适合低速多设备场景。
- **SPI**：四根线高速全双工，通过片选引脚区分设备，适合需要速度的场景（屏幕、SD 卡）。

## 核心参数速查表

| 对比项 | UART | I2C | SPI |
|--------|------|-----|-----|
| 最少引脚数 | 2（TX, RX） | 2（SCL, SDA） | 4（SCK, MOSI, MISO, CS） |
| 典型速度 | 最高几 Mbps | 100k / 400k / 1M bps | 数十 MHz |
| 最大距离 | 数米 | 同一块 PCB 或几十厘米 | 同一块 PCB 或几十厘米 |
| 通信方式 | 异步（无时钟线） | 同步（有时钟线） | 同步（有时钟线） |
| 双工方式 | 全双工 | 半双工（一根数据线） | 全双工 |
| 多设备支持 | 一对一（点对点） | 一对多（地址机制，最多 127 从机） | 一对多（CS 引脚机制） |
| 设备区分方式 | 无需区分（只有对方） | 7 位 / 10 位设备地址 | 每个从机独占一根 CS 引脚 |
| 硬件复杂度 | 最简单 | 中等（需要上拉电阻） | 较复杂（多根线） |
| 错误检测 | 可选校验位（奇偶校验） | 每个字节后有 ACK/NAK | 无内建机制 |
| 主从关系 | 对等（双方都能主动发） | 严格主从（只有主机能发起） | 严格主从（只有主机能发起） |

## 速度对比

速度的直观比较（同等条件下，传输 1KB 数据的耗时估算）：

| 协议 | 常用速度 | 1KB 耗时（约） |
|------|---------|---------------|
| UART | 115200 bps | ~90 ms |
| I2C（标准） | 100 kbps | ~100 ms |
| I2C（快速） | 400 kbps | ~25 ms |
| SPI | 10 MHz | ~1 ms |

## 如何选择

```
你要做什么？
│
├─ 串口调试输出（printf）              → UART
├─ 连接蓝牙/GPS/WiFi 模块              → UART（这些模块本身就是 UART 接口）
├─ 连接多个传感器，引脚不够用           → I2C
├─ 连接 OLED 屏（引脚紧张时）           → I2C
├─ 连接 OLED/LCD 屏（追求刷新速度）     → SPI
├─ 读写 SD 卡                          → SPI
├─ 连接高速 ADC（采集波形）             → SPI

```

# 为什么需要操作系统

裸机一般是 while(1) 大循环顺序执行，存在几个硬缺陷：

任务阻塞，响应迟钝
某一段代码耗时久（延时、传感器读取、屏幕刷新），其他功能必须等它跑完；按键、急停、串口指令会丢失，实时性差。

多任务调度逻辑复杂，代码难写难维护
要实现 “同时” 采集、显示、通信、电机控制，只能手写状态机、毫秒计时、轮询判断，业务越多逻辑越乱，极易出错。

资源冲突无法安全处理
多个任务同时操作串口、屏幕、ADC 等硬件，会出现数据错乱；裸机没有互斥保护机制，需要自己加标志位做互斥。

延时占用 CPU，资源浪费
裸机 HAL_Delay() 是忙等，CPU 空转，无法利用这段时间处理其他任务。

功能模块化拆分困难
采集、控制、UI、通信全部揉在一个大循环，新增功能、删减功能极易牵一发而动全身，可移植性差。
当

而操作系统（OS）的核心能力是**任务调度**：在多个任务之间快速切换，让每个任务看起来都在"同时运行"。

## 操作系统的对比

RTOS 优势
- 硬实时，控制延迟稳定，工业 / 运动控制刚需；
- 占用资源极少，低成本 MCU 即可运行，低功耗；
- 启动极快，断电重启快速恢复控制；
- 代码精简，故障点少，安全认证更容易。

RTOS 短板
- 生态薄弱，复杂网络、图形、AI 需要自行移植；
- 无进程隔离，一个任务崩溃会导致整机死机；
- 缺少 shell、脚本、编译工具链，复杂业务开发效率低。

Linux 优势
- 海量现成开源软件，网络、图像、算法开箱即用；
- 多进程、用户态隔离，单个应用崩溃不影响系统；
- 完善命令行、调试工具、远程运维、升级方案；
- 标准化驱动框架，外设适配更简单。

Linux 短板
- 无法做到严格硬实时，延迟波动大；
- 硬件成本高、功耗高、启动慢；
- 系统庞大，裁剪、定制、优化门槛高。

## 什么时候该上 RTOS？

当你的裸机 `while(1)` 里出现超过 3 个"必须在不同频率执行"的任务，而且你开始写大量 `if (tick % N == 0)` 这样的代码时，就该上 RTOS 了。

# FreeRTOS — 单片机上最流行的实时操作系统

FreeRTOS 是目前使用最广泛的嵌入式实时操作系统。它开源、免费、轻量（最小只需要几 KB ROM 和几百字节 RAM），几乎所有主流单片机都支持。用 STM32CubeMX 勾选一个选项就能集成，非常方便。

## 核心概念一：任务（Task）

一个任务就是一个**拥有自己栈空间的无限循环函数**。你写任务的方式和写裸机 `while(1)` 很像——区别是你可以同时跑多个任务。


## 核心概念二：调度器（Scheduler）

调度器是 FreeRTOS 的核心——它决定"下一个该跑哪个任务"。它采用的是**抢占式优先级调度**：

- 每个任务有一个优先级（数字越大优先级越高）
- 高优先级任务**随时可以抢占**低优先级任务
- 同优先级任务**时间片轮转**（每人跑一小段时间，轮流）

· 任务 A 就绪时，立刻抢占 B 和 C
· 任务 A 休眠时，B 执行
· 没有 A 和 B 时，C 执行

## 核心概念三：队列（Queue）

队列是任务之间**安全传递数据**的通道。想象一条传送带——任务 A 把数据放上去，任务 B 从另一头取下来。

**为什么用队列而不是全局变量？** 全局变量在多任务环境下非常危险——任务 A 写到一半，任务 B 来读，可能读到不完整的数据。队列内部有保护机制，保证"放进去"和"取出来"的操作是原子的（不可打断的），不会出现数据损坏。

## 核心概念四：信号量（Semaphore）

信号量是一个**同步标志**。经典用法是：一个任务等待信号量（阻塞），另一个任务在条件满足时释放信号量（唤醒等待者）。

## 与裸机对比

| 特性 | 裸机 | FreeRTOS |
|------|------|----------|
| 多任务 | 手动安排，混乱 | 自动调度，清晰 |
| 延时 | `HAL_Delay()` 忙等 | `vTaskDelay()` 让出 CPU |
| 任务间通信 | 全局变量（不安全） | 队列（安全） |
| 同步 | 标志位 | 信号量、互斥锁 |
| 代码复杂度 | 简单项目低，复杂项目高 | 各项目中等 |
| ROM 占用 | 最小 | 增加 6~12 KB |
| RAM 占用 | 最小 | 增加几 KB |

# Linux

Linux 和 MCU 区别很大
1. **硬件芯片本质不同**
MCU（单片机，Cortex-M）
无 MMU 内存管理单元，没有虚拟内存；直接操作物理地址，芯片小、RAM/Flash 极小（几 KB~ 几百 KB），成本低。
运行 Linux 的 MPU（Cortex-A）
自带 MMU，支持虚拟内存、进程地址隔离；必须搭配大容量内存（64MB 起）、大容量闪存，芯片成本更高。
2. **系统与实时性**
MCU：裸机 / FreeRTOS 等轻量 RTOS，硬实时，中断、调度延迟微秒级，时序稳定可控，适合电机、传感器实时控制。
Linux：通用分时系统，默认软实时，延迟波动大，无法用于高精度运动控制。
3. **软件架构**
MCU：单地址空间，无进程隔离，一个任务崩溃整机死机；直接操作寄存器、BSP 驱动。
Linux：内核态 + 用户态隔离，多进程独立；单个应用崩溃不影响系统，有标准驱动框架、完整 Shell、海量开源软件（Python、QT、网络工具）。
4. **资源、启动、功耗**
MCU：镜像极小，上电几毫秒启动，功耗低，适合电池供电设备。
Linux：系统镜像几十上百 MB，开机几秒，功耗更高。
5. **适用场景**
MCU：传感器采集、电机伺服、便携低功耗硬件、简单控制设备。
Linux：网关、机器视觉、大屏 UI、云端联网、AI 边缘运算、多媒体设备。

## GNU/Linux 是什么

很多初学者好奇"Linux"到底指什么。严格来说，我们日常说的"Linux 系统"其实是 **GNU/Linux**：

- **Linux**：只是一个**内核**（Kernel）。内核管理硬件（CPU、内存、硬盘、网卡），提供最基本的系统服务。
- **GNU**：一套**自由软件工具集**，包括 GCC（编译器）、Bash（命令行）、Coreutils（基本命令如 `ls`、`cp`）。这些工具是操作系统"用户能用"的部分。

内核 + 工具集 = 一个完整的操作系统。就像汽车引擎（Linux）+ 方向盘、座椅、仪表盘（GNU）= 一辆能开的车。

## Linux 发行版是什么

GNU/Linux 只是"原材料"。不同的人或组织把这些原材料打包成**发行版（Distribution）**，预装不同的软件、使用不同的包管理器（安装/卸载软件的工具）。

常见的发行版：

| 发行版 | 特点 | 适合谁 |
|--------|------|--------|
| Ubuntu | 用户友好、资料最多 | 新手 |
| Arch Linux | 滚动更新、高度自定义、KISS 原则 | 喜欢折腾的用户 |
| Debian | 极其稳定、保守 | 服务器 |
| Fedora | 新技术先行 | 开发者 |

::: tip
**Arch Linux 的 KISS 原则**：Keep It Simple, Stupid（保持简单，傻瓜）。这里的"简单"不是对用户简单，而是对系统设计者简单——系统不替你做决定，由你自己掌控一切。Arch 的用户手册（Arch Wiki）被公认为最好的 Linux 文档之一。

# TB6612FNG — 两路电机驱动

## 它是什么

TB6612FNG 是一颗**直流电机驱动芯片**。它像一个"电机翻译官"——MCU 给它低电压的逻辑信号（3.3V，几乎不费力），它输出大电流（最大 1.2A 每路）来驱动电机。

把它想象成音响的功放：手机输出微弱的音频信号，功放把它放大到能驱动喇叭。TB6612 把 MCU 的弱信号"放大"成能驱动电机的强电流。

## 它能做什么

- 同时驱动 **2 个** 直流电机（小车左轮 + 右轮）
- 每个电机可以 **正转、反转、刹车、滑行**
- 用 **PWM（脉冲宽度调制）** 控制速度
- 工作电压：电机供电 2.5V-13.5V，逻辑供电 2.7V-5.5V

## 关键引脚

| 引脚名 | 功能 | 接哪里 |
|--------|------|--------|
| VM | 电机电源 | 电池正极（如 7.4V 锂电池） |
| VCC | 芯片逻辑电源 | 3.3V 或 5V |
| GND | 地 | 电池负极 + MCU 地 |
| PWMA | A 路电机速度 | MCU 的 PWM 输出引脚 |
| AIN1, AIN2 | A 路电机方向 | MCU 的两个普通 GPIO |
| PWMB | B 路电机速度 | MCU 的 PWM 输出引脚 |
| BIN1, BIN2 | B 路电机方向 | MCU 的两个普通 GPIO |
| STBY | 使能（总开关） | MCU 的一个 GPIO，拉高 = 工作 |
| AO1, AO2 | A 路输出 | 接 A 电机的两根线 |
| BO1, BO2 | B 路输出 | 接 B 电机的两根线 |

## 真值表：怎么控制电机

以 A 路电机为例（B 路完全一样）：

| AIN1 | AIN2 | PWMA | 电机状态 | 什么时候用 |
|------|------|------|---------|-----------|
| 1 | 0 | 0-100% | **正转**（前进） | 让车前进 |
| 0 | 1 | 0-100% | **反转**（后退） | 让车后退 |
| 1 | 1 | 任意 | **刹车**（急停） | 需要立刻停下 |
| 0 | 0 | 任意 | **滑行**（自由停） | 让车自然减速 |
| 0 | 0 | 0 | **停止** | 省电待机 |

这里 1 = 高电平（3.3V），0 = 低电平（0V），PWM 的百分比 = 速度（0% 不动，100% 全速）。

::: tip 刹车 vs 滑行
- **刹车（1,1）：** 电机两个端子被短接到地，电机像被"夹住"一样立刻停下。  
- **滑行（0,0）：** 电机两个端子断开，电机自由转动，车靠摩擦力慢慢停。
:::

## PWM 频率设置

PWM 频率建议设为 **10kHz**（即周期 0.1ms）。原因：

- 低于 10kHz → 电机会发出"滋滋"的高频噪音（人耳能听到 20Hz-20kHz）
- 高于 100kHz → 驱动芯片的开关损耗增大，发热严重
- **10kHz-20kHz 是最佳区间**：人耳听不到，芯片也不会过热

```c
// 以 MSPM0 为例，设置 PWM 频率为 10kHz
// 假设定时器时钟为 48MHz
// PWM 频率 = 定时器时钟 / (预分频 × 自动重装载值)
// 10kHz = 48MHz / (48 × 100) → 预分频=48, 重装载=100
```

## 死区时间（Dead Time）

::: danger 重要安全规则
**在电机正转时，绝对不要直接切换到反转！**

必须先刹车或滑行一小段时间（约 1-5ms），再切换到反向。否则可能出现"上下桥臂同时导通"（shoot-through），瞬间大电流烧毁芯片。
:::

```c
// 错误写法（危险！）
void motor_change_direction_bad() {
    motor_forward(80);   // 正在前进
    motor_reverse(80);   // 直接反转 → 可能烧芯片！
}

// 正确写法（安全）
void motor_change_direction_good() {
    motor_forward(80);   // 正在前进
    motor_brake();       // 先刹车
    delay_ms(5);         // 等 5ms（死区时间）
    motor_reverse(80);   // 再反转 → 安全！
}
```

```c
#define AIN1_PORT  GPIOA

    // 假设 GPIO 和 PWM 已在配置工具中初始化
    // 只需设置 STBY 为高电平（使能）
    DL_GPIO_setPins(STBY_PORT, STBY_PIN);
}

// A 路电机：前进
void motor_a_forward(uint8_t speed) {
    DL_GPIO_setPins(AIN1_PORT, AIN1_PIN);      // AIN1 = 1
    DL_GPIO_clearPins(AIN2_PORT, AIN2_PIN);    // AIN2 = 0
    set_pwm_a(speed);  // 设置 PWMA 占空比 = speed%
}

// A 路电机：后退
void motor_a_reverse(uint8_t speed) {
    DL_GPIO_clearPins(AIN1_PORT, AIN1_PIN);    // AIN1 = 0
    DL_GPIO_setPins(AIN2_PORT, AIN2_PIN);      // AIN2 = 1
    set_pwm_a(speed);
}

// A 路电机：刹车
void motor_a_brake(void) {
    DL_GPIO_setPins(AIN1_PORT, AIN1_PIN);      // AIN1 = 1
    DL_GPIO_setPins(AIN2_PORT, AIN2_PIN);      // AIN2 = 1
    set_pwm_a(100);  // 刹车时占空比 100%
}

// A 路电机：滑行
void motor_a_coast(void) {
    DL_GPIO_clearPins(AIN1_PORT, AIN1_PIN);    // AIN1 = 0
    DL_GPIO_clearPins(AIN2_PORT, AIN2_PIN);    // AIN2 = 0
    set_pwm_a(0);
}

// 小车前进（两轮同速）
void car_forward(uint8_t speed) {
    motor_a_forward(speed);
    motor_b_forward(speed);
}

// 小车原地左转（左轮后退、右轮前进）
void car_turn_left(uint8_t speed) {
    motor_a_reverse(speed);
    motor_b_forward(speed);
}

// 小车停止
void car_stop(void) {
    motor_a_coast();
    motor_b_coast();
}
```

:::

# 编码器 — 测速与里程

## 编码器是什么

编码器（Encoder）是一个能告诉你"电机转了多少圈"的传感器。日常类比：自行车上的码表——轮子每转一圈，传感器就计数一次，从而算出速度和距离。编码器就是电机的"码表"。

## 霍尔编码器的原理

竞赛小车常用的是**霍尔编码器**。它由两部分组成：

1. **磁铁盘：** 装在电机转轴上，上面分布着 N 对磁极（N→S→N→S 交替排列）
2. **霍尔传感器：** 固定不动，当磁极交替经过时，传感器输出高低电平的方波

```
磁铁盘旋转 →
  N  S  N  S  N  S
  ↓  ↓  ↓  ↓  ↓  ↓
霍尔传感器 →
  输出：高 低 高 低 高 低  ← 这就是"脉冲"
```

一个"脉冲" = 一个完整的高低电平周期。**PPR**（Pulses Per Revolution）= 磁铁盘每转一圈输出的脉冲数。

## 我们的编码器参数

竞赛标准配置的编码器：

- **电机端 PPR：** 13（磁铁盘每转一圈 13 个脉冲）
- **减速比：** 20:1（电机转 20 圈，输出轴转 1 圈）
- **输出轴 PPR：** 13 × 20 = 260 脉冲/圈（输出轴每转一圈）

### 正交编码与 4× 解码

编码器有 A、B 两路输出信号，相位差 90 度（像一个在前一个在后）：

```
A 相：┌─┐  ┌─┐  ┌─┐  ┌─┐
      └─┘  └─┘  └─┘  └─┘
B 相：  ┌─┐  ┌─┐  ┌─┐  ┌─┐
        └─┘  └─┘  └─┘  └─┘
```

MCU 的定时器可以同时看 A 和 B 的上升沿和下降沿——每个脉冲周期能检测到 4 个事件（这叫**4× 解码**）。

```
输出轴每圈计数 = 260 × 4 = 1040 个计数
```

这意味着每毫米能检测到的移动精度非常高。

## 硬件连接

编码器有 4 根线：

| 线色 | 信号 | 接哪里 |
|------|------|--------|
| 红 | VCC (3.3V 或 5V) | MCU 电源输出 |
| 黑 | GND | MCU 地 |
| 黄 | A 相输出 | 定时器的通道 1（如 TIM4_CH1） |
| 绿 | B 相输出 | 定时器的通道 2（如 TIM4_CH2） |

## 定时器的编码器模式

**这是最关键的部分。** MCU 的通用定时器（如 STM32 的 TIM2-TIM5，MSPM0 的通用定时器）有一个"编码器模式"——它可以**自动**对 A、B 两相脉冲进行 4× 解码计数，不需要 CPU 参与。

### 编码器模式做了什么

- 每当 A 相或 B 相有跳变 → 定时器的计数器自动 +1 或 -1
- 正转时计数器增加，反转时计数器减少
- CPU 不需要在中断里计数——直接读定时器的计数值就行

### 速度计算

```c
// 每 10ms 读一次编码器，计算速度
#define ENCODER_COUNTS_PER_METER  5200.0f  // 根据轮子周长计算

float calculate_speed(int32_t count_diff, float time_delta_s) {
    // count_diff: 这次读到的计数值 - 上次读到的计数值
    // time_delta_s: 两次读取的时间间隔（秒）
    
    float distance = (float)count_diff / ENCODER_COUNTS_PER_METER; // 米
    float speed = distance / time_delta_s; // 米/秒
    return speed;
}
```

### ENCODER_COUNTS_PER_METER 怎么算

```
轮子周长 = π × 直径
         = 3.1416 × 0.065m  (直径 6.5cm)
         = 0.2042m

输出轴每圈计数（4× 解码后） = 1040

每米计数数 = 1040 / 0.2042 ≈ 5093

加上一些余量，实际用 5200 左右
```

## 代码示例

```c
// ==================== encoder.h ====================
#ifndef ENCODER_H
#define ENCODER_H

#include <stdint.h>

void encoder_init(void);
int32_t encoder_get_count(void);      // 读取累计脉冲计数
void encoder_reset(void);             // 清零计数
float encoder_get_speed(void);        // 获取实时速度（m/s）
float encoder_get_distance(void);     // 获取累计里程（m）
void encoder_update(void);            // 定期调用（如每 10ms）

#endif

// ==================== encoder.c ====================
#include "encoder.h"
#include "ti_msp_dl_config.h"

#define COUNTS_PER_METER    5200.0f   // 根据你的轮子计算
#define UPDATE_INTERVAL_S   0.01f     // 10ms 更新一次

static int32_t g_encoder_count = 0;   // 累计计数
static int32_t g_last_count = 0;      // 上次读数
static float g_speed = 0.0f;          // 当前速度
static float g_distance = 0.0f;       // 累计里程

void encoder_init(void) {
    // 定时器已配置为编码器模式（在 .syscfg 或 CubeMX 中配置）
    // 启用定时器
    DL_TimerG_startCounter(ENCODER_TIMER_INST);
}

int32_t encoder_get_count(void) {
    // 直接读定时器的计数器值
    // 注意：根据定时器位数（16位或32位），可能溢出
    return (int32_t)DL_TimerG_getCounterValue(ENCODER_TIMER_INST);
}

void encoder_reset(void) {
    DL_TimerG_setCounterValue(ENCODER_TIMER_INST, 0);
    g_encoder_count = 0;
    g_last_count = 0;
    g_speed = 0.0f;
    g_distance = 0.0f;
}

void encoder_update(void) {
    // 读当前计数值
    int32_t current_count = encoder_get_count();
    
    // 计算差值（处理定时器溢出）
    int32_t diff = current_count - g_last_count;
    
    // 更新累计
    g_encoder_count += diff;
    g_last_count = current_count;
    
    // 计算速度
    g_speed = (float)diff / COUNTS_PER_METER / UPDATE_INTERVAL_S;
    
    // 计算里程
    g_distance = (float)g_encoder_count / COUNTS_PER_METER;
}

float encoder_get_speed(void) {
    return g_speed;
}

float encoder_get_distance(void) {
    return g_distance;
}

// ==================== 在 main.c 中使用 ====================
// 定时器中断或主循环中每 10ms 调用一次
void TIMER_10MS_IRQHandler(void) {
    encoder_update();
    
    // 现在可以用 encoder_get_speed() 获取实时速度
    float current_speed = encoder_get_speed();
    // 用于 PID 控制……
}
```

## 编码器 vs 不用编码器

| 比较 | 不用编码器（开环） | 用编码器（闭环） |
|------|------------------|-----------------|
| 速度控制 | "我发了 80% 占空比，它大概是 0.8m/s" | "实际速度 0.76m/s，差 0.04，补一点" |
| 里程 | 算不了，只能靠估计 | 精确到毫米 |
| 上坡 | 速度变慢，代码不知道 | 检测到速度下降，自动加大油门 |
| 电池没电 | 越来越慢，代码不知道 | 检测到速度异常，可以报警 |

::: tip 一句话总结编码器的价值
**不用编码器，你的代码在"猜"车在哪里。用编码器，你的代码"知道"车在哪里。**
:::

## 常见问题

### 读数方向反了

正转时计数器减少？交换 A 相和 B 相的接线，或者在软件中取反。

### 读数跳动

- 检查编码器供电是否稳定（3.3V 或 5V）
- 信号线不要太长（< 30cm）
- 如果线长，加 100nF 电容在信号线和地之间

### 定时器溢出

16 位定时器的最大值是 65535，如果车跑很长时间计数器会溢出。代码中需要处理这种"回绕"情况。

---

## 小结

编码器让你知道"轮子转了多少圈"——有了它，小车从盲开变成感知行驶。A/B 相通过正交解码得到方向和圈数，再结合轮子周长算出速度和里程。花点时间把编码器调通，后面的 PID 闭环控制才能做。

# 8 路灰度传感器

## 它是怎么"看"到线的

灰度传感器是竞赛小车最常用的巡线传感器。它的原理非常简单：

```
        红外 LED（发光）
           │
           ▼
    ═══════════════════  ← 地面
    白色区域：反射强     黑色线：反射弱（被吸收）
           │                    │
           ▼                    ▼
    光电三极管收到多      光电三极管收到少
           │                    │
           ▼                    ▼
       输出电压高           输出电压低
       → "白色"             → "黑色"
```

用日常经验理解：穿着黑衣服在太阳下会更热，因为黑色吸收了更多光线。灰度传感器就是用这个原理——黑色吸收红外光，反射少，传感器输出低电压。

## 硬件结构

### 8 路传感器排列

8 个传感器一字排开，间距约 12mm（和赛道黑线宽度匹配）：

```
传感器编号:  0    1    2    3    4    5    6    7
            ○    ○    ○    ○    ○    ○    ○    ○
                     车头方向 →
                     中线 ↕ (传感器 3 和 4 之间)
```

黑线宽度通常在 20-30mm。理想情况下，黑线会同时覆盖 2-3 个传感器。

### 3-to-8 多路选择器

::: tip 为什么需要多路选择器
8 个传感器需要 8 个 ADC 引脚来读——但 MCU 的 ADC 通道可能不够。用一片 **74HC4051**（3-to-8 模拟多路选择器），只需要 3 个 GPIO 选择通道 + 1 个 ADC 输入 = 只用 4 个 MCU 引脚就读到了全部 8 路！
:::

```
         74HC4051 多路选择器
         ┌─────────────┐
AD0 ─────│ S0       Y0 │──── 传感器 0
AD1 ─────│ S1       Y1 │──── 传感器 1
AD2 ─────│ S2       Y2 │──── 传感器 2
         │           ...│
   ┌─────│ Z        Y7 │──── 传感器 7
   │     └─────────────┘
   ▼
ADC 引脚（读电压）
```

**AD0-AD2 三根地址线**相当于一个"选择器"——000 选传感器 0，001 选传感器 1，……，111 选传感器 7。

## 读取灰度值

### 基本读取流程

```c
// 读取指定通道的灰度值
uint16_t grayscale_read_channel(uint8_t channel) {
    // 1. 设置地址线（AD0, AD1, AD2）
    DL_GPIO_writePins(AD0_PORT, AD0_PIN, (channel & 0x01) ? 1 : 0);
    DL_GPIO_writePins(AD1_PORT, AD1_PIN, (channel & 0x02) ? 1 : 0);
    DL_GPIO_writePins(AD2_PORT, AD2_PIN, (channel & 0x04) ? 1 : 0);
    
    // 2. 等待信号稳定（>1μs，实际用 10μs 更保险）
    delay_us(10);
    
    // 3. 读 ADC
    return ADC_read();
}
```

::: warning 等待时间很重要
切换通道后，多路选择器和 ADC 都需要时间稳定。如果不等就直接读，读到的是上一个通道的残留值。10μs 是一个安全的值。
:::

### 读取全部 8 路

```c
// 一次读取全部 8 路灰度值，存入数组
void grayscale_read_all(uint16_t values[8]) {
    for (uint8_t i = 0; i < 8; i++) {
        values[i] = grayscale_read_channel(i);
    }
}
```

## 从灰度值到黑线位置

### 阈值判断

首先需要设定一个"黑/白分界线"——阈值。高于阈值 = 白色，低于阈值 = 黑色。

```c
// 灰度值越高 = 越白，越低 = 越黑
// 阈值一般取 "白值" 和 "黑值" 的中间值
// 比如：白值 3000，黑值 500 → 阈值 = 1750

#define GRAYSCALE_THRESHOLD 1750  // 根据实际情况调整！

// 判断每个传感器是否在黑线上
uint8_t is_black[8];
for (int i = 0; i < 8; i++) {
    is_black[i] = (values[i] < GRAYSCALE_THRESHOLD) ? 1 : 0;
}
```

### 计算黑线位置：加权平均法

**这是最常用的巡线算法。** 给每个传感器一个"位置权重"，落在黑线上的传感器按权重取平均：

```c
// 传感器 0-7 的物理位置（以中线为 0，单位 mm）
// 传感器 0 在最左（-42mm），传感器 7 在最右（+42mm）
float sensor_positions[8] = {-42, -30, -18, -6, 6, 18, 30, 42};

float calculate_line_position(uint16_t values[8]) {
    float weighted_sum = 0.0f;
    float total_intensity = 0.0f;
    
    for (int i = 0; i < 8; i++) {
        // 将灰度值转换为"黑的程度"
        // 值越小（越黑）→ 权重越大
        float darkness = 1.0f - (float)values[i] / 4095.0f; // 假设 12 位 ADC
        
        weighted_sum += sensor_positions[i] * darkness;
        total_intensity += darkness;
    }
    
    if (total_intensity < 0.01f) {
        return 0.0f; // 没检测到黑线，默认在中间
    }
    
    return weighted_sum / total_intensity; // 黑线的加权位置
}
```

返回值的含义：
- `0` → 黑线在正中间（完美巡线状态）
- `+20` → 黑线偏右 20mm（车需要右转纠正）
- `-30` → 黑线偏左 30mm（车需要左转纠正）

## P 控制器的巡线应用

有了黑线位置，就可以用 P 控制器驱动机器回到中线上：

```c
void line_following_loop(void) {
    uint16_t gray_values[8];
    float line_pos;        // 黑线位置
    float error;           // 误差
    float correction;      // 修正量
    float Kp = 0.05f;      // P 系数（需要实验调整！）
    int base_speed = 60;   // 基础速度
    
    while (1) {
        // 1. 读灰度
        grayscale_read_all(gray_values);
        
        // 2. 计算黑线位置
        line_pos = calculate_line_position(gray_values);
        
        // 3. 计算误差（目标 = 0，即黑线在中间）
        error = 0.0f - line_pos;
        
        // 4. P 控制：修正量 = Kp × 误差
        correction = Kp * error;
        
        // 5. 驱动电机
        int left_speed  = base_speed + (int)correction;
        int right_speed = base_speed - (int)correction;
        
        // 限制速度范围
        if (left_speed  > 100) left_speed  = 100;
        if (left_speed  < -100) left_speed  = -100;
        if (right_speed > 100) right_speed = 100;
        if (right_speed < -100) right_speed = -100;
        
        motor_a_set_speed(left_speed);
        motor_b_set_speed(right_speed);
        
        delay_ms(10); // 10ms 控制周期
    }
}
```

这就是最基础的**闭环巡线**——传感器检测位置 → 计算误差 → 调整电机 → 小车回到中线上，形成完整的反馈闭环。

## 校准的重要性

::: danger 不校准 = 跑飞
灰度传感器的读数受环境光、地面材质、红外 LED 老化等因素影响。在实验室调好的阈值，拿到赛场上可能完全不对。**一定要在现场校准！**
:::

### 简单的自动校准

```c
// 在校准模式下：把车放在白色区域和黑色区域上各读一遍
void grayscale_calibrate(uint16_t white_ref[8], uint16_t black_ref[8]) {
    printf("请把车放在白色区域，按按键...\r\n");
    wait_button_press();
    grayscale_read_all(white_ref);
    
    printf("请把车放在黑色线上，按按键...\r\n");
    wait_button_press();
    grayscale_read_all(black_ref);
    
    // 阈值 = 中间值
    for (int i = 0; i < 8; i++) {
        threshold[i] = (white_ref[i] + black_ref[i]) / 2;
    }
}
```

## 灰度传感器 vs 其他巡线方案

| 方案 | 分辨率 | 速度 | 成本 | 适合 |
|------|--------|------|------|------|
| 8 路灰度 | 8 点 | 快（~0.1ms） | ~15元 | 简单巡线、十字路口 |
| 16 路灰度 | 16 点 | 快 | ~25元 | 复杂赛道 |
| 线性 CCD | 128 点 | 中（~1ms） | ~40元 | 高精度巡线 |
| 摄像头 | 全画面 | 慢（~10ms） | ~120元 | 复杂视觉任务 |

---

## 小结

8 路灰度传感器是入门巡线的最优选择——8 个光电对管一字排开，哪个看到黑线就能算出偏移量。安装高度 5-15mm，先校准再使用，数据用加权平均或线性插值处理。简单、可靠、够快、够用。

# JY61P 陀螺仪

## 它是什么

JY61P 是一款 **6 轴姿态传感器模块**。这里的"6 轴"是指它内部集成了两种传感器：

| 传感器 | 数量 | 测量什么 | 单位 |
|--------|------|---------|------|
| 加速度计 | 3 轴（X, Y, Z） | 线性加速度（含重力方向） | g（1g = 9.8 m/s²） |
| 陀螺仪 | 3 轴（X, Y, Z） | 角速度（旋转快慢） | °/s（度/秒） |

把它想象成你的**内耳（前庭系统）**——它能感知朝向、倾斜和旋转。人闭着眼睛走路不会倒，靠的就是内耳。小车转弯要精确，靠的就是陀螺仪。

## 为什么要用 JY61P

普通的陀螺仪芯片（如 MPU6050）只输出原始的角速度和加速度数据。要得到角度，你需要自己写复杂的姿态解算算法（互补滤波、卡尔曼滤波）。而 **JY61P 内置了姿态解算**——它直接输出**欧拉角**（滚转角、俯仰角、偏航角），你直接读就行。

::: tip 欧拉角是什么
欧拉角是用三个数字描述三维空间中物体的朝向：
- **滚转角（Roll）：** 左右倾斜（小车翻了吗？）
- **俯仰角（Pitch）：** 前后倾斜（上坡还是下坡？）
- **偏航角（Yaw）：** 水平旋转（转弯了多少度？）← 竞赛中最常用的
:::

## 硬件连接

JY61P 使用 UART（串口）通信，4 根线：

| JY61P 引脚 | 接 MCU | 说明 |
|-----------|--------|------|
| VCC | 3.3V 或 5V | 供电 |
| GND | GND | 地 |
| TX | MCU 的 RX | JY61P 发送 → MCU 接收 |
| RX | MCU 的 TX | （可选）MCU 发送 → JY61P 接收 |

### 通信参数

- **波特率：** 115200 bps（比常见的 9600 快很多，因为数据量大）
- **数据位：** 8
- **停止位：** 1
- **校验位：** 无

```c
// UART 初始化（以 MSPM0 为例）
// 波特率 = 115200, 8N1
DL_UART_Main_init(UART_INST, 115200);
```

## 数据包格式

JY61P 会自动持续发送数据包——不需要 MCU 先请求，它自己一直发。每个数据包的格式：

```
数据包格式（加速度包，0x51）：
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ 0x55 │ 0x51 │ AxL  │ AxH  │ AyL  │ AyH  │ AzL  │ AzH  │ TL   │ TH   │ SUM  │
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
  帧头   包类型  加速度 X    加速度 Y    加速度 Z    温度       校验和

数据包格式（角速度包，0x52）：
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ 0x55 │ 0x52 │ wxL  │ wxH  │ wyL  │ wyH  │ wzL  │ wzH  │ TL   │ TH   │ SUM  │
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘

数据包格式（角度包，0x53）：
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ 0x55 │ 0x53 │ RollL│ RollH│PitchL│PitchH│ YawL │ YawH │ TL   │ TH   │ SUM  │
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
```

每个包 11 字节。数据的低位在前（小端序）。校验和 = 除了帧头 0x55 外所有字节累加和的低 8 位。

## 数据解析代码

```c
// ==================== jy61p.h ====================
#ifndef JY61P_H
#define JY61P_H

#include <stdint.h>

typedef struct {
    float roll;     // 滚转角（度）
    float pitch;    // 俯仰角（度）
    float yaw;      // 偏航角（度）
    float wx, wy, wz; // 角速度（度/秒）
    float ax, ay, az; // 加速度（g）
} jy61p_data_t;

void jy61p_init(void);
void jy61p_parse_byte(uint8_t byte);
void jy61p_get_data(jy61p_data_t *data);
float jy61p_get_yaw(void);  // 最常用：直接获取偏航角

#endif

// ==================== jy61p.c ====================
#include "jy61p.h"
#include <string.h>

#define JY61P_BUF_SIZE  11

static uint8_t  rx_buf[JY61P_BUF_SIZE];
static uint8_t  rx_index = 0;
static jy61p_data_t g_imu_data = {0};

void jy61p_init(void) {
    // UART 已在配置工具中初始化（115200, 8N1）
    rx_index = 0;
    memset(&g_imu_data, 0, sizeof(g_imu_data));
}

// 这个函数在 UART 接收中断中调用，每次收到一个字节就调用一次
void jy61p_parse_byte(uint8_t byte) {
    rx_buf[rx_index] = byte;
    rx_index++;
    
    // 还没收到帧头就不处理
    if (rx_index == 1 && rx_buf[0] != 0x55) {
        rx_index = 0; // 不是帧头，重置
        return;
    }
    
    // 还没收满一个包（11 字节）就继续等
    if (rx_index < 11) {
        return;
    }
    
    // 收满 11 字节，校验
    rx_index = 0;
    
    // 计算校验和
    uint8_t sum = 0;
    for (int i = 0; i < 10; i++) {
        sum += rx_buf[i];
    }
    if (sum != rx_buf[10]) {
        return; // 校验失败，丢弃
    }
    
    // 根据包类型解析数据
    switch (rx_buf[1]) {
        case 0x51: // 加速度包
            g_imu_data.ax = (int16_t)((rx_buf[3] << 8) | rx_buf[2]) / 32768.0f * 16.0f;
            g_imu_data.ay = (int16_t)((rx_buf[5] << 8) | rx_buf[4]) / 32768.0f * 16.0f;
            g_imu_data.az = (int16_t)((rx_buf[7] << 8) | rx_buf[6]) / 32768.0f * 16.0f;
            break;
        case 0x52: // 角速度包
            g_imu_data.wx = (int16_t)((rx_buf[3] << 8) | rx_buf[2]) / 32768.0f * 2000.0f;
            g_imu_data.wy = (int16_t)((rx_buf[5] << 8) | rx_buf[4]) / 32768.0f * 2000.0f;
            g_imu_data.wz = (int16_t)((rx_buf[7] << 8) | rx_buf[6]) / 32768.0f * 2000.0f;
            break;
        case 0x53: // 角度包（竞赛中最常用）
            g_imu_data.roll  = (int16_t)((rx_buf[3] << 8) | rx_buf[2]) / 32768.0f * 180.0f;
            g_imu_data.pitch = (int16_t)((rx_buf[5] << 8) | rx_buf[4]) / 32768.0f * 180.0f;
            g_imu_data.yaw   = (int16_t)((rx_buf[7] << 8) | rx_buf[6]) / 32768.0f * 180.0f;
            break;
    }
}

void jy61p_get_data(jy61p_data_t *data) {
    *data = g_imu_data;
}

float jy61p_get_yaw(void) {
    return g_imu_data.yaw;
}

// ==================== 在 UART 中断中使用 ====================
// UART 接收中断回调
void UART_IRQHandler(void) {
    uint8_t byte = DL_UART_Main_receiveData(UART_INST);
    jy61p_parse_byte(byte);
}
```

## 用偏航角实现精确转弯

```c
// 让车精确右转 90 度
void turn_right_90_degrees(void) {
    float start_yaw = jy61p_get_yaw();  // 读取起始角度
    float target_yaw = start_yaw + 90.0f;
    
    // 处理角度回绕（Yaw 范围是 -180 到 +180）
    if (target_yaw > 180.0f) {
        target_yaw -= 360.0f;
    }
    
    // 原地右转
    motor_a_forward(50);   // 左轮前进
    motor_b_reverse(50);   // 右轮后退 → 原地右转
    
    float current_yaw;
    do {
        current_yaw = jy61p_get_yaw();
        // 小延时防止 CPU 跑冒烟
        delay_ms(5);
    } while (current_yaw < target_yaw); // 还没转到目标角度，继续转
    
    motor_stop(); // 转到位，停车
}
```

## 陀螺仪的两个重要特性

### 1. 上电稳定时间

JY61P 上电后需要 **2-5 秒** 进行自我校准。在这期间数据是不准的。代码中需要等待：

```c
void jy61p_wait_stable(void) {
    printf("等待陀螺仪稳定...\r\n");
    delay_ms(3000);  // 上电后等 3 秒
    printf("陀螺仪就绪！\r\n");
}
```

### 2. 漂移（Drift）

这是所有陀螺仪都有的问题：短时间（几秒到几十秒）精度很好，但长时间（几分钟）会慢慢"漂移"——角度读数会悄悄地偏掉。

| 特性 | 陀螺仪 | 编码器 |
|------|--------|--------|
| 短期精度 | 高 | 中 |
| 长期精度 | 低（漂移） | 高（不漂移） |
| 测什么 | 角速度 → 积分 = 角度 | 轮子转了多少圈 |

::: tip 陀螺仪和编码器互补
在竞赛中，通常用 **编码器 + 陀螺仪结合** 来测角度。短时间用陀螺仪（响应快），长时间用编码器校准（左轮和右轮的里程差可以算出转弯角度），互补滤波得到可靠的姿态估计。
:::

## 常见问题排查

| 现象 | 可能原因 | 解决方法 |
|------|---------|---------|
| 读到全是 0 | 波特率不对 | 确认是 115200 |
| 数据乱跳 | 接线松动 | 检查 TX/RX/GND 是否接触良好 |
| 角度一直偏 | 上电时车在晃动 | 上电后保持静止 5 秒 |
| Yaw 角一直往一个方向飘 | 正常漂移 | 用编码器辅助校准 |

---

## 小结

JY61P 把复杂的姿态解算封装在模块内部，你只需要读串口就能拿到角度和角速度。上电后等 5 秒让它稳定，Z 轴会漂移是正常的——需要用其他传感器（如编码器）做融合校正。

---

# MaixCAM Pro — 视觉模块

## 它是什么

MaixCAM Pro 是一块小型视觉处理模块，搭载了 **K210 芯片**——这是一颗专为 AI 视觉任务设计的处理器。把它接在小车上，车就有了"眼睛"。

**核心规格：**

| 参数 | 数值 | 说明 |
|------|------|------|
| 芯片 | K210 | RISC-V 双核 64 位 |
| 主频 | 400 MHz | 可超频到 800 MHz |
| KPU | 神经网络加速器 | 0.8 TOPS（专门跑 AI 模型） |
| RAM | 8 MB | 够跑中等规模模型 |
| 摄像头 | OV2640 (200 万像素) | 可拍照、可录视频 |
| 屏幕 | 2.4 寸 TFT LCD | 调试用（车跑起来不看） |

## 它不是干什么的

::: warning MaixCAM Pro 不做什么
- **不输出原始图像给 MCU。** 如果你想让 MCU 处理摄像头图像，MaixCAM 不合适——它自己处理图像，只把结果发给 MCU。
- **不替代 MCU。** MaixCAM 和 MCU 各司其职：MaixCAM 负责"看"，MCU 负责"控制"。
:::

## 它和 MCU 的分工

把 MCU 想象成司机，MaixCAM 想象成副驾驶：

```
MaixCAM（副驾驶，K210）：
  "前面有一个红色标记，在左侧 20cm 处"
  "黑线位置：车偏右了 5cm"
  "前方检测到停止线"

MCU（司机，STM32/MSPM0）：
  收到 "偏右了 5cm" → 计算左转纠正 → 控制电机
  收到 "红色标记" → 状态切换 → 执行对应任务
```

## 通信方式

MaixCAM 通过 **UART（串口）** 和 MCU 通信。它用文本格式发送结果：

```
数据格式（MaixCAM → MCU）：
L:120,R:80,C:1\r\n

含义：
L:120  → 黑线在画面左侧 120 像素处
R:80   → 黑线在画面右侧 80 像素处  
C:1    → 颜色标记类型 = 1（如 1=红色, 2=蓝色, 3=绿色）
\r\n   → 回车换行（包结束标志）
```

这是一个简单的文本协议——每条数据以 `\r\n` 结尾，内容用逗号分隔，字段用冒号分隔。MCU 解析这些文本就行。

## MCU 端的解析代码

```c
// ==================== maxicam.h ====================
#ifndef MAXICAM_H
#define MAXICAM_H

#include <stdint.h>

typedef struct {
    int16_t left_line;      // 左边线位置（像素），-1 = 没看到
    int16_t right_line;     // 右边线位置（像素），-1 = 没看到
    int16_t center_line;    // 中线位置（像素），-1 = 没看到
    uint8_t color_id;       // 颜色标记类型（0 = 无）
    int16_t color_x;        // 颜色标记的 X 坐标
    int16_t color_y;        // 颜色标记的 Y 坐标
} maxicam_data_t;

void maxicam_init(void);
void maxicam_parse_char(char c);
void maxicam_get_data(maxicam_data_t *data);

#endif

// ==================== maxicam.c ====================
#include "maxicam.h"
#include <string.h>
#include <stdio.h>

#define MAXICAM_BUF_SIZE    64

static char rx_buf[MAXICAM_BUF_SIZE];
static uint8_t rx_len = 0;
static maxicam_data_t g_cam_data = {0};

void maxicam_init(void) {
    // UART 初始化（波特率根据 K210 端脚本设置，通常 115200 或 921600）
    rx_len = 0;
    memset(&g_cam_data, 0, sizeof(g_cam_data));
}

// 在 UART 接收中断中逐个字符调用
void maxicam_parse_char(char c) {
    if (c == '\n') {
        // 收到一行的结束符
        if (rx_len > 0 && rx_buf[rx_len - 1] == '\r') {
            rx_buf[rx_len - 1] = '\0'; // 去掉 \r
        }
        rx_buf[rx_len] = '\0';
        
        // 解析 "L:120,R:80,C:1" 格式
        int l, r, c;
        if (sscanf(rx_buf, "L:%d,R:%d,C:%d", &l, &r, &c) == 3) {
            g_cam_data.left_line  = l;
            g_cam_data.right_line = r;
            g_cam_data.color_id   = c;
            g_cam_data.center_line = (l + r) / 2;
        }
        
        rx_len = 0; // 重置缓冲
    } else {
        // 普通字符，存入缓冲
        if (rx_len < MAXICAM_BUF_SIZE - 1) {
            rx_buf[rx_len++] = c;
        }
    }
}

void maxicam_get_data(maxicam_data_t *data) {
    *data = g_cam_data;
}

// ==================== 使用示例 ====================
void vision_line_following(void) {
    maxicam_data_t cam;
    int base_speed = 60;
    float Kp = 0.2f;
    int screen_center = 160; // 假设画面宽度 320，中点是 160
    
    while (1) {
        maxicam_get_data(&cam);
        
        if (cam.center_line > 0) {
            // 计算偏差
            int error = screen_center - cam.center_line;
            int correction = (int)(Kp * error);
            
            int left  = base_speed - correction;
            int right = base_speed + correction;
            
            // 限幅
            if (left  > 100) left  = 100;
            if (left  < -100) left  = -100;
            if (right > 100) right = 100;
            if (right < -100) right = -100;
            
            motor_a_set_speed(left);
            motor_b_set_speed(right);
        } else {
            // 没有看到线，停车或进入搜索模式
            car_stop();
        }
        
        delay_ms(20);
    }
}
```

## MaixCAM 端的程序

K210 运行自己的程序（MicroPython 或 C），和 MCU 的代码是**完全独立的**。你需要单独烧录到 K210 上。

```python
# MaixCAM 端的 MicroPython 示例（仅供参考，理解原理即可）
# 这个程序跑在 K210 上，不是跑在 MCU 上！

import sensor, image, time
from fpioa_manager import fm
from machine import UART

# 初始化摄像头
sensor.reset()
sensor.set_pixformat(sensor.RGB565)
sensor.set_framesize(sensor.QVGA)  # 320×240

# 初始化串口（连接 MCU）
fm.register(6, fm.fpioa.UART1_TX, force=True)
fm.register(7, fm.fpioa.UART1_RX, force=True)
uart = UART(UART.UART1, 115200)

while True:
    img = sensor.snapshot()
    
    # 找黑线（简化示例）
    lines = img.find_lines(threshold=1000, theta_margin=25, rho_margin=25)
    
    left_x = -1
    right_x = -1
    color_id = 0
    
    # 寻找红色标记
    blobs = img.find_blobs([(0, 50, 20, 80, -20, 40)], pixels_threshold=200)
    if blobs:
        color_id = 1  # 检测到红色
    
    # 发送给 MCU
    uart.write("L:%d,R:%d,C:%d\r\n" % (left_x, right_x, color_id))
    time.sleep_ms(20)
```

## MaixCAM 适合做什么

| 任务 | 适合度 | 说明 |
|------|--------|------|
| 道路边界检测 | ✅ 很好 | 视野广，能看远 |
| 十字路口识别 | ✅ 很好 | 灰度看不到全貌 |
| 颜色标记识别 | ✅ 很好 | 彩色摄像头天然优势 |
| 简单分类（形状等） | ✅ 好 | KPU 可以做轻量级分类 |
| 精确距离测量 | ⚠️ 一般 | 单目摄像头缺乏深度信息 |
| 替代灰度巡线 | ⚠️ 不推荐 | 灰度更简单更可靠 |

## 常见问题

::: warning 通信延迟
MaixCAM 处理一帧 + 发送数据需要 20-50ms。如果你的控制循环是 5ms（200Hz），摄像头数据每 4-10 个控制周期才更新一次。**不要用摄像头数据做高频闭环控制，它适合做"战略级"的感知（检测路口、识别标记）。**
:::

| 问题 | 原因 | 解决 |
|------|------|------|
| MCU 收不到数据 | 波特率不匹配 | 检查两端波特率一致 |
| 数据格式乱 | 文本解析有 bug | 用串口助手先看 K210 发的原始文本 |
| 画面卡顿 | K210 处理太慢 | 降低分辨率（QVGA 够用） |
| 通信断断续续 | 线太长 | 串口线 < 30cm，或降低波特率 |

---

## 小结

MaixCAM 让小车有了"眼睛"，但它只解决"看到什么"的问题，不解决"怎么走"的问题。控制决策始终在 MCU 端——摄像头数据适合做战略级感知（检测路口、识别标记），不适合做高频闭环控制。

# 无线串口调试与无线烧录

## 为什么需要无线

你的小车在赛道上跑着，后面拖着一根长长的 USB 线——画面很滑稽，更糟糕的是：

- USB 线拖在地上，影响小车运动（转弯时被拽住）
- 线长有限（USB 规范不超过 5 米）
- 不小心绊到线 → 小车飞出去 → 所有东西摔地上

**车在跑，线必须解放。**

## 方案：蓝牙无线串口

最简单、最便宜的无线调试方案：**用一个蓝牙串口模块，把你的 printf() 信息通过蓝牙发送到电脑。**

```
小车端                          电脑端
┌──────────┐    TX/RX    ┌──────────┐    蓝牙    ┌──────────┐
│   MCU    │←──────────→│ HC-05    │~~~无线的~~~│ 电脑蓝牙  │
│          │  有线连接    │ 蓝牙模块  │            │ 虚拟串口  │
└──────────┘             └──────────┘            └──────────┘
                                                        │
                                                  串口助手软件
                                                  (显示 printf)
```

### 原理类比

就像把有线耳机换成蓝牙耳机——原来通过 USB 线传输的串口数据，现在通过蓝牙无线传输。电脑上会多出一个"虚拟串口"（COM 口），和插着 USB 线时看到的串口一模一样。

## 需要的硬件

| 硬件 | 价格 | 说明 |
|------|------|------|
| HC-05 蓝牙模块 | ~12 元 | 主从一体，最常用 |
| HC-06 蓝牙模块 | ~10 元 | 仅从机模式，更简单 |
| 电脑蓝牙 | 自带或 USB 蓝牙适配器 ~15 元 | 台式机通常需要外接 |

推荐 **HC-05**——功能更全，可以进入 AT 命令模式修改配置（波特率、名称、密码）。

## 硬件接线

HC-05 和 MCU 之间的连接：

```
MCU 端                         HC-05 蓝牙模块
UART TX ───────────────────→  RX
UART RX ←───────────────────  TX
3.3V   ───────────────────→  VCC
GND    ────────────────────  GND
```

::: danger 电压警告
HC-05 的 VCC 标注支持 3.6V-6V，但逻辑电平是 3.3V。如果你的 MCU 是 5V 系统（如 Arduino），需要在 RX 线上加电平转换（分压电阻），否则可能烧坏 HC-05。
:::

## 配置 HC-05

第一次使用需要配置参数。给 HC-05 上电时按住模块上的小按键（或把 EN 引脚拉高），进入 AT 命令模式（指示灯慢闪）。

```
// 在串口助手中发送 AT 命令（波特率 38400，HC-05 默认 AT 模式波特率）
AT                    // 测试连接 → 返回 OK
AT+NAME=MyCar        // 设置蓝牙名称为 "MyCar"
AT+PSWD=1234         // 设置配对密码
AT+UART=115200,0,0   // 设置波特率 = 115200, 1 停止位, 无校验
AT+ROLE=0            // 设为从机模式
```

配置好后断电重启，HC-05 就回到正常通信模式（指示灯快闪）。

## MCU 端代码

使用蓝牙串口和普通串口**完全一样**——你不必修改任何 printf 相关的代码：

```c
// 初始化蓝牙串口（和普通串口一样！）
void wireless_uart_init(void) {
    // UART 配置：115200, 8N1
    DL_UART_Main_init(BLUETOOTH_UART_INST, 115200);
}

// 发送调试信息（和 printf 一样！）
void debug_printf(const char *format, ...) {
    char buf[128];
    va_list args;
    va_start(args, format);
    vsnprintf(buf, sizeof(buf), format, args);
    va_end(args);
    
    // 通过蓝牙串口发送
    for (int i = 0; buf[i] != '\0'; i++) {
        DL_UART_Main_transmitDataBlocking(BLUETOOTH_UART_INST, buf[i]);
    }
}

// 使用示例
void main(void) {
    wireless_uart_init();
    
    float speed = 0.75;  // m/s
    float yaw = 45.3;    // 度
    
    debug_printf("速度: %.2f m/s, 角度: %.1f 度\r\n", speed, yaw);
    // 电脑端的串口助手实时显示这些信息！
}
```

## 实战中的无线调试

### 实时监控变量

把小车的关键变量通过蓝牙发出来，在电脑上实时观察：

```c
// 每 100ms 发一次调试数据
void debug_telemetry(void) {
    static uint32_t last_time = 0;
    uint32_t now = get_system_tick_ms();
    
    if (now - last_time >= 100) {
        last_time = now;
        
        // 发送格式化的遥测数据
        debug_printf("S:%.2f Y:%.1f L:%d R:%d E:%d\r\n",
            encoder_get_speed(),   // 速度
            jy61p_get_yaw(),       // 偏航角
            motor_left_pwm,        // 左轮 PWM
            motor_right_pwm,       // 右轮 PWM
            line_error             // 巡线误差
        );
    }
}
```

在电脑端用串口助手或自制的 Python 脚本，可以实时绘图，看到速度曲线的变化——**比事后分析高效得多。**

## 无线烧录（OTA）

::: tip OTA = Over-The-Air 空中升级
无线烧录是通过蓝牙把新的固件程序发送到 MCU，MCU 的 Bootloader（启动引导程序）接收后写入 Flash，实现"不插线更新代码"。
:::

### 原理

```
1. 电脑编译生成 .hex 或 .bin 固件文件
2. 通过蓝牙串口发送给 MCU
3. MCU 的 Bootloader 接收固件 → 写入 Flash
4. 写入完成 → MCU 自动复位 → 运行新程序
```

### 条件

- MCU 需要烧录了支持 OTA 的 Bootloader
- Bootloader 和应用程序分占不同的 Flash 区域
- 蓝牙串口必须稳定（无线环境干扰小）

### 竞赛中的价值

想象这个场景：小车在赛道上，你发现参数需要调整。如果每次都要：

```
无线方案：改代码 → 编译 → 无线下载（10秒）→ 小车继续跑 ✅
有线方案：追小车 → 抓回来 → 插线 → 下载 → 放回去（2分钟）❌
```

::: warning 无线烧录的局限性
- 速度比有线慢（蓝牙串口通常 115200 bps）
- 不稳定时可能传一半断了
- 不是所有竞赛 MCU 都有现成的 OTA Bootloader
- **有线烧录作为兜底方案始终保留**
:::

## HC-05 vs HC-06 对比

| 特性 | HC-05 | HC-06 |
|------|-------|-------|
| 主从模式 | 主/从都可 | 仅从机 |
| 可配置性 | AT 命令丰富 | AT 命令有限 |
| 价格 | ~12 元 | ~10 元 |
| 适用场景 | 需要灵活配置 | 只做从机（够用） |
| 推荐 | ✅ 首选 | ✅ 够用 |

## 常见问题

| 问题 | 可能原因 | 解决方法 |
|------|---------|---------|
| 搜不到蓝牙 | HC-05 未上电 / 已连接其他设备 | 检查供电，断开其他连接 |
| 连上了但没数据 | 波特率不匹配 | 检查 MCU、HC-05、电脑串口助手三者波特率一致 |
| 数据乱码 | 波特率或校验位不对 | 都用 8N1 |
| 通信距离短 | 金属障碍物 / 电池电压低 | 模块远离电机和金属件 |

花 12 元买一个 HC-05 蓝牙模块，你就能在赛场上无线调试——看着实时数据调 PID 参数，而不是追着小车跑来跑去。这是竞赛中最值得投入的小成本改进。

# 实战路线图：从点灯到巡线小车

## 这篇文章是什么

你读了很多理论知识——GPIO、PWM、UART、I2C、PID……但你可能在想：**"我到底该怎么开始？"**

这篇文章就是你的答案。它按照从易到难的顺序，列出了 **12 个步骤**，每一步你都会学到新概念、用到新硬件、写出新代码。当你完成全部 12 步时，你就拥有了一辆完整的、能巡线、能精确转弯的竞赛小车。

## 这个路线图的设计哲学

1. **一次只加一个新东西。** 每步只引入一个新技术，不会让你同时面对五个陌生概念。
2. **每一步都能看到东西在动。** 光、电机转、屏幕亮——有反馈才有成就感。
3. **前面每一步都在为后面的闭环铺路。** 点灯学会了 GPIO → 驱动电机也是 GPIO；串口学会了通信 → 陀螺仪也是串口。

---

## 第 1 步：点亮 LED（GPIO 输出）

**学到的概念：** GPIO（通用输入输出）是什么、高电平、低电平、推挽输出、限流电阻

**需要的硬件：** MCU 开发板 ×1、LED 灯珠 ×1、220Ω 电阻 ×1、面包板

**为什么这步重要：** GPIO 是所有硬件控制的入口——点灯就是"GPIO 输出高电平"。后续你控制电机方向、拉高 STBY 使能、选通多路选择器，本质上都是在做同一件事：**让某个 MCU 引脚输出高电平或低电平**。

```c
#include "ti_msp_dl_config.h"

int main(void) {
    SYSCFG_DL_init();  // 系统初始化
    
    while (1) {
        DL_GPIO_setPins(LED_PORT, LED_PIN);   // 亮
        delay_ms(500);
        DL_GPIO_clearPins(LED_PORT, LED_PIN); // 灭
        delay_ms(500);
    }
}
```

::: tip "Hello World" 的变体
看完闪烁后，尝试修改延时时间让它闪得更快或更慢。然后尝试用两个 LED 交替亮灭。这能帮你建立"软件控制硬件"的直觉。
:::

---

## 第 2 步：按键控制（GPIO 输入 + 上下拉）

**学到的概念：** GPIO 输入模式、上拉电阻、下拉电阻、消抖

**需要的硬件：** 按键 ×1、10kΩ 电阻 ×1（或利用 MCU 内部上拉）

**核心代码：**

```c
// 读取按键状态（按下 = 低电平，因为接了上拉电阻）
if (DL_GPIO_readPins(BUTTON_PORT, BUTTON_PIN) == 0) {
    delay_ms(20);  // 消抖：等 20ms 再确认
    if (DL_GPIO_readPins(BUTTON_PORT, BUTTON_PIN) == 0) {
        // 按键真的被按下了
        DL_GPIO_togglePins(LED_PORT, LED_PIN);  // 切换 LED
        while (DL_GPIO_readPins(BUTTON_PORT, BUTTON_PIN) == 0);  // 等松开
    }
}
```

::: warning 按键抖动
机械按键按下时不是干净的高低跳变——它会抖动几十微秒。如果不消抖（等 20ms 再读），一次按键可能被你代码读到 5 次。"消抖"这个操作在编码器、限位开关等所有机械触点场景都要用。
:::

---

## 第 3 步：串口打印（UART + printf）

**学到的概念：** UART（通用异步收发传输器）、波特率、数据帧格式（起始位、数据位、停止位）、printf 重定向

**需要的硬件：** USB 转串口模块 ×1（或直接用板载调试器）

**核心代码：**

```c
#include <stdio.h>

// 重定向 printf 到 UART
int fputc(int ch, FILE *f) {
    DL_UART_Main_transmitDataBlocking(DEBUG_UART_INST, (uint8_t)ch);

int main(void) {
```

没有串口打印，你就像一个蒙着眼睛的医生在做手术。有了 serial print，你可以随时看到变量的值、程序运行到哪一步。**这是嵌入式开发的"超能力"——在本课程之后的所有步骤中，始终用 printf 来观察代码行为。**

## 第 4 步：呼吸灯（定时器 + PWM）

**学到的概念：** 定时器（Timer）、PWM（脉冲宽度调制）、占空比（Duty Cycle）、频率

**需要的硬件：** 和步骤 1 一样（LED + 电阻）

**核心代码：**

```c
// 呼吸灯：占空比从 0% 慢慢变到 100%，再慢慢变回 0%
int duty = 0;
int direction = 1;

while (1) {
    set_pwm_duty(duty);  // 设置占空比 = duty%
    duty += direction;
    if (duty >= 100) direction = -1;
    if (duty <= 0)   direction = +1;
    delay_ms(10);  // 控制呼吸速度
}
```

**PWM 的直观理解：** 如果你用开关来回拨动控制灯的亮度——"开"的时间比例越大，灯越亮。PWM 就是这个"快速开关"——频率高到人眼看不出来闪烁（>50Hz），只能感知到平均亮度。

## 第 5 步：读取电位器（ADC）

**学到的概念：** ADC（模数转换器）、分辨率（12 位 = 0-4095）、参考电压、采样时间

**需要的硬件：** 电位器（旋转可调电阻）×1

**核心代码：**

```c
uint16_t adc_value;

// 读取 ADC 值
adc_value = DL_ADC12_getConversionResult(ADC_INST);

// 计算电压（假设 3.3V 参考电压）
float voltage = (float)adc_value / 4095.0f * 3.3f;

printf("ADC 原始值: %d, 电压: %.2fV\r\n", adc_value, voltage);
```

**和后面内容的关系：** 灰度的光电三极管输出也是模拟电压 → 也是通过 ADC 读取。电池电压检测也是 ADC。输入本质都是"一个变化的电压值"。

## 第 7 步：控制舵机（PWM + 占空比映射）

**学到的概念：** 舵机控制协议（500-2500μs 脉冲宽度对应 0-180°）、占空比到角度的映射

**需要的硬件：** SG90 舵机 ×1

```c
// 舵机角度到 PWM 占空比的映射
// SG90: 0.5ms = 0°, 1.5ms = 90°, 2.5ms = 180°
// 50Hz 周期 = 20ms
void servo_set_angle(uint8_t angle) {
    // angle: 0-180 度
    // 脉冲宽度 = 500 + (angle / 180) * 2000 微秒
    uint16_t pulse_us = 500 + (uint32_t)angle * 2000 / 180;
    
    // 占空比 = 脉冲宽度 / 周期
    uint16_t duty = (uint32_t)pulse_us * 1000 / 20000;  // 0-1000 对应 0-100%
    set_pwm_duty(duty);
}
```

## 第 9 步：测量速度（编码器 + 定时器正交解码）

**学到的概念：** 正交编码、4× 解码、速度闭环的概念、定时器的编码器模式

**需要的硬件：** 霍尔编码器（集成在电机尾部）×2

**核心代码：**

```c
// 每 10ms 调用一次
void speed_update(void) {
    int32_t count_now = DL_TimerG_getCounterValue(ENCODER_TIMER);
    int32_t count_diff = count_now - g_last_count;
    g_last_count = count_now;
    
    // 速度 = 距离 / 时间
```

## 第 10 步：巡线（灰度传感器 + P 控制器 + 电机 = 闭环）

**学到的概念：** 闭环控制、反馈、误差、P 控制器、加权平均法

**需要的硬件：** 8 路灰度传感器模块 ×1

**核心代码：**

```c
// 这是整个路线图的核心——闭环巡线
void line_following_loop(void) {
    uint16_t gray[8];
    float Kp = 0.05f;      // P 系数
    int base_speed = 60;   // 基础速度
    
    while (1) {
        // 1. 读传感器（步骤 5 的 ADC）
        grayscale_read_all(gray);
        
        // 2. 计算黑线位置（加权平均）
        float line_pos = calculate_line_position(gray);
        
        // 3. 计算误差
        float error = 0.0f - line_pos;
        
        // 4. P 控制：修正 = Kp × 误差
        float correction = Kp * error;
        
        // 5. 控制电机（步骤 8 的电机驱动）
        motor_a_set_speed(base_speed + (int)correction);
        motor_b_set_speed(base_speed - (int)correction);
        
        delay_ms(10); // 100Hz 控制频率
    }
}
```

**闭环的魔力：** 把这个循环跑起来——车自己沿着黑线走，不需要你干预。它的工作方式是：偏离→检测→纠正→偏离→检测→纠正……每秒钟 100 次的"看-算-调"循环。

## 第 11 步：精确转弯（陀螺仪 + PID）

**学到的概念：** PID 控制器（比例 + 积分 + 微分）、陀螺仪角度反馈、角度闭环

**需要的硬件：** JY61P 陀螺仪模块 ×1

```c
// 用 PID 控制精确转弯到目标角度
float pid_angle_control(float target_angle, float current_angle) {
    static float integral = 0;
    static float last_error = 0;
    
    float Kp = 2.0f, Ki = 0.01f, Kd = 0.5f;
    
    // 计算角度误差（处理 -180 到 +180 的环绕）
    float error = target_angle - current_angle;
    if (error > 180.0f)  error -= 360.0f;
    if (error < -180.0f) error += 360.0f;
    
    // PID 三项
    integral += error;
    float derivative = error - last_error;
    last_error = error;
    
    // 输出 = P + I + D
    float output = Kp * error + Ki * integral + Kd * derivative;
    
    // 限幅
    if (output > 100.0f)  output = 100.0f;
    if (output < -100.0f) output = -100.0f;
    
    return output;
}

// 精确右转 90 度
void turn_right_90_exact(void) {
    float target = jy61p_get_yaw() + 90.0f;
    if (target > 180.0f) target -= 360.0f;
    
    while (1) {
        float current = jy61p_get_yaw();
        float output = pid_angle_control(target, current);
        
        // output > 0 → 右转，output < 0 → 左转
        motor_a_set_speed((int)output);
        motor_b_set_speed(-(int)output);
        
        // 误差小于 2 度，认为到位
        float error = target - current;
        if (error < 0) error = -error;
        if (error < 2.0f) break;
        
        delay_ms(5);
    }
    car_stop();
}
```

::: tip 从 P 到 PID
- **P（比例）：** 你当前偏离了多远？偏离越大，纠正力度越大。
- **I（积分）：** 你过去累计偏了多久？如果 P 不够把你拉回来，I 会累积误差直到把你拉回来。
- **D（微分）：** 你现在偏的速度是多快？如果你正在快速偏离，D 会"提前刹车"防止冲过头。

P 能跑起来，PI 能跑准，PID 能跑稳。
:::

**学到的概念：** 有限状态机（FSM）、多传感器融合、系统集成

**需要的硬件：** 前面所有硬件的集合 + 蓝牙 HC-05（无线调试）

```c
typedef enum {
    STATE_IDLE,         // 空闲（等待开始）
    STATE_LINE_FOLLOW,  // 巡线中
    STATE_TURN_LEFT,    // 左转
    STATE_TURN_RIGHT,   // 右转

    STATE_ERROR         // 错误处理
} car_state_t;

// 状态机主循环
void state_machine_loop(void) {
    car_state_t state = STATE_IDLE;
    
    while (1) {
        // 1. 更新所有传感器数据
        grayscale_read_all(gray_values);
        encoder_update();
        maxicam_get_data(&cam_data);
        float yaw = jy61p_get_yaw();
        
        // 2. 发送遥测数据（无线调试）
        debug_telemetry(speed, yaw, state);
        
        // 3. 状态机决策
        switch (state) {
            case STATE_IDLE:
                if (start_button_pressed()) {
                    state = STATE_LINE_FOLLOW;
                }
                break;
                
            case STATE_LINE_FOLLOW:
                line_following_p_control();
                
                // 检测路口
                if (detect_cross(gray_values)) {
                    state = STATE_CROSS_DETECT;
                }
                // 检测停止线
                if (detect_stop_line(cam_data)) {
                    state = STATE_STOP;
                }
                break;
                
            case STATE_TURN_LEFT:
                turn_angle_exact(-90.0f);
                state = STATE_LINE_FOLLOW;
                break;
                
            case STATE_STOP:
                car_stop();
                // 任务完成
                break;
                
            default:
                state = STATE_ERROR;
                break;
        }
        
        delay_ms(10); // 100Hz 主循环
    }
}
```

# PCB 设计全流程概览

## 什么是 PCB？

**PCB**（Printed Circuit Board，印刷电路板）就是你在电子设备里看到的那块绿色（或其他颜色）的板子，上面焊满了各种芯片和元件。它的作用是把电子元件用铜线（走线）连接起来，替代一团乱麻的飞线。

打个比方：PCB 就像一座城市的地图。芯片是建筑，铜走线是道路，电源线是主干道，信号线是小巷。设计 PCB 就是当城市规划师——你把"房子"放在哪儿，"路"怎么修，都得提前想好。

## 完整流程（七步走）

设计一块 PCB 并拿到手，大致需要以下步骤：

1. **需求分析** —— 这块板子要干什么？控制几个电机？采集什么传感器？
2. **元件选型** —— 根据需求挑选芯片和元件（MCU、电源芯片、连接器等）
3. **原理图设计** —— 在电脑上用 EDA 软件画出电路连接关系
4. **PCB 布局布线** —— 把原理图变成实物布局，放置元件、画走线
5. **打样制造** —— 把设计文件发给工厂生产
6. **焊接** —— 把元件焊到板子上
7. **测试** —— 通电，验证功能

## 工具选择：嘉立创 EDA（立创 EDA）

对于初学者和竞赛场景，强烈推荐**嘉立创 EDA**（立创 EDA）。它是国产免费的 EDA 软件，有以下优势：

- **完全免费**，不需要动辄数万元的商业授权
- **中文界面**，学习成本低
- **一键下单**：设计完成后直接在软件内下单打样，无需导出文件折腾
- **海量元件库**：立创商城的元件几乎都有封装，拖出来就能用
- **在线 + 客户端**：既可以在网页上用，也可以下载客户端离线使用

其他选择如 Altium Designer、KiCad（开源免费）也常用，但对于竞赛来说，立创 EDA 的"设计即下单"体验无可匹敌。

::: tip 省时技巧
立创 EDA 的元件库和立创商城是打通的。你选元件时就能看到库存和价格，不会出现"设计好了发现买不到"的窘境。
:::

## 时间安排

对于一块中等复杂度的控制电路板：

| 阶段 | 预计时间 | 说明 |
|------|----------|------|
| 原理图设计 | 0.5 ~ 1 天 | 电路不复杂时很快 |
| PCB 布局布线 | 0.5 ~ 1 天 | 两层板通常够用 |
| 打样制造 | 3 ~ 5 天 | 嘉立创快板可加急到 24 小时 |
| 焊接 | 半天 ~ 1 天 | 手工焊，熟能生巧 |
| 测试 | 半天 | 通电、烧录、验证 |

总共约 **5 ~ 8 天**可以完成一轮迭代。

## 竞赛策略：模块化设计

比赛中时间紧张，临时设计整块板子来不及。正确的做法是**赛前积累**：

- 赛前把常用模块（电源模块、MCU 最小系统、电机驱动、传感器接口）都设计好、打样验证通过
- 比赛时根据具体题目，将已有模块**拼合**成一块主板，或直接用排针排母连接各模块

::: warning 血的教训
不要等到比赛才开始画板子！至少提前一个月把所有可能用到的模块都验证通过。一块没调通的板子在比赛现场就是一块废铁。
:::

## 原理图 vs PCB 布局

初学者容易混淆这两个概念：

- **原理图（Schematic）**：描述的是"电路逻辑"——谁和谁连接。元件可以随意摆放，连线只是逻辑关系。类似于代码里的变量引用关系。
- **PCB 布局（Layout）**：描述的是"物理实现"——元件实际放在哪儿，铜走线怎么走。这决定了板子的电气性能和可制造性。类似于把变量分配到具体的内存地址。

原理图是对（连接正确），PCB 还要好（性能好、能生产）。

```c
// 类比：原理图 = 函数声明（逻辑关系）
void setup() {
    pinMode(13, OUTPUT);      // LED 连在引脚 13
    pinMode(A0, INPUT);       // 传感器连在 A0
}

// PCB 布局 = 实际布线（物理实现）
// 引脚 13 到 LED 的铜线距离：2cm
// A0 到传感器的铜线不能靠近电机驱动线
// 这些都是在 PCB 设计阶段决定的
```

# 电源设计 — LDO 与 DC-DC

## 电源为什么重要？

所有芯片都需要电源才能工作。如果供电不稳，MCU 会随机重启、ADC 读数乱跳、电机莫名抖动。电源是电路的心脏——心脏不好，全身都出问题。

**耐心检查供电线路**是排查硬件故障的第一原则。很多看似是代码 bug 的问题，根因其实是电源噪声或电压跌落。

## 两种降压方案

嵌入式中最常见的需求是：把电池（7.4V、12V 等）降到 MCU 需要的 3.3V 或 5V。有两种主流方案：

### 方案一：线性稳压器（LDO）

**LDO**（Low Dropout Regulator，低压差线性稳压器）是一种简单的降压芯片。典型型号：**AMS1117-3.3**。

工作原理：把多余电压以**热量**的形式消耗掉。输入 5V，输出 3.3V，多出来的 1.7V 就变成热能散掉了。

打比方：LDO 就像你把一桶水的龙头开小一点——流出来的水少了，但多余的水压在管子里变成了阻力（热）。

**优点：**
- 电路极简，通常只需输入输出各一个电容
- 输出纹波极低（噪声小），适合给 ADC、运放等模拟电路供电
- 价格便宜，几分钱到几毛钱

**缺点：**
- 效率低，压差越大越浪费。输入 12V 输出 3.3V 时效率只有 27.5%，73% 的电变成热量

**效率与功耗公式：**

```c
// LDO 功耗计算
效率 = Vout / Vin           // 例如：3.3V / 12V ≈ 27.5%
发热功率 = (Vin - Vout) × I  // 12V → 3.3V，电流 500mA
         = (12 - 3.3) × 0.5
         = 4.35W              // 4.35 瓦纯发热！会烫手
```

### 方案二：DC-DC 开关电源（Buck）

**DC-DC Buck**（降压型开关电源）通过高频开关 + 电感储能来实现降压。典型型号：**MP1584**、**LM2596**。

打比方：DC-DC 就像你快速开关水龙头，每次开一瞬间接一点水到桶里，关掉等水流稳定——这样接的水量刚好是你需要的，几乎不浪费。

**优点：**
- 效率高（85%~95%），发热小
- 支持大电流（几安培）
- 输入电压范围宽

**缺点：**
- 电路复杂，需要电感、二极管、多个电容
- 输出有开关纹波（几十 mV 的高频噪声）
- 布局要求高，走线不好会导致严重 EMI

## 选型对比

| 特性 | LDO（AMS1117） | DC-DC Buck（MP1584） |
|------|----------------|----------------------|
| 效率 | 低（压差大时 < 30%） | 高（~90%） |
| 输出噪声 | 极低 | 有几十 mV 纹波 |
| 电路复杂度 | 极简 | 中等 |
| 最大电流 | 通常 1A | 2A ~ 3A |
| 发热 | 压差大时严重 | 轻微 |
| 适合场景 | 模拟电路、小电流 | 大电流、大压差 |
| 价格 | 几毛钱 | 几块钱 |

## 选型原则

::: tip 选型口诀
**小电流（<500mA）+ 小压差（<2V）→ LDO**  
**大电流或大压差 → DC-DC**  
**模拟精密电路（ADC、运放）→ LDO 供电，或用 DC-DC + LDO 两级（先 DC-DC 粗降再 LDO 精稳）**
:::

例如：12V 电池给 STM32（3.3V 100mA）供电 —— 先用 DC-DC 降到 5V，再用 LDO（AMS1117-3.3）降到 3.3V。既保证了效率，又保证了电源纯净。

## 去耦电容

**每个芯片的 VCC 引脚旁边**都必须放去耦电容。标准配置：

- 一个 **100nF（0.1µF）** 陶瓷电容 —— 滤高频噪声
- 一个 **10µF** 电解电容或陶瓷电容 —— 储能，应对电流瞬变

打比方：去耦电容就像芯片家门口的**迷你水库**。芯片突然需要大水（比如 GPIO 翻转瞬间），水库立刻供水，不用等远处的主电源慢慢送过来。

```c
// 没有去耦电容的后果（类比）
// 芯片："我要喝水！"
// 主电源："等等，水在路上，堵车..."
// → 芯片电压瞬间跌落 → 复位 / 逻辑错误

// 有去耦电容
// 芯片："我要喝水！"
// 电容（家门口的水缸）："来，先喝我的！"
// 主电源慢慢补充水缸
```

## 滤波电路

除了去耦电容，还可以加入额外的滤波来净化电源：

- **LC 滤波**：一个电感（L）串在电源线上，一个电容（C）并在地上。电感阻挡高频噪声通过，电容把噪声短路到地。
- **π 型滤波**：电容 → 电感 → 电容，形似 π 字符。比单级 LC 滤波效果更好。

::: warning 注意
磁珠（Ferrite Bead）不是电感！磁珠专门吸收高频噪声并将其转化为热量，而电感是储能元件。在电源滤波中磁珠常被误用电感符号，注意区分。
:::

# USB 协议基础

## USB 不仅仅是供电！

很多初学者以为 USB 就是"5V 电源线"。**错！** USB 是一条既能供电又能通信的复杂总线。电源只是它的一个功能，数据传输才是它的核心。

## USB 2.0 物理接口

标准 USB 2.0 有 4 根引脚：

| 引脚 | 名称 | 功能 | 常见线色 |
|------|------|------|----------|
| 1 | VBUS | +5V 电源 | 红色 |
| 2 | D- | 数据负 | 白色 |
| 3 | D+ | 数据正 | 绿色 |
| 4 | GND | 地线 | 黑色 |

::: tip 怎么记
VCC/红、D-/白、D+/绿、GND/黑 —— 这是标准配色。但市面上很多线不按标准配色，要靠万用表实测。
:::

## 差分信号

D+ 和 D- 不是独立的两根信号线，它们是**差分对**（Differential Pair）。差分信号的原理是：

- 发送端在 D+ 和 D- 上同时发送**极性相反的信号**
- 如果 D+ 输出高（3.3V），D- 就同时输出低（0V），反之亦然
- 接收端**只看 D+ 和 D- 之间的差值**

打比方：两个人并排走在街上，一个人喊"一"，另一个人同时喊"负一"。远处有人受到街上噪音干扰，但因为他同时听到了"一"和"负一"，把两者一减，噪声（同时加在两人身上的）就被消掉了。

**差分信号的优势：抗干扰能力强。** 外部电磁噪声会同时耦合到两根线上（共模噪声），但接收端只关心差值，噪声被自然抵消。

```c
// 差分信号原理（简化模拟）
发送端:
  D+ = 1    // 3.3V
  D- = 0    // 0V

噪声耦合（例如电机干扰在两根线上都叠加了 0.5V）:
  D+ = 1.5  // 3.3 + 0.5
  D- = 0.5  // 0   + 0.5

接收端做差:
  结果 = (D+) - (D-) = 1.5 - 0.5 = 1  // 噪声被消掉了！
```

## USB 枚举过程

把 USB 设备插到电脑上时，并不是插上就能用的。中间有一个"自我介绍"的过程，叫做**枚举**（Enumeration）：

1. **检测插入**：主机检测到 D+ 或 D- 被上拉电阻拉高
2. **识别速度**：D+ 上拉 = 全速设备（12Mbps），D- 上拉 = 低速设备（1.5Mbps）
3. **获取描述符**：主机问"你是谁？"，设备回答 VID（厂商标识）/PID（产品标识）
4. **加载驱动**：主机根据 VID/PID 找到对应驱动程序
5. **配置完成**：设备被分配地址，可以开始通信

打比方：USB 枚举就像你去住酒店。插 USB = 你走进大堂，主机（前台）识别你来了，问你要身份证（VID/PID），查系统（驱动库），给你房卡（地址），然后你才能入住（通信）。

## USB 供电能力

::: danger 重要限制
USB 2.0 标准最大仅提供 **5V / 500mA**（2.5W）。不要指望用电脑的 USB 口直接驱动电机、舵机等大电流负载——轻则供电不足设备不工作，重则烧掉电脑 USB 口的保险（甚至烧主板）。
:::

不同 USB 标准的供电能力：

| 标准 | 最大电压 | 最大电流 | 最大功率 |
|------|----------|----------|----------|
| USB 2.0 | 5V | 500mA | 2.5W |
| USB 3.0 | 5V | 900mA | 4.5W |
| USB-C (PD) | 最高 20V | 最高 5A | 100W |

## 竞赛中的 USB

在你的控制题中，USB 通常有两个用途：

1. **调试串口**：STM32 的 USB 模拟成虚拟串口（VCP），用于 printf 打印调试信息
2. **供电**：开发阶段从电脑取电，但正式比赛时必须用独立电池/电源

::: warning 血的教训
调试电机驱动板时，**务必使用 USB 隔离器**（详见隔离章节）。电机产生的反向电动势可以高达几十伏，能瞬间击穿电脑 USB 口。一块几块钱的隔离器不买，代价可能是几千块的维修费。
:::

# PCB 布局布线基础

## 数字与模拟：水火不容的邻居

电路板上大致有两类信号，它们的"性格"截然相反：

- **数字部分**（MCU、逻辑芯片、PWM 输出）：信号是方波，电压在 0 和 3.3V 之间猛跳。每次跳变都产生高频噪声。数字电路是**噪声制造者**。
- **模拟部分**（ADC 输入、运放、传感器信号）：信号是连续变化的微小电压，可能只有几 mV。任何外来噪声都会污染测量结果。模拟电路是**噪声受害者**。

打比方：数字部分是摇滚乐队排练室，模拟部分是图书馆自习室。两者放在一起，自习室的人根本看不进书。

### 分区与单点接地

布局的首要原则：**数字区和模拟区分开放**。

- 板子左边放数字电路，右边放模拟电路（或上/下分区）
- 两区之间的信号走线尽量减少
- **单点接地**：数字地和模拟地各自铺铜，只在**一个点**（通常是电源入口附近）汇合。这样数字区的地电流噪声不会流经模拟区的地平面

```c
// 布局错误 vs 正确的类比
// 错误：数字地电流 → 流经 → 模拟地 → 回到电源
//        ↓
//       模拟地平面被数字噪声"污染"
//       ADC 读数在 0~4095 之间随机跳动（但其实输入没变）

// 正确：数字地 → 直接回电源
//       模拟地 → 直接回电源（另一条路）
//       两路在电源入口汇合 → 各走各的，互不干扰
```

## 丝印（Silkscreen）

丝印是 PCB 上那层**白色文字**。它不是电路的一部分，是给人看的标注。

丝印应该标注什么：

- 元件位号（R1、C3、U2），方便对照 BOM 表焊接
- 连接器功能（"电机 A"、"UART TX"、"5V IN"）
- 跳线说明（"短接 = 使能"）
- 测试点标记
- 引脚 1 标记（小圆点或缺口），防止芯片焊反
- 板子名称、版本号、日期

::: tip 好习惯
设计时多花 5 分钟写清楚丝印，焊接时能省你半小时的猜谜时间。三个月后回看旧板子，你会感谢当时写了备注的自己。
:::

## 元件封装：从易到难

**封装**（Package）是元件在 PCB 上的"脚印"——焊盘大小、间距、形状。不同封装对手工焊接的友好程度天差地别：

| 封装类型 | 全称 | 引脚位置 | 焊接难度 | 适合方式 |
|----------|------|----------|----------|----------|
| SOP-8 | Small Outline Package | 两侧引出 | 容易 | 烙铁手焊 |
| LQFP-48 | Low-profile Quad Flat Package | 四侧引出 | 中等 | 烙铁拖焊 |
| QFN-32 | Quad Flat No-leads | 底部焊盘 | 较难 | 热风枪 / 加热台 |
| BGA | Ball Grid Array | 底部锡球阵列 | 极难 | 工厂回流焊 |

### SOP（最容易）

引脚在两侧，间距较大（通常 1.27mm）。标准烙铁就能轻松焊接。竞赛中尽量选 SOP 封装的芯片。

### LQFP（需要技巧）

引脚在四侧，密集（间距 0.5mm 或 0.8mm）。需要用**拖焊**技术：固定芯片后，用烙铁拖过一排引脚，靠助焊剂让锡自动分离。多练习几次就能掌握。

### QFN（需要热风枪）

底部焊盘，手工烙铁无法触及。必须用热风枪或加热台从下方加热。中心通常有一个大散热焊盘（Thermal Pad），必须焊好，否则芯片不工作或过热。

### BGA（初学者禁止）

底部全是锡球阵列，焊接时完全看不到焊盘。需要专业回流焊设备 + X 光检测。**竞赛场景不推荐使用。**

## 焊接基础操作

::: tip 焊接核心口诀
**350°C、助焊剂、先固定一角、拖焊、清洗**
:::

1. **温度**：烙铁调到 330~350°C。太高烧坏焊盘，太低锡不流动
2. **助焊剂**（Flux）：焊接的灵魂。涂在焊盘上，锡自动流到该去的地方
3. **固定**：先给芯片的一个角焊盘上锡，把芯片放上去，固定这个角。确保对齐后焊对角
4. **拖焊**：烙铁轻轻拖过一排引脚，锡跟着烙铁走，自动分离各脚
5. **检查**：用放大镜或手机微距看有没有连锡（短路）
6. **清洗**：用 PCB 洗板水或无水酒精（>95%）刷掉残留助焊剂

::: danger 新手常见错误
- 不用助焊剂 → 锡不流动 → 各脚黏在一起
- 烙铁温度不够 → 烙铁头黏在引脚上超过 3 秒 → 铜箔脱落
- 焊接后不清洗 → 助焊剂残留吸湿 → 半年后电路漏电
:::

# 高速信号与信号完整性

## 什么时候算"高速"？

很多初学者认为："我的 MCU 只跑 72MHz，不算高速吧？" 这个判断标准错了。

判断是否需要关注信号完整性的标准是**信号的上升时间**（从低电平到高电平的过渡时间），而不是时钟频率。现代 MCU 的 GPIO 上升时间可达 **1~3 纳秒（ns）**，这样的陡峭边沿包含大量高频分量。

::: tip 经验法则
信号上升时间 < 5ns，或信号频率 > 50MHz 时，就必须开始考虑高速信号问题了。
:::

## 阻抗匹配：别让信号"撞墙"

**阻抗**可以理解为走线对信号流动的"阻力". 当信号沿 PCB 走线传播时，走线有特征阻抗（通常 50Ω 或 90Ω）。

如果走线阻抗 ≠ 源端输出阻抗 ≠ 负载端输入阻抗，信号就会发生**反射**——部分信号沿原路弹回，干扰后续信号。

打比方：你在一个走廊里对着墙大喊，声音会弹回来（回声）。如果走廊前方是吸音材料（阻抗匹配），声音被完全吸收，没有回声。PCB 走线也是同理——阻抗不匹配，信号就在线上来回弹，导致接收端看到错误的电平。

反射的后果：
- 接收端看到错误的高低电平
- 信号过冲（overshoot）和振铃（ringing）——电压短暂超出正常范围
- 通信出错、数据丢包

### 什么时候必须做阻抗匹配？

- USB 差分线（D+/D-）：**必须**匹配 90Ω 差分阻抗
- 高速 SPI / SDIO（>25MHz）：推荐匹配
- 射频信号（天线）：**必须**匹配 50Ω
- 普通 GPIO、I2C、UART 等低速信号：不需要

## 蛇形走线（Serpentine Traces）

你可能会在一些 PCB 上看到走线故意走成锯齿形（像蛇一样弯曲），这不是设计失误。

**目的：使多根并行信号线的长度相等。**

例如：并行 LCD 接口有 16 根数据线。如果其中一根走了 5cm、另一根走了 8cm，信号到达的时间就不一样（"时滞"或 skew），接收端在采样时可能采到错误数据。

```c
// 蛇形走线的用途（类比）
// 16 根数据线同时出发
// 直的那根：→→→→→→→→  8cm，到达时间 t=0.4ns
// 弯的那根：↗↘↗↘↗↘↗↘  12cm，到达时间 t=0.6ns
// → 时间差 0.2ns → 接收端采样时，弯的那根数据还没到！

// 加蛇形走线后：
// 直的那根：→→→↗↘↗↘  12cm（绕了路）
// 弯的那根：↗↘↗↘↗↘↗↘  12cm
// → 长度相等 → 同时到达 → 数据正确
```

打比方：马拉松比赛中，所有选手都要跑 42.195 公里。不能因为某条路是直的就让一些选手少跑。蛇形走线就是让直路上的信号也绕一绕路，保证所有人跑的距离一样。

## EMI（电磁干扰）

**EMI**（Electromagnetic Interference，电磁干扰）：快速变化的电流会在周围空间产生电磁波，干扰附近的电路或设备。

### 电机噪声 —— 头号敌人

在控制竞赛中，**电机是最大的噪声源**。电机转动时，电刷（有刷电机）或 PWM 开关（无刷电调）会产生强烈的电磁脉冲。

电机启动瞬间，你可能观察到：
- OLED 屏幕花屏
- 串口打印乱码
- ADC 读数突变
- MCU 复位

这些都是 EMI 在作祟。

### 应对措施

| 措施 | 原理 | 实施方法 |
|------|------|----------|
| 分离电源 | 电机噪声不进入 MCU 电源 | 电机用独立电池或独立 DC-DC |
| 续流二极管 | 吸收电机线圈关断时的反压尖峰 | 在电机驱动输出端并联肖特基二极管（如 SS34） |
| 信号远离 | 减少噪声耦合 | 信号线距离电机线 > 5cm，切忌平行走线 |
| 双绞线 | 抵消辐射 | 电机供电线互相绞在一起 |
| 加屏蔽 | 阻挡辐射 | 关键电路加金属屏蔽壳 |

```c
// 最简单的电机抗干扰电路
// 在电机两端焊接一个 104 (100nF) 陶瓷电容
// 滤掉电刷火花产生的高频噪声
//
//   电机 + ──┬── + 驱动输出
//            │
//           ─── 104 电容
//           ───
//            │
//   电机 - ──┴── - 驱动输出
```

::: danger 重大原则
**电机供电线和信号线永远不要绑在一起走！** 两者的距离越大越好。如果必须在同一接插件上，中间加地线隔开。
:::

## 接地与铺铜

对于两层板，标准的做法是：
- **顶层**：信号走线 + 局部铺地
- **底层**：完整地平面（大面积铺铜）
- 在空旷区域打**过孔**（Via），把顶层地和底层地连接起来，形成法拉第笼效应

完整的地平面为信号提供低阻抗的回流路径，这是抑制 EMI 最有效的手段之一。

# 常用连接器

## 选连接器 = 选"插头插座"

连接器（Connector）决定了你的信号和电流能不能可靠地从板子传到外设。选错连接器 = 接触不良、电流不够烧毁、或者干脆插不进去。以下是竞赛中最常见的几种。

---

## 2.54mm 排针 / 排母（Dupont 连接器）

**最常见、最通用、最便宜。**

- **间距**：2.54mm（0.1 英寸）
- **额定电流**：约 1~3A（视镀层和线径）
- **典型用途**：信号连接、模块对接、面包板跳线

排针（公头）是一排金属针，排母（母座）是带孔的插座。两者配合形成插拔连接。Dupont 端子可以手工压接（有专用压线钳），也可以买成品杜邦线。

**优点：** 通用性强，所有开发板都有 2.54mm 排针接口，面包板完美兼容。

**缺点：** 体积较大，防反插能力弱（插反只靠习惯），高频信号质量差。

---

## 螺丝端子（Screw Terminal）

**给电源接线用的。**

- **间距**：常见 3.5mm、5.0mm、5.08mm
- **额定电流**：5A ~ 15A（看规格）
- **典型用途**：电源输入、电机输出、大电流连接

用螺丝刀拧紧螺丝，夹住导线。连接非常牢固，能通过大电流，而且徒手就能接线——不需要焊接或压接工具。

::: tip 适用场景
电池到主板的电源线、电机驱动输出线——凡是电流 > 2A 的地方，优先用螺丝端子而非排针。
:::

---

## XT30 / XT60

**大电流电源连接器，源自航模 / 无人机领域。**

| 型号 | 额定电流 | 典型用途 |
|------|----------|----------|
| XT30 | 30A | 小功率电机、电池组 |
| XT60 | 60A | 大功率驱动、比赛机器人 |

黄色塑料外壳，香蕉头镀金。插拔可靠，接触电阻极低。竞赛中用锂聚合物电池（LiPo，如 3S 11.1V）时，电池端几乎标配 XT60 或 XT30。

**优点：** 大电流、防反插（物理防呆）、防松脱。

**缺点：** 体积大、需要焊接、只能一对一连接（不带分支）。

---

## PH2.0 / GH1.25

**微型连接器，给传感器和小模块用的。**

| 型号 | 间距 | 典型用途 |
|------|------|----------|
| PH2.0 | 2.0mm | 常见传感器模块、小电机 |
| GH1.25 | 1.25mm | 紧凑空间、微型模块 |

这些连接器比 Dupont 小得多，适合空间紧张的设计。很多成品传感器（如超声波模块、红外对管）都预装了 PH2.0 或 XH2.54 接口。

::: warning 注意间距
PH2.0（2.0mm 间距）和 XH2.54（2.54mm 间距）外观相似，初学者极易买错。买之前用尺子量一下旧模块的引脚间距，或在规格书上确认。
:::

---

## 连接器选型总表

| 连接器 | 间距 | 电流 | 插拔次数 | 防反插 | 主要用途 |
|--------|------|------|----------|--------|----------|
| Dupont 排针 | 2.54mm | 1~3A | 数百次 | 无 | 信号、模块 |
| 螺丝端子 | 5.0mm | 5~15A | 数十次 | 无 | 电源、电机 |
| XT60 | - | 60A | 数千次 | 有 | 电池、大功率 |
| XT30 | - | 30A | 数千次 | 有 | 电池、中功率 |
| PH2.0 | 2.0mm | <1A | 数十次 | 有 | 传感器、小模块 |
| GH1.25 | 1.25mm | <1A | 数十次 | 有 | 微型模块 |

---

## 选型原则

::: tip 一句话指南
**信号选排针，电源选端子，电池选 XT60/XT30，传感器看间距。**
:::

- 电流 < 1A 且需要频繁插拔 → Dupont 排针
- 电流 > 2A 或需要牢固 → 螺丝端子
- 电池供电的移动设备 → XT30/XT60
- 空间有限的微型设备 → PH2.0 / GH1.25
- 需要防反插 → 选用带防呆槽的连接器

# 隔离的作用与目的

## 为什么要隔离？

在嵌入式开发中，你经常用 USB 线把开发板连到电脑上调程序。大部分时候一切正常——直到你开始调试**电机驱动**。

::: danger 这不是吓唬你
电机在关断瞬间会产生**反向电动势**（Back-EMF），电压可以瞬间飙到几十甚至上百伏。如果这个高压通过地线、信号线或电源线倒灌回电脑 USB 口，可能直接把 USB 控制器甚至是主板烧掉。修一次几百到几千块不等。
:::

打比方：隔离就像医院的**隔离病房**。传染病人（电机噪声/高压）和健康人（电脑）之间有一道物理屏障，病菌（高压）过不去，但医生还能透过窗户（隔离信号）看到病人情况。

---

## 电气隔离 vs 非隔离

电气隔离（Galvanic Isolation）是指在两个电路之间**没有任何直接的导电路径**，信号和电源都通过磁场（变压器）、电场（电容耦合）或光（光电耦合器）传递。

```c
// 非隔离连接（危险！）
电脑 USB ─────── 导线 ─────── STM32 开发板
                                  │
                                  ├── GPIO ── 电机驱动 ── 电机
                                  │
        // 电机反压通过 GPIO → MCU → USB → 电脑！
        // 电流路径存在 → 高压找到回家的路

// 隔离连接（安全）
电脑 USB ── 隔离芯片 ─┊┊┊┊ 隔离屏障 ┊┊┊┊── STM32 开发板
                       （磁场耦合）            │
                                         电机驱动 ── 电机
        // 屏障两侧没有导线连接 → 高压无路可走
```

---

## USB 隔离器

竞赛调试时**最重要的保护设备**，没有之一。

一个 USB 隔离器内部包含一块**隔离芯片**（常见型号 **ADuM3160**），通过芯片内部的微型变压器传递 USB 信号，而电源通过隔离型 DC-DC 模块独立提供。物理上，输入和输出之间没有任何铜导线连接。

市面上成品 USB 隔离器几十元到上百元不等。如果你经常调试电机驱动板，这个投资极为划算。

**什么时候必须用 USB 隔离器：**

- 调试任何涉及电机的电路板
- 调试高压电路（例如电池串联到 24V+）
- 电路板从外部大功率电源取电
- 调试自己画的 PCB（第一次通电，可能有短路或设计错误）

::: tip 经验
养成习惯：调试新板子时，第一件事不是插 USB，而是先用万用表量一下 3.3V 和 GND 是否短路。确认无误后，再用 USB 隔离器连接。等板子完全调通，确认无高压风险后，才考虑直连。
:::

---

## 其他隔离方式

### 光电耦合器（Optocoupler）

**原理**：电信号 → LED 发光 → 光敏三极管接收 → 恢复成电信号。输入输出之间只有光路，没有电路。

**典型型号**：PC817（低速）、6N137（高速）。用于数字信号隔离，如隔离 UART、隔离 GPIO。

**优点**：便宜（几毛钱）、单向、耐压高（数千伏）。

**缺点**：速度慢（普通光耦只能到几十 kHz）、需要限流电阻。

### 隔离型 DC-DC 模块

普通 DC-DC 输入输出的地在内部是通的（非隔离）。隔离型 DC-DC（如 B0505S-1W）输入和输出的地完全分离，各自独立。

**用途**：为隔离后的电路单独供电。例如隔离型 CAN 总线、隔离型 RS-485 电路中，总线侧的芯片需要用隔离电源供电。

### 继电器

继电器也是隔离元件——控制线圈和触点之间没有电气连接。用低压控制高压（如 3.3V 控制 220V 交流）时，继电器提供物理隔离。

---

## 隔离的应用场景总结

| 场景 | 隔离方式 | 隔离等级 |
|------|----------|----------|
| 调试电机板子 | USB 隔离器 (ADuM3160) | 几千伏 |
| 数字信号传递 | 光耦 (PC817/6N137) | 几千伏 |
| 隔离电源 | 隔离 DC-DC (B0505S) | 1000~3000V |
| 高压通断 | 继电器 | 几千伏 |
| CAN/RS-485 总线 | 隔离收发器 | 几千伏 |
| 高压测量 | 隔离运放 / 霍尔传感器 | 几千伏 |

---

::: warning 竞赛特别提醒
比赛现场电脑是**你自己的**，烧了没人赔。所以：

1. 所有电机驱动相关调试必须经过 USB 隔离器
2. 备一个备用 USB 隔离器（东西不贵，关键时刻救命）
3. MCU 和电机之间尽量用光耦隔离关键信号
4. 电机独立供电，地和 MCU 地必要时才单点连接
:::

## 小结

隔离的核心目的：保护你的电脑。电机转动时会产生反向电动势和电磁干扰，通过 USB 线传回电脑可能烧毁 USB 口甚至主板。隔离方案的选择：低速信号用光耦，高速信号用磁耦或隔离芯片，预算紧张时至少给调试口加一个 USB 隔离器。
---

# 焊接与调试

画完 PCB、打完板子，你收到一包绿色（或蓝色、黑色）的电路板和一堆元件。接下来就是把它们焊上去——这一步让无数新手崩溃，但其实掌握方法后并不可怕。

## 工具清单

| 工具 | 必要程度 | 预算 | 说明 |
|------|---------|------|------|
| **温控电烙铁**（T12/936 焊台） | 必需 | ¥50~150 | 别用 ¥9.9 的外热式烙铁，温度不准 |
| 焊锡丝（0.5~0.8mm，含助焊剂） | 必需 | ¥10/卷 | 选含铅 63/37 的，流动性好、熔点低 |
| 助焊剂（松香或免洗助焊膏） | 强烈推荐 | ¥5 | 让焊锡"听话"地待在焊盘上 |
| 吸锡带/吸锡器 | 推荐 | ¥5~10 | 焊错了吸掉重来 |
| 镊子（尖头，防静电） | 必需 | ¥10 | 夹贴片元件用 |
| 万用表 | 必需 | ¥30~100 | 测短路、测电压、测通断 |
| 放大镜/手机微距镜头 | 推荐 | ¥0~20 | 检查 LQFP 密脚焊接 |

## 手工焊接技巧

### 直插元件（DIP）

这是最简单的——引脚穿过 PCB 孔，翻过来焊：

```
 PCB 正面                  PCB 背面（焊面）
  ┌──┐                     焊锡
  │电│  ← 元件本体          ⊂⊃
  │阻│                     ⊂⊃  ← 焊点（像小山包，光滑闪亮）
  └──┘                      │
  ├─┤ ← 引脚穿过            ├─┤ ← 剪掉多余引脚
```

1. 烙铁温度设 **320°C~350°C**（含铅锡）
2. 烙铁头同时接触焊盘和引脚，等 1~2 秒
3. 送焊锡丝到焊盘 + 引脚的交界处（不是送到烙铁头上！）
4. 焊锡熔化流满焊盘后，先撤锡丝，再撤烙铁
5. 焊点应该**光滑、锥形、有光泽**。哑光灰色 = 冷焊（烙铁温度不够或移开了）→ 重新加热融化

### 贴片元件（SMD）

贴片元件没有引脚穿过 PCB，而是直接坐在焊盘上：

**贴片电阻/电容（0805 / 0603 封装）：**

1. 在一个焊盘上先焊一小坨锡
2. 镊子夹住元件放到位，烙铁融化那坨锡 → 固定住一边
3. 再焊另一边
4. 回头把第一边修整一下

**LQFP 芯片（多脚密间距，如 STM32）：**

1. 先对齐芯片（1 脚标记和 PCB 丝印一致）
2. 对角焊两个引脚固定芯片
3. 四边都涂上助焊剂
4. **拖焊法**：烙铁头蘸少量焊锡，沿引脚方向缓缓拖动，焊锡被助焊剂引导只留在焊盘+引脚上
5. 短路了？涂助焊剂 + 吸锡带一拖就开

::: tip 新手最容易犯的错
焊锡直接送烙铁头上 → 助焊剂瞬间挥发 → 焊锡不流动 → 焊出"刺猬球"。**烙铁头的作用是加热焊盘和引脚，不是融化焊锡。**
:::

## 上电前检查清单

焊完别急着插电！先走一遍这个检查清单，能避免 90% 的"冒烟惨案"：

| # | 检查项 | 工具 | 异常处理 |
|---|--------|------|---------|
| 1 | 电源到地短路？（蜂鸣档） | 万用表 | 响 = 短路 → 查焊点、查元件方向 |
| 2 | 电解电容极性是否反了？ | 目视 | 反了 = 通电必炸 |
| 3 | IC 方向是否正确？（1 脚标记） | 目视 | 查丝印→查芯片 datasheet |
| 4 | 二极管方向是否正确？ | 目视 | 阴极端（有标记）朝电路高压侧 |
| 5 | 焊点是否有明显桥连（短路）？ | 放大镜 | 密脚芯片区域重点检查 |
| 6 | 是否有虚焊？（焊点灰暗、不光滑） | 目视 | 重新加热 |
| 7 | 未焊接的焊盘是否干净？ | 目视 | 锡珠溅到不该在的地方 = 清理 |

## 第一次上电

1. **用可调电源，不用电池！** 设电压 3.3V 或 5V，限流 100mA
2. 看电流——大于预期？立刻断电！
3. 手背摸每个芯片——烫手？断电！
4. 闻——有焦味？断电！
5. 以上都正常，万用表测 3.3V / 5V / Vmotor 测试点电压是否正常
6. 全部正常 → 可以接调试器烧程序

::: danger 新板子上电须知
**新板子第一次上电决不能用电池直连。** 可调电源有限流保护，电池没有——短路就烧了。
:::

## 常见问题排查

| 现象 | 可能原因 | 怎么查 |
|------|---------|--------|
| 完全没反应 | 电源短路 / 开路 | 测 3.3V 有没有 |
| MCU 连不上调试器 | SWD 引脚被占用、芯片方向反了、复位电路异常 | 查 SWD 接线、测 NRST 引脚电平 |
| 跑一会儿就死机 | 供电不稳、晶振不起振、看门狗没喂 | 示波器看电源纹波 |
| 某个 GPIO 不工作 | 虚焊、引脚和相邻引脚短路 | 放大镜检查 |
| 电机不转 | 驱动芯片没供电、EN 脚没使能、PWM 频率不对 | 万用表测驱动芯片 VMOT、EN 引脚电压 |

## 小结

焊接是电赛选手的基本功。温控烙铁 + 好焊锡 + 助焊剂 + 耐心，四样齐全就能焊好。上电前用万用表测一遍电源到地的短路，这个习惯能让你少烧十块板子。

# 车体方案选择

## 为什么车体很重要

想象你要送一个外卖：骑自行车、开小轿车、还是开卡车？不同的车适合不同的路况。竞赛小车的车体方案选择也是同样的道理——**车体决定了你能完成什么任务**。

## 三种主流方案

### 1. 两轮驱动（2WD）

```
      前轮（万向轮/从动轮）
      ○
┌─────────────────┐
│                 │
│    车体平台      │
│                 │
└─────────────────┘
   ⬤           ⬤
   左电机       右电机
```

两轮驱动就像超市的购物车——两个主动轮各自独立转动，前方一个万向轮保持平衡。

**优点：**
- 结构简单，零件少，不容易坏
- 转弯靠"差速"：左轮快、右轮慢，车就右转
- 成本低，两轮驱动模块约30元
- 编程简单：只需控制两个电机的转速差

**缺点：**
- 爬坡能力一般
- 光滑地面上可能打滑
- 不能原地旋转

**适用场景：** 平坦地面、网格导航。竞赛中约 **90% 的任务**都可以用 2WD 完成。

### 2. 四轮驱动（4WD）

四个轮子都有动力——像越野车。抓地力更强，但转弯时需要四个轮子协调转速，否则会出现"轮胎打滑"的刺耳声音。

**优点：** 爬坡强、过障碍物强  
**缺点：** 结构复杂、转弯可能打滑、耗电更快、代码更复杂

### 3. 麦克纳姆轮 / 全向轮

这是一种"神奇"的轮子——轮毂上装了一圈斜着的小滚轮。它可以横着走，就像螃蟹一样。

**优点：** 全方位移动，可以原地旋转  
**缺点：** 价格贵（一套约200-400元）、对地面平整度要求极高、控制复杂（需4个电机同时协调）、耗电快

## 方案对比表

| 特性 | 2WD | 4WD | 麦克纳姆轮 |
|------|-----|-----|-----------|
| 价格 | ~30元 | ~80元 | ~200-400元 |
| 转弯方式 | 差速转向 | 差速转向 | 全向移动 |
| 编程难度 | 低 | 中 | 高 |
| 爬坡能力 | 一般 | 强 | 弱（易打滑） |
| 平地表现 | 优秀 | 良好 | 优秀 |
| 适合新手 | ✅ 是 | ⚠️ 一般 | ❌ 不推荐 |

## 如何选择

::: tip 新手建议
**第一次参赛，请选 2WD。** 理由很简单：简单的东西更容易跑起来。你不需要在赛前花三天调试一个复杂的底盘——先把车跑起来，完成任务，比什么都重要。
:::

选择车体时问自己三个问题：

1. **任务需要什么？** 看往年赛题：车需要上坡吗？需要横移吗？如果不需要，2WD 就是最优解。
2. **地面是什么材质？** PVC 赛道、木地板 → 2WD 没问题。沙地、草皮 → 考虑 4WD。
3. **你的时间够吗？** 复杂的底盘需要更复杂的控制代码。在竞赛中，**时间是最稀缺的资源**。

## 常见误区

::: danger 别掉进"堆料"陷阱
很多新手觉得"配置越高越好"——麦克纳姆轮看起来很酷，就想上。但复杂的硬件需要复杂的控制算法（运动学逆解、PID 调参），当你在比赛现场调试到凌晨三点时，你会后悔为什么当初不选简单的 2WD。
:::

## 实际推荐配置

- **车架：** 亚克力切割板（淘宝定制，约15元）或 3D 打印
- **电机：** TT 减速电机 ×2（带 1:48 或 1:120 减速比）
- **轮子：** 直径 65mm 橡胶轮（摩擦力好）
- **前轮：** 万向球头轮（几块钱一个）

这套配置总成本不超过 60 元，却能完成绝大部分竞赛任务。

---

## 小结

车体方案的选择没有标准答案——三轮结构简单灵活但高速容易翻，四轮稳但转弯半径大，履带越野强但速度慢。选型的关键是理解你的赛题对机动性、稳定性和速度的权重。

# 电机扭矩估算

## 什么是扭矩

扭矩，通俗地说就是"旋转的力气"。你拧瓶盖时需要用力——这个"拧的力气"就是扭矩。电机扭矩决定了你的小车能不能从静止开始跑起来、能不能爬上坡。

**扭矩的单位**是 N·m（牛·米）或 mN·m（毫牛·米）。竞赛小车用的电机，扭矩通常在 50-300 mN·m 这个范围。

## 你需要这些数据

在估算扭矩之前，先收集（或测量）以下数据：

| 参数 | 含义 | 怎么得到 |
|------|------|---------|
| 车重 m | 整车重量（含电池） | 用厨房秤称一下，单位 kg |
| 轮子半径 r | 驱动轮半径 | 用尺子量，单位 m |
| 期望加速度 a | 要多快加速 | 新手取 0.5 m/s²（平路够用） |
| 摩擦系数 μ | 轮子与地面的摩擦 | 橡胶轮+PVC ≈ 0.6 |
| 减速比 G | 电机自带减速箱 | 看电机型号（如 1:48） |
| 爬坡角度 θ | 最大坡度 | 竞赛通常 < 10°，取 0° 估算 |

## 估算公式

最基本的需求扭矩：

```
所需扭矩 = (车重 × 重力加速度 × 摩擦系数 × 轮子半径) / 减速比
```

用数学写出来：

```
T_need = (m × g × μ × r) / G
```

其中 g = 9.8 m/s²（重力加速度）。

### 算一个实际例子

假设你的小车：

- 车重：0.5 kg（500克，含电池和所有模块）
- 轮子半径：0.0325 m（直径 6.5cm ÷ 2）
- 摩擦系数：0.6
- 减速比：1:48

```
T_need = (0.5 × 9.8 × 0.6 × 0.0325) / 48
       = 0.09555 / 48
       = 0.00199 N·m
       ≈ 2 mN·m
```

::: tip 理解这个数字
2 mN·m 非常小！即使是玩具 TT 电机，堵转扭矩通常也有 100-300 mN·m。所以结论是：**竞赛小车用的常见电机，扭矩通常是够的。**
:::

## 实际中的扭矩估算

### 常见电机参考扭矩

| 电机型号 | 额定扭矩 | 堵转扭矩 | 适用场景 |
|---------|---------|---------|---------|
| TT 电机 (1:48) | ~50 mN·m | ~300 mN·m | 0.3-0.8 kg 小车 |
| N20 减速电机 (1:30) | ~30 mN·m | ~150 mN·m | 0.2-0.5 kg 小车 |
| N20 减速电机 (1:100) | ~80 mN·m | ~400 mN·m | 0.5-1.0 kg 小车 |
| 370 电机 (1:50) | ~100 mN·m | ~600 mN·m | 大车/重车 |

### 为什么你的小车一般不需要担心扭矩

竞赛小车有几个特点让"扭矩不够"很少成为问题：

1. **车很轻。** 即使加上电池和所有传感器，大部分竞赛小车不到 1 kg。
2. **用减速电机。** 减速箱把转速换成扭矩（减速比越大，扭矩越大，但速度越慢）。
3. **地面平坦。** 竞赛赛道通常是平整的 PVC 或木地板。

## 如果车真的跑不动怎么办

如果通电后车轮不转（或者一转就卡住），按顺序排查：

1. **减小车轮与车架的摩擦。** 检查轮子是否蹭到了车架——这是最常见的原因。
2. **增大减速比。** 把 1:48 换成 1:120 → 扭矩翻 2.5 倍，但速度降 2.5 倍。
3. **减轻重量。** 用更轻的电池、更小的电路板。
4. **换大扭矩电机。** 但注意大扭矩电机通常更重、更耗电。

::: warning 扭矩和速度的取舍
扭矩和速度是反比关系——减速比增大，扭矩大但速度慢。你能找到的最佳平衡点是：**在竞赛允许的最大时间内跑完全程的最慢速度**——这样需要的扭矩最小。
:::

## 实际测试方法

纸上算不如动手测。最简单的测试：

```c
// 测试车能不能起步
void torque_test() {
    // 从 20% 占空比开始，逐步增加
    for (int duty = 20; duty <= 100; duty += 10) {
        motor_forward(duty);  // 设置占空比
        delay_ms(500);        // 等半秒
        // 观察：车轮动了吗？车动了吗？
        motor_stop();
        delay_ms(500);
    }
}
```

把车放在**实际的竞赛地面上**（不是桌面上！），从小到大增加 PWM 占空比。记录车轮开始转动时的最小占空比——这就是你的"起步阈值"。如果 50% 以上才能动，说明扭矩不足，需要排查。

---

## 小结

扭矩估算只需要称一下车重、量一下轮子半径，代入公式就能算出需求扭矩。竞赛小车用常见的减速电机（TT电机、N20），扭矩通常是够的——车跑不动时先查轮子是否蹭车架，再考虑换电机。

# 传感器与主控选型

## 一个基本原则

这可能是新手最难接受的一句话。我们天然喜欢"更高配"——像素更高的摄像头、精度更高的传感器、算力更强的芯片。但在嵌入式系统中，更高的配置意味着：

- 更多的数据需要传输和处理
- 更多的 CPU 时间被占用
- 更多的内存被消耗
- 更复杂的代码

> **够用就好，不要过度设计。**

## 精度与资源的权衡

想象你去买菜：你可以用一杆秤（简单、够用），也可以用实验室的分析天平（精确到 0.0001 克）。但用天平买菜是浪费——你需要的是一个"刚好够用"的工具。

| 传感器类型 | 数据量 | 占用资源 | 精度 | 够用吗 |
|-----------|--------|---------|------|-------|
| 灰度传感器 ×8 | 8 个 GPIO 电平 | 几乎为 0 | 中（区分黑白灰） | ✅ 巡线足够 |
| 红外测距 | 1 个 ADC | 极低 | 低 | ✅ 避障足够 |
| 超声波 | 1 个 GPIO（脉冲） | 低 | 中 | ✅ 测距足够 |
| 摄像头（UART 输出坐标） | 几十字节/帧 | 中（一个 UART） | 中 | ✅ 视觉任务 |
| 摄像头（完整图像） | 几 MB/秒 | 极高（专用总线） | 高 | ⚠️ 通常用不到 |

::: tip 灰度 vs 摄像头的例子
8 路灰度传感器只需要 8 个 GPIO 引脚，读一次几十微秒。而一个摄像头通过 UART 持续发送数据流，需要占用一个完整的串口并持续处理。对于"车子沿黑线走"这个任务，灰度传感器完全够用——不需要摄像头。
:::

## 主控芯片（MCU）选型

MCU（Microcontroller，微控制器）就是小车的大脑。选 MCU 看这几个指标：

### 关键参数

| 参数 | 是什么意思 | 怎么判断够不够 |
|------|-----------|--------------|
| GPIO 数量 | 可用的输入输出引脚 | 数一数你需要接几个传感器、几个电机 |
| Flash | 程序存储空间 | 简单项目 32KB，中等 128KB，复杂 256KB+ |
| RAM（SRAM） | 运行内存 | 程序运行时变量占的空间 |
| 主频 | 运行速度 | 单位 MHz，越高越快 |
| 外设接口 | UART、SPI、I2C 等 | 每接一个模块可能就需要一个接口 |
| 价格 | 芯片单价 | 初学选 10-30 元的足够 |

### 两款推荐 MCU 对比

| 特性 | MSPM0G3507 | STM32F103VET6 |
|------|-----------|---------------|
| 内核 | ARM Cortex-M0+ | ARM Cortex-M3 |
| 主频 | 80 MHz | 72 MHz |
| Flash | 128 KB | 512 KB |
| RAM | 32 KB | 64 KB |
| UART | 4 个 | 5 个 |
| SPI | 2 个 | 3 个 |
| I2C | 2 个 | 2 个 |
| ADC | 12 位 | 12 位 |
| 价格 | ~15 元 | ~25 元 |
| 适合项目 | 中等复杂度 | 较大型项目 |

::: tip MSPM0G3507 够用吗？
对于典型的竞赛项目（巡线 + 编码器 + 陀螺仪 + 电机控制 + OLED），MSPM0G3507 的 128KB Flash 和 32KB RAM **绰绰有余**。只有当你需要做以下事情时，才考虑升级：
:::

### 什么时候该升级 MCU

- **大量浮点运算：** 比如 FOC（磁场定向控制）高级电机算法
- **神经网络推理：** 在 MCU 上跑 TinyML 模型
- **复杂显示 UI：** 带触摸屏的图形界面
- **多传感器融合：** 同时处理摄像头图像 + 雷达点云

## 传感器选型清单

对于竞赛来说，有一份"标配"传感器清单：

### 标配（必备）

| 传感器 | 用途 | 接口 | 价格 |
|--------|------|------|------|
| 8 路灰度 | 巡线 | 8 GPIO | ~15元 |
| 霍尔编码器 ×2 | 测速/里程 | 4 GPIO（定时器） | ~10元/个 |
| 陀螺仪 JY61P | 角度/转向 | UART ×1 | ~40元 |
| 超声波 HC-SR04 | 避障 | 1 GPIO + 1 定时器 | ~5元 |

### 进阶（按需选配）

| 传感器 | 用途 | 接口 | 价格 |
|--------|------|------|------|
| MaixCAM Pro | 视觉识别 | UART ×1 | ~120元 |
| 红外测距 GP2Y0A | 精确距离 | ADC ×1 | ~15元 |
| OLED 显示屏 | 调试信息 | I2C ×1 | ~8元 |
| 蓝牙模块 HC-05 | 无线调试 | UART ×1 | ~12元 |

### 规划你的接口资源

在选 MCU 和传感器时，做一个"接口预算表"：

```
需要使用的接口：
├── UART1: JY61P 陀螺仪
├── UART2: MaixCAM 视觉模块
├── UART3: 蓝牙无线调试
├── I2C1: OLED 显示屏
├── SPI1: （保留）
├── GPIO A0-A7: 灰度传感器
├── GPIO B0-B1: 电机方向控制
├── GPIO PWM: 电机速度控制（2路）
├── GPIO 定时器: 编码器 ×2（4路）
└── ADC: 电池电压检测
```

这样列出来，你就能看清楚：**选的那颗 MCU 接口够不够？**

## 经验法则

::: danger 常见错误
新手容易"贪多"——把所有传感器都装上去。结果：接口冲突、代码臃肿、功耗高、调试困难。**先确定任务需求，再选最少最简的传感器。**
:::

## 小结

选型从"需要感知什么"开始，不是从"什么最酷"开始。四条经验：先定任务再选传感器；每个传感器只解决一个问题；能用简单方案就别上摄像头；接口算清楚再下单。

# 分板还是合板？

## 问题是什么

当你的小车需要：主控芯片（MCU）、电源管理、电机驱动、传感器接口……这些电路是全部画在一块 PCB 上（合板），还是分开几块小板（分板）？

这就像租房——你可以租一个大开间（合板），也可以租两室一厅（分板）。各有各的道理。

## 方案一：合板（单板方案）

把所有电路放在一块 PCB 上。

```
┌──────────────────────────────────┐
│           单块 PCB               │
│                                  │
│  ┌────────┐  ┌──────┐  ┌──────┐ │
│  │  MCU   │  │ 电源  │  │ 电机  │ │
│  │ 主控   │  │ 管理  │  │ 驱动  │ │
│  └────────┘  └──────┘  └──────┘ │
│  ┌──────────────────────────┐    │
│  │     传感器接口区          │    │
│  └──────────────────────────┘    │
└──────────────────────────────────┘
```

| 优点 | 缺点 |
|------|------|
| 省空间，体积小 | 电机噪声通过 PCB 耦合到 MCU |
| 走线少，不需要排线 | 一个模块坏了 → 换整块板 |
| 看起来整洁 | 调试时相互干扰 |
| 批量成本低 | 修改困难（改一处重做全部） |

## 方案二：分板（多板方案）

每个功能块独立成板，用排线连接。

```
┌──────────┐   排线   ┌──────────┐   排线   ┌──────────┐
│  MCU 板  │←────────→│ 电源板   │←────────→│ 电机驱动板│
└──────────┘          └──────────┘          └──────────┘
          \                  / 
           └──────┬─────────┘
              ┌──────────┐
              │ 传感器板  │
              └──────────┘
```

| 优点 | 缺点 |
|------|------|
| 噪声隔离——电机噪声不会干扰 MCU | 体积大，占空间 |
| 一个板坏了只换一块 | 排线多，可能松动 |
| 独立调试——先调电源再调 MCU | 成本稍高（多块 PCB） |
| 方便迭代修改 | 排线接错会烧板子 |

## 为什么电机是最关键的问题

::: danger 电机 = 最强的噪声源
电机运行时会产生强烈的电磁干扰（EMI）。想象一下：电机内部的线圈在快速通断电流，就像一个微型无线电发射台——它会把"噪声"通过导线和空气辐射出去。

如果 MCU 和电机驱动在同一块板上，这些噪声会让：
- ADC 读数跳动（灰度传感器不准）
- 编码器脉冲丢失（里程计算错误）
- 陀螺仪数据乱跳（角度漂移）
- MCU 死机复位（最可怕的情况）
:::

### 日常类比

把电机想象成一台运转中的洗衣机。如果你在洗衣机旁边看书（MCU 运行），你会被噪音干扰。如果书桌在另一个房间（分板），就安静多了。

## 竞赛推荐方案

```
推荐：MCU + 传感器 在一块板，电机驱动独立一块板
```

```
┌──────────────────────┐        ┌──────────────┐
│      核心板           │        │  电机驱动板   │
│                      │  排线   │              │
│  MCU + 传感器接口    │←──────→│  TB6612 ×2   │
│  + 电源管理 + OLED   │  4线    │  + 编码器接口 │
│                      │        │              │
└──────────────────────┘        └──────────────┘
```

**为什么这样分：**

1. **电机驱动板**产生噪声，独立出去就隔离了噪声源
2. **核心板**包含 MCU 和传感器，保持信号干净
3. 两块板之间的连接只有 4 根线（PWM ×2 + 方向 ×2），简单可靠
4. 电驱板烧了（竞赛中常见！）只换一小块，10 分钟搞定

## 选择决策树

```
你的电路中有电机吗？
├── 没有 → 合板，一块搞定
└── 有
    ├── 只有一个直流电机 → 可以合板，注意布局
    └── 有两个以上电机 → 强烈建议分板
```

## 实际布局注意事项

如果选择合板，布局上要注意：

1. **物理距离：** MCU 和电机驱动芯片之间至少留 3cm
2. **地线隔离：** 电机回路的地线和 MCU 的地线分开走，最后在电源入口处单点汇合
3. **电源去耦：** 每块芯片旁边放一个 100nF 小电容（像每个房间放一个灭火器）
4. **走线：** 大电流线（电机供电）走粗线，远离信号线

如果选择分板：

1. **排线选软排线**（FPC）或杜邦线，不要太硬的
2. **接口加防呆**（正反插不对称），防止插反烧板
3. **预留测试点**——小圆焊盘，方便万用表测量

::: warning 分板的一个坑
排线接插件用久了会松动。在赛前检查所有排线是否插紧——用热熔胶固定关键接插件是一个简单有效的方法。
:::

---

## 小结

分板降低风险、方便更换，合板紧凑、省空间。竞赛中可靠性比美观重要——分板带来的"换板自由"远比分板多占的那点空间有价值。

# 成本与时间权衡

## 竞赛中最稀缺的资源

很多人以为竞赛中最大的限制是"预算不够"——错了。**时间是竞赛中最稀缺的资源。** 你在赛场上只有固定的小时数来完成所有任务。一块 30 元的模块你随时可以买到，但 2 天的调试时间你永远拿不回来。

## 两种路线对比

| 维度 | 用成品模块 | 自己画 PCB |
|------|-----------|-----------|
| 时间投入 | 即插即用，1-2 小时 | 原理图 + 布局 + 等生产 3-5 天 |
| 金钱成本 | 每个模块 5-50 元 | 打样 5 元（嘉立创）+ 元件 5-20 元 |
| 可靠性 | 厂家测试过，稳定 | 需要自己调试，可能有 bug |
| 灵活性 | 模块固定功能 | 可以定制任何功能 |
| 体积 | 多个模块组合，体积大 | 集成度高，体积小 |
| 适合阶段 | 学习期、快速验证 | 备赛期、最终版本 |

## 成品模块到底贵不贵

我们来算一笔真实的账：

| 功能 | 成品模块价格 | 自己画 PCB 成本 |
|------|------------|----------------|
| 电机驱动 | TB6612 模块 ~15 元 | PCB 5 元 + 芯片 3 元 + 阻容 2 元 = ~10 元 |
| 陀螺仪 | JY61P 模块 ~40 元 | PCB 5 元 + 芯片 8 元 + 外围 3 元 = ~16 元 |
| 蓝牙串口 | HC-05 模块 ~12 元 | PCB 5 元 + 模块 8 元 + 外围 2 元 = ~15 元 |
| 灰度传感器 | 8 路模块 ~15 元 | PCB 5 元 + 元件 5 元 = ~10 元 |
| **总计** | **~82 元** | **~51 元** |

**差价约 31 元。** 但自己画 PCB 需要：画图（4-8 小时）+ 等生产（3-5 天）+ 焊接（2-4 小时）+ 调试（？小时）。

::: tip 结论
**用成品模块多花 31 元，但节省至少 3-5 天时间。** 在竞赛中，3 天时间价值远超 31 元。
:::

## 竞赛时间策略

### 赛前的"模块库"策略

经验丰富的团队会这样准备：

```
备赛阶段（赛前 1-2 个月）：
├── 完成常用模块的 PCB 设计：电机驱动板、MCU 核心板、电源板
├── 每种板做 3-5 块，充分测试
├── 把代码驱动库都写好、调通
└── 建立"即插即用"的模块库

比赛当晚（拿到赛题后）：
├── 从模块库中挑选需要的模块
├── 用排线连接 → 1 小时内搭建完硬件
├── 剩余所有时间写代码、调参数
└── 如果某块板坏了 → 立刻换备用的
```

### 应急方案：洞洞板

如果你必须在赛场上现场搭建电路，且没有备用 PCB：

::: warning 应急手段
**洞洞板 + 跳线（飞线）——又丑又乱，但它能跑！**

准备一个"应急焊接包"：洞洞板 ×2、细导线一卷、烙铁、焊锡、助焊剂。半小时能搭完一个简单的电机驱动电路。
:::

## 什么值得花钱，什么值得花时间

### 花钱买时间（推荐）

- **成品传感器模块：** 灰度、陀螺仪、超声波——买模块最省心
- **电机驱动模块：** TB6612 模块 15 元，自己画省不了多少钱
- **MCU 核心板：** 买 STM32/MSPM0 最小系统板，别自己焊 LQFP 封装
- **电池：** 航模锂电池 + 充电器，别自己组

### 花时间省钱（合理）

- **连接器/排线：** 买散料自己做比买成品排线便宜，但费时间
- **简单外围电路：** 按键、LED、分压电阻——在 MCU 核心板余位上焊

### 既不值得花钱也不值得花时间

- **自己刻 PCB：** 有毒、精度差、浪费时间——现在嘉立创 5 元打样不香吗
- **自己绕电机：** 除非你是电机专业的

## 一个真实的时间预算

假设比赛是"48 小时制作"：

```
48 小时时间预算：
├── 理解赛题、方案讨论：3 小时
├── 硬件搭建（用预制的模块库）：2 小时
├── 基础代码框架：4 小时
├── 核心算法（巡线/PID/状态机）：18 小时
├── 调试 + 优化：15 小时
├── 写报告：4 小时
└── 休息 + 吃饭：2 小时
```

如果你没有模块库，需要现场画 PCB：

```
48 小时时间预算（现场画 PCB 版）：
├── 理解赛题、方案讨论：3 小时
├── 画原理图 + PCB：8 小时
├── 等 PCB 生产：❌ 根本来不及！（最快 24 小时加急）
└── 结论：比赛结束了你板子还没到
```

::: danger 现场画 PCB 是灾难
在竞赛现场画 PCB 几乎不可能——生产周期至少 24 小时（加急），正常 3-5 天。**竞赛中 90% 的硬件方案应该是预制的模块。**
:::

## 实用建议

1. **赛前把该做的 PCB 都做好、测好。** 赛场上只做"组合"和"编程"。
2. **每种关键模块至少做两块。** 烧了一块还有备用的。
3. **准备一套全模块方案作为备用。** 如果自己画的板出了问题，立刻切回模块方案。
4. **不要在现场焊接 MCU 芯片。** QFP 封装新手焊半小时还可能虚焊——用成品核心板。

---

## 小结

竞赛的隐性预算是时间。用现成模块替代自研电路、用熟悉的平台替代"更先进但没摸过"的方案——省下的时间可以用来调参和测试，这些在赛场上才是真正的竞争力。

---

# SolidWorks vs Blender

## 两个工具，两种世界

初学者常问："我要给机器人设计外壳，该学哪个 3D 软件？" 答案取决于你的目标：**精确制造**还是**好看展示**。

---

## SolidWorks —— 工程师的工具

**SolidWorks** 是工业界标准的机械 CAD（Computer-Aided Design，计算机辅助设计）软件。它的核心是**参数化建模**。

什么是参数化？你画一个方块，给它设定：长度 = 50mm。之后任何时候你改这个数字为 80mm，方块自动变长，所有和它相关的孔、槽、装配关系自动更新。这不是"画了个方块"，而是"定义了一个长 50mm 的方块"。

**SolidWorks 的核心优势：**

- **尺寸精确**：所有特征都有数值约束，公差可控
- **参数可回溯**：修改历史中的任意一步，后续自动更新
- **工程图导出**：一键生成带尺寸标注的工程图
- **标准格式**：导出 STEP、IGES 给工厂加工

**SolidWorks 的缺点：**

- 商业软件，年费数万元（学生版有折扣）
- 学习曲线陡峭
- 对有机曲面（如人体工学外壳）支持不如专业曲面软件

---

## Blender —— 艺术家的工具

**Blender** 是开源免费的三维软件。它的核心是**多边形建模**——用大量的三角面片"捏"出形状。

Blender 没有参数化约束的概念。你拉伸一个面，它就是拉伸了。没有历史记录，没有尺寸约束，一切都是直接的网格操作。

**Blender 的核心优势：**

- 完全免费开源
- 渲染效果极佳（Cycles 渲染器）
- 动画系统强大
- 社区庞大，教程丰富

**Blender 的致命短板（对工程而言）：**

- **没有参数化**：改一个尺寸需要手动调整所有相关特征
- **没有约束系统**：无法定义"这个孔距离边缘 5mm"
- **导出不可靠**：导出的 STL 可能包含破损网格
- **单位系统薄弱**：设计时不强调精确尺寸

::: warning 重要区分
Blender 可以导出 STL 文件用于 3D 打印，但这不意味着它是工程设计工具。用 Blender 做工程设计就像用画笔代替直尺——你可以画出直线，但没有"直"的保证。
:::

---

## 对比总表

| 特性 | SolidWorks | Blender |
|------|------------|---------|
| 建模方式 | 参数化（基于特征） | 多边形（基于网格） |
| 尺寸精度 | 精确到 µm | 无精度保证 |
| 工程图 | 自动生成 | 不支持 |
| 导出格式 | STEP、IGES、STL | 主要是 STL、OBJ |
| 价格 | 数万元/年 | 免费 |
| 学习难度 | 中高 | 中 |
| 适合场景 | 机械零件、外壳制造 | 渲染、动画、游戏 |
| 装配体支持 | 完整 | 有限 |

---

## 竞赛中该用什么？

**优先使用 SolidWorks**（或同类工程 CAD）设计：
- 底盘 / 底板（需要精确的 M3 螺孔定位）
- 传感器支架
- 编码器安装座
- 任何需要螺丝固定的部件

### 免费替代方案

如果无法获取 SolidWorks：

| 软件 | 特点 |
|------|------|
| **FreeCAD** | 开源，参数化，功能接近 SolidWorks | 
| **Fusion 360** | 学生免费，Autodesk 出品，云协作 |
| **Onshape** | 浏览器运行，免费版文件公开 |
| **TinkerCAD** | 极简入门，拖拽式建模，小学生都能上手 |

::: tip 竞赛建议
大多数竞赛团队使用 Fusion 360（学生免费）或 SolidWorks（学校授权）。FreeCAD 是完全免费的最佳选择。关键是**学会参数化建模思想**，工具只是载体。
:::

### 3D 打印 vs 激光切割

- **3D 打印**（FDM/光固化）：适合复杂的、有厚度的三维结构（传感器外壳、电池仓）
- **激光切割**：适合平板结构（亚克力底板、支架）。比亚克力便宜且快，详见下一章

```c
// 设计思维对比
// 不好的设计：整块铝 CNC 铣出来 → 贵、慢、重
// 好的设计：亚克力平板 + 3D 打印连接件 + 标准螺丝装配
//           → 便宜、快、轻、可修改
```

## 小结

SolidWorks 做工程零件，输出 STEP 给 CNC 加工；Blender 做渲染展示，效果好看但不是工程文件。电赛结构设计用 SolidWorks / Fusion 360 / FreeCAD，3D 打印和激光切割搭配使用——平板结构用亚克力，复杂三维结构用打印件。

# 亚克力板

## 什么是亚克力？

**亚克力**（Acrylic，学名 PMMA，聚甲基丙烯酸甲酯）是一种透明的塑料，俗称"有机玻璃"。它最大的优势是可以用**激光切割**的方式进行快速、精确、廉价的加工。

打比方：3D 打印像是用热熔胶枪一层层堆出形体（慢但自由），激光切割像是在纸上用剪刀精确地剪出形状（快但只限于平板）。亚克力板就是那张"纸"。

---

## 为什么竞赛中推荐亚克力？

竞赛准备时间紧张，预算有限。亚克力 + 激光切割天然适配：

- **便宜到离谱**：A4 大小的 3mm 亚克力板，淘宝几块钱一张。激光切割服务费通常十几元
- **快**：设计好 CAD 文件发给店家，半天到一天就切好寄出
- **精确**：激光切割精度可达 0.1mm，远高于 FDM 3D 打印机（通常 0.2~0.5mm）
- **够硬**：3mm 亚克力已经有不错的刚性，5mm 则非常坚固
- **透明**：可以看到里面的电路，便于调试和展示
- **螺丝装配**：直接用 M3 螺丝 + 螺母连接，拆装方便

---

## 设计规范

### 厚度选择

| 厚度 | 适用场景 |
|------|----------|
| 2mm | 轻量面板、装饰盖板 |
| 3mm | 通用底盘、支架（最常用） |
| 5mm | 承重底盘、电机安装座 |
| 8mm+ | 重型结构，极少使用 |

**3mm 是最常用的厚度**，兼顾强度、重量和成本。

### 螺丝孔设计

标准 M3 螺丝需要 **3.2mm ~ 3.4mm** 直径的通孔。留太小螺丝拧不进去，留太大螺丝晃荡。

```c
// 亚克力板螺丝孔设计示例
// M3 螺丝外径 = 3.0mm
// 通孔直径建议 = 3.2mm（紧配合）
//           或 = 3.4mm（松配合，有调整空间）
//
// 螺帽 / 螺母槽：
// M3 螺母对边距 = 5.5mm
// 六角槽内切圆 ≈ 6.0~6.2mm（螺母能放进去）
// 深度 = 螺母厚度 2.4mm（如果板厚 5mm，可铣沉头槽）
```

### 倒角与圆角

激光切割的内角一定是直角（激光束直径约 0.1~0.2mm）。但直角内角是应力集中点，受力时容易开裂。如果 CAD 允许，内角加 R2~R3 的**圆角**能大幅提高强度。

同样，板子外角的直角很尖锐，容易伤人。外角至少加 R1~R2 的圆角。

---

## 完整工作流

### 第一步：CAD 设计

在 SolidWorks / Fusion 360 / FreeCAD 中画好 2D 草图，或直接画 3D 零件后导出面。设计时注意：

- 所有特征在一个平面上（亚克力是平板，不能做三维曲面）
- 包含所有螺丝孔、线缆过槽、传感器窗口
- 检查孔位是否对齐（用装配功能把电路板虚拟放进来看）

### 第二步：导出 DXF

**DXF**（Drawing Exchange Format）是激光切割行业的标准格式。从 CAD 软件导出 DXF 时注意：

- 只导出一个平面（正视图或俯视图）
- 比例设为 1:1
- 单位设为毫米

::: warning 常见错误
导出 DXF 时比例设成了 1:10 或 1:2 —— 切出来的板子要么大到离谱，要么小到不能用。**导出后一定检查尺寸：在 DXF 查看器里量一个已知孔距，确认和设计一致。**
:::

### 第三步：发给切割服务

淘宝搜索"亚克力激光切割"，把 DXF 发给店家，告诉他们：
- 材料：亚克力（透明 / 黑色 / 白色）
- 厚度：3mm（或其他）
- 数量：几片
- 需要快递

通常 1~2 天就能收到。

### 第四步：装配

收到的亚克力片边缘可能有轻微毛刺，用细砂纸轻轻打磨即可。用 M3 螺丝 + 螺母 + 铜柱（六角螺柱）把电路板、传感器、电机等固定在亚克力板上。

---

## 亚克力 vs 3D 打印：什么时候用哪个？

| 需求 | 选亚克力 | 选 3D 打印 |
|------|----------|-----------|
| 平板结构（底盘、面板） | 最优 | 也可以用，但慢 |
| 复杂三维结构 | 不行 | 最优 |
| 需要超低成本 | 极低（几元） | 中等（几十元） |
| 需要透明 | 天生透明 | 部分材料可透明 |
| 需要超快交付 | 当天可取 | 通常需要几小时到一天 |
| 后期修改 | 重新切一块（便宜） | 重新打印（时间成本） |

::: tip 组合使用的智慧
最佳实践：**亚克力做"板"，3D 打印做"块"**。底盘用亚克力激光切割（便宜、快、平），传感器支架用 3D 打印（有厚度、有造型），两者用 M3 螺丝连接。
:::

---

## 成本参考（2024 年行情）

| 项目 | 价格 |
|------|------|
| 3mm 透明亚克力 A4 板材 | 3~5 元 |
| 激光切割加工费（A4 幅面） | 10~20 元 |
| M3×10 螺丝 + 螺母 100 套 | 5~10 元 |
| M3 铜柱（六角，6mm/10mm/15mm） | 2~5 元/10 个 |

总计：一块完整的机器人底盘（亚克力 + 紧固件）不到 **30 元**。用坏了换一块也就一杯奶茶的钱。

## 小结

亚克力板是电赛结构件的首选——轻、便宜、透光、易加工。2-3mm 厚度够用于大部分支架和底板。加工方式首选激光切割，精确到 0.1mm。和 3D 打印搭配使用：亚克力做"板"，打印做"块"。

# 直流有刷电机 vs 无刷电机

电赛控制题里，电机是你的"腿"和"手"。选错电机类型，后面的电路和代码都得重来。两种最常见的直流电机——**有刷**和**无刷**，看起来都能转，但内部原理和应用场景差别很大。

## 有刷直流电机（Brushed DC Motor）

有刷电机的结构最简单：

```
   [N 极永磁体]
      │
   ═══╪═══ 线圈（转子，通电后产生磁场）
      │
   [S 极永磁体]
      │
   ┌──┴──┐
   │ 电刷 ├── 换向器（commutator）
   └─────┘
```

- **转子**（Rotor）：线圈，通电后产生磁场
- **定子**（Stator）：永磁体，固定不动
- **电刷 + 换向器**：把电流通到旋转的线圈上，同时自动切换电流方向

电刷是**机械接触**的——碳刷压在旋转的换向器上，时间久了会磨损、打火花。但结构简单，用 PWM 就能调速，驱动电路也便宜（一个 H 桥芯片几块钱）。

| 特性 | 有刷电机 |
|------|---------|
| 结构 | 简单 |
| 驱动电路 | H 桥，便宜（TB6612 / L298N） |
| 寿命 | 几千小时（电刷磨损） |
| 效率 | ~75% |
| 噪声 | 有电刷摩擦声和电火花干扰 |
| 价格 | 便宜 |

## 无刷直流电机（Brushless DC Motor，BLDC）

无刷电机把"谁转谁不转"反了过来：

- **转子**变成永磁体（转）
- **定子**变成线圈（不转）
- 不需要电刷——通过**电子换向**，由控制器（ESC）按顺序给不同线圈通电

```
   ┌── 线圈 A ──┐
   │            │
   │    ┌──┐    │
   │    │永│    │  ← 转子（永磁体，旋转）
   │    │磁│    │
   │    └──┘    │
   │            │
   └── 线圈 C ──┘
      线圈 B
        ↑
   三相驱动线（UVW）
```

| 特性 | 无刷电机 |
|------|---------|
| 结构 | 复杂 |
| 驱动电路 | 三相逆变器 + 控制算法（ESC），贵 |
| 寿命 | 几万小时（无机械接触） |
| 效率 | ~90% |
| 噪声 | 安静 |
| 价格 | 贵 |

::: tip 电赛里的选择
控制题（巡线小车、平衡车）绝大多数用**有刷电机**就够了。无刷电机虽然强，但 ESC 驱动复杂，调参难度大，性价比不高。
:::

## 两种电机的驱动方式

**有刷电机：H 桥**

```
   VCC
    │
   ┌┴─┐
   │Q1│  当 Q1+Q4 导通 → 电机正转
   └┬─┘  当 Q2+Q3 导通 → 电机反转
    ├─────── M ───────┤
   ┌┴─┐              ┌┴─┐
   │Q2│              │Q3│
   └┬─┘              └┬─┘
    └─────── GND ─────┘
```

仅需一个 PWM 信号就能调速——这是 TB6612 做的事（详见第十篇）。

**无刷电机：三相逆变器（ESC）**

需要 6 个 MOSFET、精密的换向时序和反电动势检测。电赛中用现成的电调模块（ESC）即可，不需要自己设计。

## 何时选哪种

| 场景 | 推荐 |
|------|------|
| 巡线小车、物流小车 | 有刷 DC + TB6612 |
| 高速竞速、四轴飞行器 | 无刷 + ESC |
| 风扇、水泵 | 有刷 DC（低成本） |
| 精密位置控制 | 步进电机（见下篇） |

## 小结

有刷电机便宜好驱动，是电赛小车的首选。无刷电机效率高寿命长，但驱动复杂。入门阶段先搞懂有刷电机 + H 桥，这就是你 90% 的比赛需求。

# 步进电机原理与驱动

## 什么是步进电机

普通有刷电机通电就**连续转**，你不知道它转了多少圈。步进电机每给一个脉冲就转动一个**固定角度**（步距角），你能精确控制它的**位置**和**转速**——不用编码器也能做到开环定位。

核心原理：电机内部有多组线圈（相），按一定顺序给线圈通电，转子就会被"拽"到下一个角度位置，一步一趋。

```
二相步进电机简化原理：

Step 1:  励磁 A+        Step 2:  励磁 B+
          │                       │
       ┌──┴──┐                ┌──┴──┐
   A+ ─┤      ├─ A-       A+ ─┤      ├─ A-
       │ 转子 │                │ 转子→│
   B+ ─┤      ├─ B-       B+ ─┤  朝  ├─ B-
       └──┴──┘                └──┴──┘
          │                       │
         转子朝 A+               转子朝 B+
         转了 1.8°               又转了 1.8°
```

## 关键参数

| 参数 | 含义 | 典型值 |
|------|------|--------|
| 步距角 | 每步旋转角度 | 1.8°（200 步/圈） |
| 相数 | 内部线圈组数 | 2 相最常见 |
| 额定电流 | 每相最大电流 | 0.5A ~ 2A |
| 保持力矩 | 静止时的锁紧力 | N·cm 为单位 |
| 额定电压 | 驱动电压 | 通常远高于铭牌电压（靠电流斩波） |

## 驱动方式

步进电机不能直接接电源——必须用**专用驱动芯片**。

常用的驱动方式：

| 驱动方式 | 说明 | 力矩 |
|----------|------|------|
| 单相励磁（Wave） | 每次只给一相通电 | 弱 |
| 双相励磁（Full Step） | 每次给两相通电 | 强（单相的 √2 倍） |
| 半步（Half Step） | 单相/双相交替 | 介于两者之间，步距角减半 |
| 微步（Microstep） | PWM 模拟正弦电流 | 步距角极小，超平滑 |

现代驱动芯片（A4988 / TMC2209 / DRV8825）都支持微步，你只需要：

```c
// 用 STM32 定时器发脉冲，一个脉冲 = 一个微步
// 脉冲频率 = 转速，脉冲数量 = 位置
HAL_TIM_PWM_Start(&htim2, TIM_CHANNEL_1);  // 开始发脉冲
// 改变频率 = 改变速度
// 计数脉冲 = 知道位置
```

## 驱动芯片推荐

| 芯片 | 峰值电流 | 细分 | 特点 |
|------|---------|------|------|
| A4988 | 2A | 1/16 | 经典、便宜、够用 |
| DRV8825 | 2.5A | 1/32 | 电流更大、细分更细 |
| TMC2209 | 2A | 1/256 | 超静音、UART 配置 |

::: tip 电赛里用哪个
**A4988** 是最稳妥的选择——几块钱一个，驱动 42 步进电机绰绰有余。TMC2209 更安静但贵一些，除非你的项目对噪声敏感，否则不需要。
:::

## 接线

```
STM32          A4988          42步进电机
───────       ───────        ─────────
PA0 ────────→ STEP           1A ─── 黑线
PA1 ────────→ DIR            1B ─── 绿线
PA2 ────────→ EN             2A ─── 红线
               VMOT ── 12V   2B ─── 蓝线
               GND  ── GND
```

- **STEP**：每个上升沿走一步（微步）
- **DIR**：高/低电平控制方向
- **EN**：低电平使能

::: warning
步进电机的**电源 VMOT 和单片机电源必须分开**！步进电机启动瞬间电流可以到额定电流的 2-3 倍，如果和 MCU 共用一个 3.3V，MCU 会瞬间掉电重启。
:::

## 小结

步进电机适合需要**精确定位但不想要编码器**的场景：3D 打印机、CNC、自动门、云台。电赛里如果遇到"走指定距离后停下"的任务，步进电机比有刷电机 + 编码器更省事。驱动芯片接 STEP/DIR 两根线就能用，代码比你想的简单。

# 舵机原理与使用

## 舵机是什么

舵机（Servo）是一种**自带反馈控制的电机模组**。你给它一个角度信号，它自己转过去并锁死在那里——内部有电机、齿轮、电位器和控制电路，全部封装在一个小盒子里。

它和普通电机的本质区别：**舵机是位置闭环的，普通电机是开环的。**

## 控制信号

舵机用 **PWM 脉冲宽度** 表示目标角度：

```
周期固定 20ms（50Hz）

   ┌──┐
   │  │  0.5ms ── 0°
   └──┘
   ┌─────┐
   │     │  1.5ms ── 90°（中间位置）
   └─────┘
   ┌───────┐
   │       │  2.5ms ── 180°
   └───────┘

   |<────── 20ms ──────>|
```

| 脉冲宽度 | 对应角度 | 典型用法 |
|----------|---------|---------|
| 0.5ms | 0° | 最小极限 |
| 1.0ms | 45° | |
| 1.5ms | 90° | 中间位置 |
| 2.0ms | 135° | |
| 2.5ms | 180° | 最大极限 |

```c
// STM32 TIM 配置：PWM 频率 50Hz，用 CCR 控制脉宽
// 对于 72MHz 时钟，PSC=72, ARR=20000 → 72M/(72×20000) = 50Hz
// CCR 值 = 脉宽(ms) / 20ms × 20000
// CCR=500  → 0.5ms → 0°
// CCR=1500 → 1.5ms → 90°
// CCR=2500 → 2.5ms → 180°

void servo_set_angle(TIM_HandleTypeDef *htim, uint32_t channel, uint8_t angle) {
    // angle: 0~180
    uint32_t pulse = 500 + (angle * 2000 / 180);  // 500~2500
    __HAL_TIM_SET_COMPARE(htim, channel, pulse);
}
```

## 舵机类型

| 类型 | 特点 | 适用场景 |
|------|------|---------|
| 标准舵机 | 0°~180° 角度控制 | 机械臂、云台、转向 |
| 连续旋转舵机 | 360° 旋转，控制速度而非角度 | 简易小车驱动轮 |
| 数字舵机 | 更高精度、更快响应 | 航模、机器人关节 |
| 金属齿轮舵机 | 齿轮耐撞 | 受力较大的场合 |

::: tip 电赛常用
**SG90**（塑料齿，便宜，适合小力矩）和 **MG996R**（金属齿，大力矩，稍贵）。SG90 几块钱一个，做云台、机械臂足够；MG996R 适合需要"使劲"的场合。
:::

## 供电

舵机不能从 MCU 的 3.3V 取电！标准舵机需要 **5V~6V**，MG996R 堵转电流可达 **2.5A**。

::: danger 血的教训
一个舵机堵转就能拉低整个 5V 总线 → MCU 掉电 → 程序跑飞。**舵机必须独立供电**，并且电源芯片要有足够余量。
:::

## 小结

舵机是最省心的"位置执行器"——三根线（信号、VCC、GND），一个 PWM，角度任你摆。电赛里做机械臂、转向机构、云台，舵机是第一选择。参数速记：50Hz PWM，0.5ms~2.5ms 脉宽对应 0°~180°，独立供电。

# 电机选型与供电计算

前几篇分别介绍了有刷、无刷、步进、舵机四种电机。真正动手做项目时，问题变成——**"我的车到底用哪种？用几个？电池要多大？"**

## 选型决策流程

```
你的任务需要什么？
  ├─ 只需要"转"（驱动车轮、风扇）→ 有刷直流电机
  │   ├─ 需要调速 → 有刷 + TB6612 + PWM
  │   ├─ 需要知道走了多远 → 有刷 + 编码器
  │   └─ 需要高速 → 无刷 + ESC
  │
  ├─ 需要精确位置（走 N 步停下）→ 步进电机
  │   └─ 不需要编码器就能定位，但速度慢
  │
  └─ 需要摆到某个角度 → 舵机
      └─ 自带闭环，即插即用
```

## 关键参数速查

| 你要知道 | 怎么看 | 为什么重要 |
|----------|--------|-----------|
| 转速 | 数据手册上的 RPM 或 KV 值 | 决定车能跑多快 |
| 力矩/扭矩 | 数据手册上的 Torque（N·cm 或 kg·cm） | 决定能不能拉动你的车 |
| 额定电压 | 标签上的 Voltage | 决定了电池电压 |
| 额定电流 | 数据手册上的 Rated Current | 决定了驱动芯片和电源 |
| 堵转电流 | Stall Current | **电源必须能扛住这个电流，否则直接掉电** |

## 小车电机估算实例

一个典型的巡线小车（3kg，两个驱动轮 + 一个万向轮）：

**需要的推力：**
```
F = m × g × μ = 3kg × 10 × 0.1 ≈ 3N
```
（轮子在赛道上滚动摩擦系数 μ 约 0.05~0.15，取 0.1）

**需要的力矩（每个电机）：**
```
T = F × r / 2 = 3N × 0.03m / 2 ≈ 0.045 N·m = 4.5 N·cm
```
（轮子半径 r 约 3cm，两个电机分担）

市面上常见的 TT 马达（黄色减速电机）力矩约 100 N·cm，转速 200 RPM，对小车轮绰绰有余。

## 供电计算

假设你的小车：
- 2 个 TT 马达，每个额定 0.5A，堵转 2A
- 1 个舵机，堵转 2.5A
- MCU + 传感器，约 0.2A
- 其他（蓝牙、LED），约 0.1A

**极端情况总电流：** 2×2A + 2.5A + 0.2A + 0.1A = **6.8A**

::: warning 留余量
电源设计要按**堵转电流之和 × 1.5**来算余量。6.8A × 1.5 ≈ **10A**。选一块能持续输出 10A 的电池和驱动板。
:::

**续航估算：**
```
续航 = 电池容量 / 平均电流
     = 2000mAh / 2A（平均行驶电流）
     ≈ 1 小时
```

比赛一般不需要跑 1 小时，但调试阶段一天可能跑几十次——带足电池，或者准备可更换的电池方案。

## 常用电机速查表

| 电机 | 额定电压 | 额定电流 | 转速 | 力矩 | 价格 | 用途 |
|------|---------|---------|------|------|------|------|
| TT 马达（黄） | 6V | 0.5A | 200 RPM | ~100 N·cm | ¥5 | 小车驱动轮 |
| N20 减速电机 | 6V | 0.15A | 100~1000 RPM | ~20 N·cm | ¥10 | 小型机构 |
| 42 步进电机 | 12V | 1.5A | 步进 | ~40 N·cm | ¥20 | 精确定位 |
| SG90 舵机 | 5V | 0.75A 堵转 | — | 1.8 kg·cm | ¥5 | 轻量云台 |
| MG996R 舵机 | 6V | 2.5A 堵转 | — | 10 kg·cm | ¥20 | 重型机械臂 |
| 2212 无刷 | 11.1V | 12A | 1000 KV | — | ¥40 | 四轴飞行器 |

## 小结

选电机不是越贵越好，是**够用就好**。TT 马达 + TB6612 能覆盖 90% 的电赛小车需求。选型时先算力矩够不够、再算电流扛不扛得住、最后算续航够不够。电源永远留 1.5 倍余量。

# OLED 屏（I2C/SPI）

## 为什么需要屏幕

串口打印在电脑上看没问题，但你不能每次都开着电脑看数据。在比赛现场——或者你的小车在跑的时候——有一块小屏幕实时显示**传感器值、电池电压、当前状态**，调试效率直接翻倍。

## 0.96 寸 SSD1306 OLED

这是嵌入式世界里最常见的显示屏，99% 的电赛选手第一块屏幕就是它。

| 参数 | 值 |
|------|-----|
| 尺寸 | 0.96 英寸 |
| 分辨率 | 128×64 像素 |
| 颜色 | 单色（蓝/白/黄蓝双色） |
| 驱动芯片 | SSD1306 |
| 接口 | I2C 或 SPI |
| 工作电压 | 3.3V |
| 价格 | ¥5~10 |

## I2C vs SPI 版本

| | I2C 版本 | SPI 版本 |
|------|---------|----------|
| 引脚数 | 4 根（VCC GND SCL SDA） | 7 根 |
| 刷新速度 | 较慢 | 快 |
| 占用 MCU 资源 | 少 | 多 |
| 推荐 | ✅ 引脚紧张时首选 | 需要高频刷新时用 |

::: tip 新手推荐
买 **I2C 版本的 SSD1306 模块**（蓝色小方块，4 个引脚）。引脚少、接线简单。I2C 地址默认 0x3C，扫描 I2C 设备时看到它就对了。
:::

## 接线

```
STM32                SSD1306 (I2C)
───────              ──────────
3.3V  ───────────── VCC
GND   ───────────── GND
PB6(SCL) ────────── SCL
PB7(SDA) ────────── SDA
```

## 代码示例（STM32 HAL + u8g2）

```c
// 初始化 I2C（CubeMX 配置好 I2C1）
// 使用 u8g2 库（最流行的单色屏图形库）

#include "u8g2.h"

u8g2_t u8g2;

// 初始化函数（回调送数据到 I2C）
uint8_t u8x8_byte_hw_i2c(u8x8_t *u8x8, uint8_t msg, uint8_t arg_int, void *arg_ptr) {
    switch (msg) {
        case U8X8_MSG_BYTE_SEND:
            HAL_I2C_Master_Transmit(&hi2c1, 0x3C << 1, arg_ptr, arg_int, 100);
            break;
        case U8X8_MSG_BYTE_INIT:
        case U8X8_MSG_BYTE_SET_DC:
        case U8X8_MSG_BYTE_START_TRANSFER:
        case U8X8_MSG_BYTE_END_TRANSFER:
            break;
    }
    return 0;
}

// 在 main() 中
u8g2_Setup_ssd1306_i2c_128x64_noname_f(&u8g2, U8G2_R0, u8x8_byte_hw_i2c, NULL);
u8g2_InitDisplay(&u8g2);
u8g2_SetPowerSave(&u8g2, 0);  // 唤醒屏幕

// 画东西
u8g2_ClearBuffer(&u8g2);
u8g2_SetFont(&u8g2, u8g2_font_6x10_tf);
u8g2_DrawStr(&u8g2, 0, 10, "Hello 电赛!");
u8g2_DrawStr(&u8g2, 0, 30, "Speed: 100");
u8g2_SendBuffer(&u8g2);  // 一次性刷新到屏幕
```

## OLED 在电赛里的典型用法

- **显示实时数据**：速度、位置、传感器值
- **显示系统状态**：当前模式（巡线/避障/停车）
- **参数调试界面**：按键翻页，修改 PID 参数
- **错误提示**：陀螺仪初始化失败？电机堵转？屏幕上直接显示

::: tip
屏幕刷新不要放 while(1) 里每循环一次刷一次——那会严重拖慢主循环。建议用定时器中断，每 100ms 刷新一次屏幕。
:::

## 小结

一块 5 块钱的 OLED 屏，加上 u8g2 库，15 行代码就能让你的小车"开口说话"。I2C 版本接线最少，推荐作为你的第一块显示屏。

# TFT 屏

## TFT vs OLED：什么时候升级

OLED 很好，但它只有 128×64 像素、单色、不到 1 寸大。当你需要**彩色、大画面、触摸交互**时，就该 TFT 屏上场了。

| | SSD1306 OLED | TFT LCD |
|------|-------------|---------|
| 颜色 | 单色 | 65536 色（16bit）或更多 |
| 分辨率 | 128×64 | 240×320 起步，可达 800×480 |
| 尺寸 | 0.96" | 1.8" ~ 3.5" |
| 刷新率 | 较慢 | 可达 30+ FPS |
| 功耗 | 极低 | 中等（有背光） |
| 价格 | ¥5~10 | ¥15~50 |
| 驱动方式 | I2C/SPI | 多为 SPI，大屏用并口 |

## 常用 TFT 驱动芯片

市面上大多数小尺寸 TFT 屏用这几个驱动芯片：

| 驱动 IC | 分辨率 | 接口 | 常见搭配 |
|---------|--------|------|---------|
| ST7735 | 128×160 | SPI | 1.8" 屏 |
| ST7789 | 240×240 / 240×320 | SPI | 1.3"~2.4" 屏 |
| ILI9341 | 240×320 | SPI/并口 | 2.4"~3.2" 屏 |

## SPI 接线

TFT SPI 屏通常 8 根线：

```
STM32                TFT
───────              ───
3.3V  ────────────── VCC
GND   ────────────── GND
PA5(SCK) ─────────── SCL
PA7(MOSI) ────────── SDA (MOSI)
PA4 ──────────────── CS   (片选)
PA3 ──────────────── DC   (数据/命令选择)
PA2 ──────────────── RST  (复位)
           ┌─────── BL   (背光，可接 PWM 调亮度或直接接 3.3V)
```

## 图形库选择

自己手写打点画线太累了。用现成的图形库：

| 库 | 特点 | 适合 |
|----|------|------|
| **TFT_eSPI** | 轻量、Arduino 生态最流行 | 简单 UI、数据展示 |
| **LVGL** | 重量级、界面漂亮、支持触屏 | 复杂交互、动画 |
| **uGUI** | 超轻量（仅 3KB RAM） | 资源极有限的 MCU |

::: tip 电赛推荐
**TFT_eSPI** 对新手最友好，配置好引脚后几行代码就能画图。**LVGL** 是趋势，界面效果接近手机 App，但需要更多的 Flash 和 RAM（STM32F4 以上才比较舒服）。
:::

## TFT_eSPI 快速上手

```c
// 配置好后（CubeMX 初始化 SPI + GPIO），只需：
#include <TFT_eSPI.h>

TFT_eSPI tft = TFT_eSPI();

void setup() {
    tft.init();
    tft.setRotation(1);
    tft.fillScreen(TFT_BLACK);
    tft.setTextColor(TFT_WHITE, TFT_BLACK);
    tft.drawString("Speed:", 10, 10, 2);
    tft.drawFloat(speed, 1, 80, 10, 4);  // 大字号显示速度
}
```

## 触摸屏

常见的 2.4"~3.2" TFT 经常带电阻式触摸（XPT2046 芯片，SPI 接口）。可以做参数设置界面——在赛场上用手指点一下改 PID 参数，比按键翻来翻去快得多。

::: warning
TFT 屏 + LVGL + 触屏这三件套确实很酷，但**别花太多时间在 UI 上**。比赛看的是功能能不能跑，不是界面好不好看。数据能用就行，别追求花哨的动画。
:::

## 小结

TFT 屏给小车加上了彩色显示。新手从 TFT_eSPI + ST7789 开始，够用且简单。追求好看界面的用 LVGL。但别花太多时间在 UI 上——比赛比的是功能，不是界面。

# 按键矩阵与编码器旋钮

屏幕负责"说"，按键和旋钮负责"听"。你的小车需要一个**人机交互接口**——切换模式、调整参数、确认操作。

## 独立按键

最简单的输入方式：

```
3.3V ─── 电阻(10kΩ) ──┬── 按键 ── GND
                      │
                      └── MCU GPIO（配置为上拉输入）
```

按下去 → GPIO 读到低电平。没按 → 上拉电阻拉到高电平。

```c
// 读取按键（带消抖）
if (HAL_GPIO_ReadPin(KEY1_GPIO_Port, KEY1_Pin) == GPIO_PIN_RESET) {
    HAL_Delay(20);  // 消抖
    if (HAL_GPIO_ReadPin(KEY1_GPIO_Port, KEY1_Pin) == GPIO_PIN_RESET) {
        // 按键确实被按下了
        mode = (mode + 1) % MODE_COUNT;
        while (HAL_GPIO_ReadPin(KEY1_GPIO_Port, KEY1_Pin) == GPIO_PIN_RESET);  // 等待释放
    }
}
```

::: warning
上面代码里的 `HAL_Delay(20)` 和 `while(等待释放)` 会**阻塞整个程序**。小车跑着跑着你按个键，车就停了。更好的做法是用定时器轮询做非阻塞按键检测。
:::

## 按键矩阵

按键多了（比如 4×4=16 个按键），每个按键一个 GPIO 引脚太浪费。矩阵接法：

```
       COL1  COL2  COL3  COL4
        │     │     │     │
ROW1 ──┼──[K1]──[K2]──[K3]──[K4]
ROW2 ──┼──[K5]──[K6]──[K7]──[K8]
ROW3 ──┼──[K9]──[K10]─[K11]─[K12]
ROW4 ──┼──[K13]─[K14]─[K15]─[K16]
```

- 行（ROW）设为输入上拉
- 列（COL）设为输出，逐列拉低
- 读行引脚，知道哪行被拉低 → 就知道哪个键被按下
- 4+4=8 个引脚管 16 个按键，效率翻倍

## 旋转编码器（Rotary Encoder）

用一个旋钮调节参数，比反复按键快 10 倍。

### 工作原理

编码器有两个输出（A 相和 B 相），旋转时输出的方波相位差 90°：

```
  正转（顺时针）：
  A: ┌─┐   反转（逆时针）：
  B: ──┌─┐  A: ──┌─┐
             B: ┌─┐
     A 先变  →  B 先变
```

判断 A 和 B 谁先变化就知道方向。

```c
// 编码器中断处理
void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin) {
    if (GPIO_Pin == ENCODER_A_Pin) {
        if (HAL_GPIO_ReadPin(ENCODER_B_GPIO_Port, ENCODER_B_Pin) == GPIO_PIN_SET) {
            value++;  // 顺时针：A 下降沿时 B 是高
        } else {
            value--;  // 逆时针
        }
    }
}
```

::: tip 带按钮的编码器
很多旋转编码器自带**按下功能**（像鼠标滚轮可以点）。旋转调参，按下确认——一个元件搞定选择和确认两个操作。
:::

## 电赛里的人机交互方案

| 方案 | 硬件 | 适合场景 |
|------|------|---------|
| 1 个按键 + OLED | 最少引脚 | 简单模式切换 |
| 3 个按键（上/下/确认）+ OLED | 3 个 GPIO | 菜单翻页 + 参数调节 |
| 编码器 + 按键 + OLED | 3~5 个 GPIO | 快速调参（PID 调试场景） |
| TFT 触摸屏 | SPI + SPI | 复杂交互，图形界面 |

## 小结

按键是最简单的人机交互，矩阵接法省引脚。编码器旋钮是调参利器——PID 调试时转一转比按几十次键快得多。所有按键检测都应该**非阻塞**，别让按键拖累你的主循环。

# 电池选型

电源是电赛项目的"心脏"。电源没选好，你的电路和代码再漂亮也是废铁。这一章帮你搞清楚——我那辆小车到底该用什么电池？

## 电池类型速览

| 类型 | 标称电压 | 能量密度 | 放电能力 | 安全性 | 价格 | 典型用途 |
|------|---------|---------|---------|-------|------|---------|
| **锂聚合物（Li-Po）** | 3.7V/串 | 高 | 很强（20C~50C） | 需保护板 | 中 | 航模、大电流设备 |
| **18650 锂电池** | 3.7V | 高 | 中等（5C~10C） | 好 | 低 | 移动电源、一般设备 |
| **镍氢（NiMH）** | 1.2V | 中 | 中等 | 好 | 低 | 传统竞赛小车 |
| **磷酸铁锂（LiFePO4）** | 3.2V | 中 | 强 | 极好 | 较高 | 高安全需求场景 |
| **碱性/碳性干电池** | 1.5V | 低 | 弱 | 极好 | 极低 | 不推荐电赛使用 |

## 关键参数解释

### 电压（V）

单节 Li-Po 满电 4.2V，标称 3.7V，放电截止 3.0V（不能再低了！）。

常见的组合：
- **1S**：3.7V，适合直接给 3.3V LDO 供电
- **2S**：7.4V，适合给 5V/3.3V 系统供电，需要降压
- **3S**：11.1V，适合驱动 12V 电机，需要降压给 MCU

### 容量（mAh）

1000mAh = 以 1A 电流放电能撑 1 小时。实际可用容量约标称的 80%（放电深度不宜超过 80%，否则损伤电池寿命）。

### C 数（放电倍率）

C 数 × 容量 = 最大持续放电电流。

```
2200mAh 25C 电池 → 2.2A × 25 = 55A 持续放电！
2200mAh 5C  电池 → 2.2A × 5  = 11A
```

电赛小车一般 5C~10C 就够，航模才需要高 C 数。

::: danger 锂电池安全
锂电池**过放（<3.0V）会永久损坏，过充（>4.2V）会起火爆炸**。必须使用带保护板的电池，或者外加保护电路！电赛现场冒烟不是你想看到的结果。
:::

## 电赛场景推荐

| 场景 | 推荐 | 理由 |
|------|------|------|
| 巡线小车 | 2S Li-Po（7.4V）或 6 节 NiMH（7.2V） | 电机要 5~12V，MCU 降压取 3.3V |
| 小型车/传感器平台 | 2 节 18650 串联（7.4V） | 便宜、安全、够用 |
| 大功率赛车 | 3S Li-Po（11.1V） | 高电压带来高转速 |
| 四轴飞行器 | 3S~4S 高 C 数 Li-Po | 需要大电流瞬时放电 |

## 小结

新手从 **2S Li-Po（7.4V）+ BMS 保护板** 开始。电压适中——通过 DC-DC 降压给 MCU（3.3V），直接供电给电机驱动（6~12V 都能接受）。18650 比软包 Li-Po 更安全，推荐入门使用。

# 电压转换方案

电池电压很少刚好是 MCU 需要的 3.3V 或电机需要的 12V。你需要**电压转换**——把电池电压变成各模块需要的电压。

## LDO 与 DC-DC 回顾

第九篇电源设计章已经详细讲过 LDO 和 DC-DC 的原理，这里聚焦在**如何组合使用**。

| | LDO（线性稳压） | DC-DC（开关稳压） |
|------|---------------|-----------------|
| 效率 | 低（压差大的话发热严重） | 高（85%~95%） |
| 纹波 | 极低 | 有开关噪声 |
| 外围元件 | 输入/输出各一个电容 | 电感 + 电容 + 二极管/同步管 |
| 适合 | 模拟电路、MCU 供电 | 电机、大电流、大压差 |
| 典型芯片 | AMS1117-3.3, HT7533 | MP1584, MT3608 |

## 一个典型的电赛电源树

```
电池 2S Li-Po (7.4V)
  │
  ├─ 直接供电 ─────→ TB6612 电机驱动（Vmotor，6~12V）
  │
  ├─ DC-DC Buck ──→ 5V ─┬─→ 舵机（5V，高峰值电流）
  │    (MP1584)          │
  │                      └─→ LDO ──→ 3.3V ──→ MCU + OLED + 传感器
  │                           (AMS1117-3.3)
  │
  └─ 可选：DC-DC Boost ──→ 12V ──→ 步进电机 / 其他高压外设
       (MT3608)
```

## 为什么不用 7.4V 直接进 LDO 出 3.3V？

因为 LDO 消耗的功率 = (Vin - Vout) × Iout。假设系统需要 500mA：

```
P_loss = (7.4 - 3.3) × 0.5 = 2.05W
```

2W 的热量集中在一块指甲盖大小的 LDO 上，很快就能烫伤你的手指。这就是为什么**先 DC-DC 把电压降到 5V，再 LDO 出 3.3V**——压差从 4.1V 降到 1.7V，热量降到 1/3。

## 常用转换芯片速查

| 型号 | 类型 | 输入电压 | 输出电压 | 输出电流 | 特点 |
|------|------|---------|---------|---------|------|
| AMS1117-3.3 | LDO | 4.75V~15V | 3.3V | 1A | 经典但压差大时发热 |
| HT7533 | LDO | 最高 30V | 3.3V | 100mA | 微小电流、静态功耗 2.5μA |
| MP1584 | Buck | 4.5V~28V | 可调 | 3A | 高频（1.5MHz）、外围小 |
| MT3608 | Boost | 2V~24V | 可调（最高 28V） | 2A | 升压经典方案 |
| TPS63070 | Buck-Boost | 2V~16V | 3.3V 固定 | 2A | 自动升降压，电池电压变化也不怕 |

::: tip 实战经验
直接用现成的降压模块（MP1584 模块 ¥3~5 一块）比自己设计电路省事得多。比赛不是上板课，能买模块就别自己焊——省下来的时间用来调 PID。
:::

## 小结

先 DC-DC 降压到 5V，再 LDO 出 3.3V——这是电赛供电的标准答案。电机走一路独立电源，舵机走一路（或和电机共用但要加足够的电容），MCU 和传感器走干净的 3.3V。

# 功率预算与续航估算

## 为什么要算

你的小车在赛道上跑了 30 秒突然没电了，或者电池带上车才发现 5 分钟就没电——这都是没有做功率预算的后果。

## 功率核算表

把每个用电设备列出来，填表：

| 设备 | 数量 | 电压 | 单设备电流 | 总计电流 |
|------|------|------|-----------|---------|
| MCU（STM32F407） | 1 | 3.3V | 100mA | 100mA |
| 电机驱动 TB6612 | 1 | 7.4V | 0.2mA | —（控制端，可忽略） |
| TT 马达（行驶中） | 2 | 7.4V | 500mA | 1000mA |
| OLED 屏 | 1 | 3.3V | 20mA | 20mA |
| JY61P 陀螺仪 | 1 | 3.3V | 10mA | 10mA |
| 8 路灰度传感器 | 1 | 3.3V | 80mA | 80mA |
| HC-05 蓝牙 | 1 | 3.3V | 30mA | 30mA |
| **合计（行驶状态）** | | **7.4V** | | **~1100mA** |
| **合计（堵转状态）** | | **7.4V** | TT 马达堵转 2A×2 | **~4150mA** |

## 功率分算

```
7.4V 侧总功率（行驶）：7.4V × 1.1A = 8.1W
3.3V 侧总功率：3.3V × 0.24A = 0.8W
总功率（行驶）：≈ 8.9W
```

## 续航估算

假设 2S Li-Po，容量 2200mAh：

```
行驶续航 = 2200mAh / 1100mA ≈ 2 小时
堵转续航 = 2200mAh / 4150mA ≈ 0.5 小时（32 分钟）
```

实际情况介于两者之间。保守估计：**1~1.5 小时的实际可用时间**。

::: tip 调试 vs 比赛
调试时小车不一定一直跑，可以断续用一整天。但正式比赛前**必须换满电电池**。别拿调试了一天的电池上场。
:::

## 省电技巧

| 技巧 | 节约效果 | 实现难度 |
|------|---------|---------|
| 不跑时关电机使能 | 大幅 | 简单（EN 引脚拉高） |
| OLED 亮度调低或不刷时休眠 | 中等 | 简单 |
| 传感器按需采样，不用时关掉 | 中等 | 中等 |
| MCU 空闲时进 SLEEP 模式 | 小 | 需要调 RTOS 或中断唤醒 |

## 小结

填一张功率预算表只需要 5 分钟，但它能让你**在买电池之前就知道要买多大容量的**。行驶电流×安全系数 1.5→电池输出能力，容量÷行驶电流→续航时间，选一块能覆盖这两个数值的电池。

# 保护电路

电赛现场，一块开发板烧了可以换，但电脑烧了就是大事故。保护电路就是给你的系统上保险。

## 反接保护

电池插反了怎么办？最简单的保护：

```
电池正 ──┬──→ 电路
         │
电池负 ──┤
         │
电池正 ──┼── PMOS (S)
         │      │G
电池负 ──┤      └──→ 电池负 → PMOS (D) ──→ 电路 GND
```

更简单的方法：串联一个**肖特基二极管**。正向导通，反向截止。代价是有 0.3V~0.5V 的正向压降，大电流时会发热。

::: tip 反向保护最简单方案
用一个 **2P 的防呆连接器**（XT30 / XT60 / T 插 / JST），物理上就不可能插反。比任何电路保护都可靠。
:::

## 过流保护

电机堵转 → 电流飙升 → 烧驱动芯片 → 烧电池 → 起火。

| 方案 | 工作原理 | 响应速度 | 价格 |
|------|---------|---------|------|
| **一次性保险丝** | 电流超过额定值 → 自身熔断 | 慢（毫秒到秒） | ¥0.1 |
| **自恢复保险丝（PTC）** | 过热 → 电阻飙升 → 冷却后恢复 | 中 | ¥0.5 |
| **检流电阻 + 比较器** | 测电压 → 超过阈值关断 MOSFET | 快（微秒级） | ¥2 |
| **驱动芯片自带过流保护** | 芯片内部检流 | 快 | 0（芯片自带） |

::: tip 实战
TB6612 自带过流保护和热关断，基本够用。电池输出端加一个**自恢复保险丝（PTC）**做最后防线——2A~5A 规格的，几毛钱一个。
:::

## 防反向电动势

电机关断瞬间，线圈里的磁场能量无处释放，会产生极高的反向电压尖峰——瞬间击穿 MOSFET。

**续流二极管**是标准解法：

```
Vmotor ──┬── 二极管（阴极）──┬── 电机线圈
         │                  │
         └── 二极管（阳极）──┘
```

H 桥芯片（如 TB6612）内部集成了续流二极管，不需要外接。但用分立 MOSFET 搭 H 桥时**一定要加**。

## 静电保护（ESD）

比赛现场干燥，摸一下板子就可能导致 MCU 复位或 GPIO 击穿。关键引脚（USB 数据线、裸露的连接器引脚）加 TVS 二极管到地。

## 上电检查清单

每次上电前（尤其是新焊的板子）：

1. **万用表蜂鸣档测电源到地短路**——响 = 短路 = 别上电！
2. **目视检查**——有没有明显短路、虚焊、引脚歪斜
3. **带限流上电**——先用可调电源限流 100mA，看电流是否正常
4. **手背摸芯片温度**——异常发烫 = 立刻断电
5. **测各点电压**——3.3V 是 3.3V 吗？5V 是 5V 吗？

::: danger 新板子上电须知
**新板子第一次上电，不要用电池！** 用可调电源并设置电流限制。电池没有限流保护，短路直接烧板子。
:::

## 小结

保护电路是电赛的"安全带"——平时好像没用，出事的时候救你一条命。反接保护、过流保护、续流二极管，三样加起来成本不超过 5 元。花 5 元省下一块开发板和几天的调试工作，值。
